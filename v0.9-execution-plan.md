# EstaCoda v0.9 Execution Plan

**Version:** v0.9.0  
**Theme:** Multi-Channel Gateway, Cron Reliability, and Cross-Surface Session Operations  
**Repo Baseline:** `98fcf88` (tag: v0.8) on `main`  
**Date:** 2026-05-04  
**Status:** Ready for implementation  

---

## 1. Objective

Make EstaCoda reachable, schedulable, and operable outside the terminal without accidental context bleed or unreliable background execution.

After v0.9, a user should be able to:

1. Talk to EstaCoda from Telegram.
2. Talk to EstaCoda from Discord.
3. Talk to EstaCoda from WhatsApp (experimental live adapter).
4. Send and receive email through EstaCoda using Hermes-equivalent behavior.
5. Create, pause, resume, trigger, inspect, and remove scheduled jobs with visible execution history.
6. Continue a CLI session from Telegram when explicitly requested.
7. Keep CLI and channel sessions separate when no handoff is requested.
8. Inspect channel-triggered and cron-triggered runs from the same run/trajectory system.

## 3. Hermes-Alignment Decisions

These areas match Hermes behaviorally where specified:

| # | Area | Hermes Behavior | EstaCoda Decision |
|---|---|---|---|
| 1 | **Delivery target syntax** | `origin`, `local`, `telegram:chatId:threadId`, `discord:channelId:threadId`, `email:address`, comma-separated multi-target | Adopt syntax |
| 2 | **Email behavior** | IMAP UNSEEN polling; SMTP reply-in-thread with `In-Reply-To`/`References`; self-message/noreply/automated filters; `EMAIL_ALLOWED_USERS`/`EMAIL_ALLOW_ALL_USERS`; `EMAIL_HOME_ADDRESS`; `skip_attachments`; `MEDIA:/path` outgoing; 15s default poll; HTML body; rich MIME; inline images | Mirror behaviorally |
| 3 | **Discord session model** | DM per user; server channel per-user by default (`group_sessions_per_user: true`); thread-aware session keys | Match |
| 4 | **WhatsApp library** | Baileys (`@whiskeysockets/baileys`); QR pairing; session persistence under `~/.hermes/platforms/whatsapp/session` | Use Baileys |
| 5 | **Cron fresh session** | Every run gets new session ID (`cron_{job_id}_{timestamp}`) | Fresh session default |
| 6 | **Cron recursion guard** | `disabled_toolsets=["cronjob", "messaging", "clarify"]` inside cron | Preserve guard |
| 7 | **Cron duplicate prevention** | File lock + `advance_next_run()` before execution under lock | Match pattern |
| 8 | **Gateway process model** | Single process starts all adapters; shared runtime; cron delivery prefers live adapters with standalone fallback | Single process |
| 9 | **Approval/security** | `PairingStore`-style codes (8-char, 1h expiry, rate limits, lockout); no channel-specific approval beyond global policy | Match |
| 10 | **Home channel model** | `/sethome` on Telegram; `TELEGRAM_HOME_CHANNEL` env; equivalent for Discord/Email | Implement equivalents |

## 4. Intentional Divergences from Hermes

| # | Area | Hermes Behavior | EstaCoda Divergence | Rationale |
|---|---|---|---|---|
| 1 | **Test runner** | Python `pytest` | **Vitest** (not `bun:test`) | TypeScript/Bun stack; Vitest has better IDE support, mocking, coverage |
| 2 | **Cron execution history** | Job-object mutation only (`last_status` + markdown files) | **SQLite `cron_executions` table** + file output | Queryable history required by v0.9 acceptance criteria |
| 3 | **Cross-surface handoff** | Substrate-level sharing only (same DB, no explicit commands) | **Explicit `attach`/`detach` with short-lived handoff codes** | Prevents accidental context bleed |
| 4 | **Email/Discord implementation** | Python `imaplib`/`smtplib`/`discord.py` | **Node `imap`/`nodemailer`/`discord.js`** | Node/Bun runtime requirement |
| 5 | **Cron lock recovery** | Raw file lock can deadlock after crash | **PID-based stale lock stealing** | Fixes crash-deadlock without changing architecture |
| 6 | **Session surface pointers** | Direct session-key→session-ID mapping | **Explicit `surfacePointer` → `sessionId` layer** | Enables attach/detach without session migration |

## 5. Non-Goals

v0.9 will **not** attempt:

- Full enterprise multi-user admin
- Full marketplace trust system
- Team-wide shared channel sessions (per-user default is required)
- Slack adapter
- Signal adapter
- Calendar integration
- Gmail/Google API-native email integration
- Complex campaign/broadcast email behavior
- Zapier-style visual workflow builder
- Production-grade mobile app
- Public hosted gateway service
- Full GUI dashboard
- Gateway service mode (`install`/`start`/`stop` systemd/launchd integration)
- Voice messages in Discord/WhatsApp (attachment support is enough)
- Background sessions (`/background`)
- Home Assistant / Mattermost / Matrix adapters
- Full interactive setup wizard (docs + diagnostics are required; wizard is deferred)

## 6. Phases

### Phase 0 — Test Runner + DeliveryRouter
**Goal:** Foundation. Everything else depends on delivery normalization and the ability to test without real APIs.

**Workstreams:**
- A. Add Vitest test runner
- B. Add fake channel adapters
- C. Build DeliveryRouter

**Exit criteria:**
- `bun test` runs Vitest. **If Vitest/Bun interop fails, document the blocker and fallback to `bun:test` with explicit written justification.**
- `DeliveryRouter` routes to Telegram, Discord, WhatsApp, Email, local file, origin, and silent.
- Telegram adapter delivery refactored through DeliveryRouter.
- Fake adapters exist and record all sent messages for assertions.

---

### Phase 1 — Cron Reliability
**Goal:** Turn cron from "works sometimes" into "trustworthy scheduled execution."

**Workstreams:**
- D. Cron execution history store
- E. Robust tick lock + duplicate prevention
- F. Cron delivery through DeliveryRouter

**Exit criteria:**
- `CronExecutionStore` persists queryable history in SQLite.
- File lock writes `{pid, startedAt}`; stale lock recovery after 5 minutes.
- `advance_next_run()` called under lock before execution.
- Fresh session default preserved; recursion guard verified.
- Failure classes: `script-failed`, `runtime-error`, `delivery-failed`, `timeout`, `provider-error`.
- Cron output routes through DeliveryRouter to any channel.

---

### Phase 2 — CLI ↔ Telegram Handoff
**Goal:** Prove explicit cross-surface session semantics with the simplest real channel.

**Workstreams:**
- G. Surface pointer model
- H. Handoff code flow
- I. Attach/resume/detach commands

**Exit criteria:**
- `estacoda sessions list` shows CLI and channel sessions.
- `estacoda handoff telegram` generates a short-lived code (15 min expiry).
- `/attach <code>` from Telegram binds chat to CLI session.
- `/detach` from Telegram returns chat to independent session.
- `estacoda sessions detach telegram` from CLI removes binding.
- No accidental context bleed when not attached.

### Workstream E — Robust Cron Tick Lock

- **Tests required:** Lock acquisition tests; stale lock recovery tests; concurrent tick tests.

## 13. Implementation Risks

### Risk 1: Session Leakage Across Surfaces
**Likelihood:** Medium | **Impact:** High  
If the surface pointer layer is buggy, a Telegram chat could accidentally read a CLI session's history.  
**Mitigation:** Extensive unit tests for attach/detach; explicit handoff codes only; no automatic merging; audit log of all attachments.

### Risk 2: DeliveryRouter Complexity
**Likelihood:** Medium | **Impact:** High  
Normalizing delivery for 4+ channels + cron + approvals could create a brittle abstraction.  
**Mitigation:** Start with simple target string parsing; adapter registration is explicit; fake adapters test all paths; no dynamic adapter loading.

### Risk 3: Cron Duplicate Execution
**Likelihood:** Low | **Impact:** High  
If the lock is bypassed or `advance_next_run` is not atomic, a job could run twice.  
**Mitigation:** Lock file contains PID + timestamp; stale lock recovery; `advance_next_run` under lock; execution history makes duplicates visible.

### Risk 4: Email Over-Governance
**Likelihood:** Medium | **Impact:** Medium  
Well-intentioned security additions could make email stricter than Hermes.  
**Mitigation:** Code review checklist item: "Does this add email-specific approval?"; negative test: verify no email-specific approval layer exists.

### Risk 5: WhatsApp Live Adapter Fragility
**Likelihood:** High | **Impact:** Medium  
Baileys/linked-device can break with WhatsApp Web protocol changes.  
**Mitigation:** Live adapter gated behind `experimental: true`; architecture + fake adapter are stable; clear error messages when bridge fails; reconnect behavior documented.

### Risk 6: Discord Intent Misconfiguration
**Likelihood:** Medium | **Impact:** Medium  
Missing `MessageContent` intent causes silent message non-delivery.  
**Mitigation:** Diagnostics check intent configuration; startup logs warn if intents are insufficient; docs explicitly list required intents.

### Risk 7: Config/Env Fragmentation
**Likelihood:** Medium | **Impact:** Low  
Adding many new env vars could create confusion.  
**Mitigation:** Use Hermes env var names where they exist; `estacoda gateway diagnose` reports missing vars by name.

### Risk 8: Vitest/Bun Interop Failure
**Likelihood:** Low | **Impact:** Medium  
Vitest may have edge-case issues with Bun's Node compatibility layer.  
**Mitigation:** Phase 0 must validate a minimal Vitest test runs through `bun test` before expanding. If it fails, fallback to `bun:test` with explicit written justification.

## 14. Rollback Strategy

If a phase introduces critical breakage:

1. **Git revert** the offending commit(s). Each workstream should be committed independently.
2. **Feature flags:** New adapters (Discord, Email, WhatsApp) are gated by config `enabled: false` by default. If live adapter breaks, disable in config without code change.
3. **Cron:** If cron reliability changes break scheduling, `CronStore` format remains backward-compatible. Revert to v0.8 `cron-runner.ts` if necessary.
4. **DeliveryRouter:** If router causes Telegram delivery regressions, Telegram adapter can temporarily fall back to direct delivery while router is fixed.
5. **Session store:** Surface pointer layer is additive. If buggy, the store can ignore pointers and fall back to direct session-key mapping.

## 16. Implementation Decisions to Capture

No unresolved questions block implementation planning. The following decisions must be explicitly captured in the v0.9 plan:

### Decision 1 — Baileys Dependency Model
- **Required dependency:** No. Baileys is an **optional/experimental dependency**.
- **Rationale:** Baileys carries native dependency risk and WhatsApp account risk. The core v0.9 build must not fail if Baileys is missing.
- **Implementation:** Mark `@whiskeysockets/baileys` as `optionalDependencies` in `package.json`. Check availability at runtime; if missing, log warning and disable WhatsApp adapter. Fake adapter and config do not depend on Baileys.

### Decision 2 — Discord Slash Command Scope
- **Core v0.9:** Prefix commands (`!status`, `!sessions`, `!attach`, `!detach`) for all operator commands. Message content intent is sufficient.
- **v0.9.1 hardening:** Slash command registration (`/status`, `/sessions`, etc.) via Discord application commands API.
- **Rationale:** Prefix commands work immediately without application command registration latency. Slash commands require guild-specific registration and permission handling.
- **Implementation:** Discord adapter parses messages for prefix commands in v0.9. Slash command infrastructure may be stubbed but is not required for release.

### Decision 3 — Email Attachment Scope
- **v0.9 core:** File attachments (any MIME type) ingested via IMAP and cached locally. Outgoing `MEDIA:/path` attachments delivered as SMTP MIME attachments. Plain-text body. HTML body where Hermes supports it. Rich MIME parsing/rendering where Hermes supports it. Inline image/media handling where Hermes supports it. Attachment forwarding/reply behavior where Hermes supports it.
- **No v0.9.1 deferral unless hard blocker:** If a specific email media feature cannot be implemented, document the blocker in code and validation report.
- **Rationale:** Hermes email includes these behaviors. v0.9 email must be behaviorally equivalent.

### Decision 4 — Cron Stale-Lock Timeout
- **Timeout:** 5 minutes (300 seconds).
- **Recovery behavior:** On lock acquisition failure, read `{pid, startedAt}` from lock file. If PID does not exist OR `startedAt` is older than 300 seconds, delete lock file and retry acquisition once. If retry fails, skip tick and log warning.
- **Rationale:** 5 minutes is longer than any normal tick (which should complete in seconds) but short enough to recover quickly from crashes.
- **Implementation:** `CronTickLock` class encapsulates this logic. Platform-specific PID checks: Unix uses `process.kill(pid, 0)`, Windows uses `tasklist` or Node `process` module equivalents.

### Decision 5 — Session Handoff UX
- **Both models supported:**
  1. **Code-based handoff** (primary): `estacoda handoff telegram` generates code; user sends `/attach <code>` from Telegram. Safer; no session ID exposure.
  2. **Direct attach by session ID** (secondary): `estacoda channels attach telegram <session-id>` from CLI, or `/attach <session-id>` from channel. Useful for scripted/automated attachment.
- **Rationale:** Code-based is the default UX for security. Direct ID is useful for operators who know what they're doing.
- **Implementation:** `HandoffCodeStore` handles codes. `ChannelSessionStore.attach(surfacePointer, sessionId)` handles direct attaches. Both paths validate that the session exists and the user is authorized.

### Decision 6 — WhatsApp Live Adapter Flag
- **Experimental flag required:** `config.channels.whatsapp.experimental: true` (default `false`).
- **Config gate:** Even if `WHATSAPP_SESSION_PATH` and other env vars are set, the live adapter does not start unless `experimental: true` is explicitly set.
- **Rationale:** Baileys integration is high-risk. Users must opt-in knowingly.
- **Implementation:** `WhatsAppAdapter.start()` checks `config.experimental` before initializing Baileys. If `false`, logs "WhatsApp adapter configured but experimental flag is false" and returns without error. Fake adapter and diagnostics work regardless.
