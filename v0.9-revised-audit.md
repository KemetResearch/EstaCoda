## 3. Existing Infrastructure We Can Reuse

**Do not rebuild:**

1. **`ChannelGateway`** (`src/channels/channel-gateway.ts`) — The generic adapter→runtime bridge is solid. It already handles:
   - Adapter registration and lifecycle (`start`/`stop`)
   - Message preprocessing
   - Auth policy (`authorizeChannelMessage`)
   - Command parsing (`parseGatewayCommand`)
   - Approval store integration (`ChannelApprovalStore`)
   - Runtime factory injection
   - Progress event forwarding
   - Artifact delivery

2. **`ChannelAdapter` interface** (`src/contracts/channel.ts`) — Already designed for multi-channel:
   - `kind: ChannelKind` supports `discord`, `whatsapp`, `email`
   - `delivery: ChannelDelivery` abstraction
   - `start`/`stop`/`send`/`receive` hooks

3. **`PersistentChannelSessionStore`** (`src/channels/channel-session-store.ts`) — File-based session mapping with:
   - Auto-reset policies (`idle`, `daily`, `both`)
   - Session key normalization (`normalizeSessionKey`, `stableSessionKey`)
   - Per-user isolation for groups/threads

4. **`CronStore`** (`src/cron/cron-store.ts`) — Atomic file-based job storage with:
   - Safe read-modify-write (`#mutate` with temp file + rename)
   - Schedule parsing (`once`, `interval`, `cron`, `ISO timestamp`)
   - Job lifecycle (`active`/`paused`/`completed`)
   - Output persistence to `~/.estacoda/cron/output/`

5. **`TrajectoryRecorder`** (`src/trajectory/trajectory-recorder.ts`) — In-memory trajectory capture with compression. Already integrated into `createRuntime`.

6. **`Runtime` factory** (`src/runtime/create-runtime.ts`) — Already accepts `channel` parameter and creates trajectories. Cron already creates fresh runtimes.

7. **`ChannelApprovalStore`** (`src/channels/channel-approval-store.ts`) — Persistent approval grants.

8. **Security policy factory** (`src/security/security-policy-factory.ts`) — `createSecurityPolicyForMode` can be reused across all surfaces.

## 5. Architecture Risks

### Risk 1: Session Leakage Across Surfaces
**Current state:** `ChannelSessionStore` maps `sessionKey` → `sessionId`. There is **no** concept of a surface pointer to an existing CLI session. A Telegram chat always gets a `channel-telegram-*` session ID.

**Mitigation:** Implement explicit `attach`/`detach` with handoff codes. Store attachment as a separate mapping layer: `surfacePointer` → `sessionId`, where `surfacePointer` can be updated without changing the underlying session.

### Risk 2: DeliveryRouter Absence
**Current state:** `TelegramAdapter.delivery` is a hardcoded object. `ChannelGateway` calls `adapter.delivery.sendText()` directly. There is no way for cron to deliver to Telegram without importing `TelegramAdapter`.

**Mitigation:** Build `DeliveryRouter` **before** Discord/Email/WhatsApp. The router should resolve a delivery target string (`telegram:chatId`, `discord:#channel`, `email:address`, `local`, `origin`) to a delivery function. Adapters register themselves with the router.

### Risk 3: Cron Duplicate Execution
**Current state:** `withCronTickLock` uses `open(path, "wx")`. If the process crashes, the lock file remains and must be manually deleted. There is no PID or timestamp recovery.

**Mitigation:** Write `{pid, startedAt}` into the lock file. On acquisition failure, read the file. If PID is dead or startedAt is > 5 minutes old, steal the lock.

### Risk 4: Email Over-Governance
**Current state:** No email adapter exists, so the risk is in the design. Hermes email has **no special approval rules** beyond the global policy and `EMAIL_ALLOWED_USERS`.

**Mitigation:** EstaCoda email must use the same `SecurityPolicy` as every other surface. Do not add `emailRequiresExtraApproval` or similar. The only email-specific controls are: allowed senders, home address, IMAP/SMTP config, poll interval.

### Risk 5: Telegram-Only Assumptions in Gateway
**Current state:** `ChannelGateway.#handleCommand` has Telegram-specific logic:
```ts
await adapter.delivery?.sendText(..., adapter.kind === "telegram" ? {format: "html", actions: ...} : undefined)
```
This hardcodes Telegram HTML formatting and inline keyboards.

**Mitigation:** Generalize approval prompt rendering. The `ChannelAdapter` should expose a `renderApprovalPrompt(pendingApproval)` method, or the gateway should use plain text as the lowest common denominator.

### Risk 6: CLI-Only Approval Assumptions
**Current state:** `ChannelApprovalStore` persists grants, but the approval UI is adapter-specific. For Email, there is no callback button.

**Mitigation:** Email approvals should use reply-in-thread commands (`Reply "approve once"` or `Reply "approve always"`). The email adapter should parse these replies as approval commands before passing to the runtime.

### Risk 7: Recorder Gaps for Channel-Triggered Work
**Current state:** `TrajectoryRecorder` is created in `createRuntime`. However, there is no explicit `channel-triggered` trajectory event that records which surface initiated the run.

**Mitigation:** Ensure `createRuntime` receives the `channel` parameter and records a `session-start` event with `channel: "telegram"` etc. Cron runs should record `channel: "cron"`.

### Risk 8: Config/Env Fragmentation
**Current state:** `LoadedRuntimeConfig` only has `channels.telegram`. Discord, WhatsApp, Email, and Cron have no config types.

**Mitigation:** Extend `EstaCodaConfig.channels` with:
```ts
discord?: DiscordChannelConfig;
whatsapp?: WhatsAppChannelConfig;
email?: EmailChannelConfig;
```
And add `cron?: CronConfig` at the top level.

## 6. Suggested Adjustments to the v0.9 Plan

### Adjustment 1: WhatsApp Is Not Deferred — It Is Architected
The original plan suggested WhatsApp might be deferred. **Do not defer.** v0.9 must include:
- `WhatsAppChannelConfig` type and env shape (`WHATSAPP_SESSION_SECRET`, etc.)
- `WhatsAppAdapter` interface + fake implementation for tests
- Pairing model decision (Baileys linked-device)
- Diagnostics (`estacoda gateway diagnose` checks WhatsApp config)
- **Live adapter marked `@experimental`** in docs

### Adjustment 2: DeliveryRouter Is Phase 1, Not a Side Effect
The original plan listed DeliveryRouter as a component. **Elevate it to the first workstream.** No Discord, Email, or cron-to-channel delivery should be built before the router exists.

### Adjustment 3: Cross-Surface Handoff Proven with CLI↔Telegram Before Discord/Email
Do not design handoff abstractly. Build it for CLI↔Telegram first, then generalize. This validates the `surfacePointer` → `sessionId` mapping model with the simplest case.

### Adjustment 4: Cron Reliability Requires a Separate Execution History Store
The original plan's `CronExecution` type is correct, but the current code only has `lastStatus` on the job. **Add a new `CronExecutionStore`** (file-based JSONL or SQLite) separate from `CronStore`. Job store = schedule metadata. Execution store = run history.

### Adjustment 5: Hermes Email Is the Exact Target — No Stricter
Hermes email behavior to copy exactly:
- `EMAIL_ALLOWED_USERS` comma-separated
- `EMAIL_ALLOW_ALL_USERS=true`
- `EMAIL_HOME_ADDRESS` for cron default delivery
- `EMAIL_POLL_INTERVAL` (default 15s)
- Marks existing messages as seen on startup
- Self-message filter (prevents reply loops)
- `noreply@`, `mailer-daemon@`, `bounce@`, `no-reply@` silent ignore
- `Auto-Submitted`, `Precedence: bulk`, `List-Unsubscribe` header ignore
- `In-Reply-To` + `References` threading
- `MEDIA:/path` outgoing attachments
- `skip_attachments` config option

EstaCoda should **not** add:
- Email-specific approval escalation
- Attachment quarantine beyond the global policy
- Separate email security mode

### Adjustment 6: Introduce Vitest
The codebase has zero tests. The smoke framework is custom and not a substitute for unit tests. Add `vitest` to `devDependencies`, create `vitest.config.ts`, and write tests for:
- `DeliveryRouter`
- `ChannelSessionStore` (attach/detach/handoff)
- `CronExecutionStore`
- `FakeTelegramAdapter`
- `FakeDiscordAdapter`
- `FakeEmailAdapter`
- `CronTickLock`

## 7. Proposed v0.9 Workstreams

### Workstream A — DeliveryRouter & Channel Delivery Normalization
- **Objective:** Create a unified delivery layer so that any component (gateway, cron, runtime) can send messages to any channel without knowing adapter internals.
- **Files affected:** `src/channels/delivery-router.ts`, `src/channels/channel-gateway.ts`, `src/contracts/channel.ts`, `src/cron/cron-runner.ts`
- **Dependencies:** None (foundational)
- **Acceptance criteria:**
  - `DeliveryRouter` can route to Telegram, Discord, WhatsApp, Email, local file, origin, and silent.
  - Adapters register their delivery capability with the router.
  - Cron uses the router instead of a callback.
  - Telegram adapter is refactored to register with the router.
- **Tests:** Fake adapters for all 4 channels; router unit tests.
- **Risk:** Medium
- **Parallelizable:** No — blocks B, C, D, E, F

### Workstream B — Test Runner & Fake Adapters
- **Objective:** Introduce Vitest and build fake channel adapters for isolated testing.
- **Files affected:** `package.json`, `vitest.config.ts`, `src/test/fakes/*` (new directory)
- **Dependencies:** None
- **Acceptance criteria:**
  - `bun test` runs Vitest.
  - `FakeTelegramAdapter`, `FakeDiscordAdapter`, `FakeWhatsAppAdapter`, `FakeEmailAdapter` exist.
  - Each fake implements `ChannelAdapter` and records all sent messages for assertions.
- **Tests:** Self-testing (test runner tests).
- **Risk:** Low
- **Parallelizable:** Yes, with A

### Workstream C — CLI↔Telegram Cross-Surface Session Handoff
- **Objective:** Prove explicit session handoff between CLI and Telegram.
- **Files affected:** `src/cli/cli-session-store.ts`, `src/channels/channel-session-store.ts`, `src/cli/cli.ts`, `src/channels/channel-gateway.ts`
- **Dependencies:** A (DeliveryRouter for handoff notifications)
- **Acceptance criteria:**
  - `estacoda sessions list` shows CLI and channel sessions.
  - `estacoda handoff telegram` generates a short-lived code.
  - `/attach <code>` from Telegram binds that Telegram chat to the CLI session.
  - `/detach` from Telegram returns the chat to an independent session.
  - `estacoda sessions detach telegram` from CLI removes the binding.
  - No accidental context bleed when not attached.
- **Tests:** Unit tests for handoff code generation/expiry; fake Telegram attach/detach.
- **Risk:** High (session leakage risk if done wrong)
- **Parallelizable:** No — must be proven before D, E

## 8. Recommended Implementation Order

| Phase | Workstreams | Rationale |
|---|---|---|
| **Phase 0 — Foundation** | A (DeliveryRouter) + B (Test Runner + Fakes) | Everything else depends on delivery normalization and the ability to test without real APIs. |
| **Phase 1 — Handoff Proof** | C (CLI↔Telegram Handoff) + D (Telegram Hardening) | Proves the session pointer model with the simplest real channel. Telegram hardening is lower risk than new channels. |
| **Phase 2 — New Channels** | E (Discord) + F (Email) + G (WhatsApp Arch) | Discord and Email can be built in parallel using the proven router and handoff model. WhatsApp architecture happens in parallel because live implementation is deferred. |
| **Phase 3 — Cron Hardening** | H (Cron Reliability) | Cron delivery depends on the router. Building it after channels means cron can deliver to Telegram/Discord/Email immediately. |
| **Phase 4 — Operator Surface** | I (Operator CLI/Diagnostics) + J (Docs) | Integrates all components into a unified status view. |

## 9. Open Questions / Decisions Needed

1. **Test runner choice:** Confirm Vitest (recommended) or preference for Bun's built-in test runner (`bun:test`)?
2. **Cron execution store format:** JSONL file (append-only, simple) or SQLite (queryable, scalable)? The codebase already uses SQLite elsewhere.
3. **WhatsApp library:** Confirm Baileys (`@whiskeysockets/baileys`) as the linked-device library? It is the de facto standard for WhatsApp Web automation.
4. **Email library:** Use Node.js native `imap` + `nodemailer` packages, or spawn Python like Hermes? Hermes uses Python's built-in `imaplib`/`smtplib`. Node has `imap` and `nodemailer` which are standard.
5. **Discord library:** Confirm `discord.js`? It is the standard Node.js Discord library.
6. **Delivery target syntax:** Hermes uses `telegram:chatId:threadId`, `discord:#channel`, `email:address`. Should EstaCoda adopt the same syntax exactly?

The repo has solid foundations (`ChannelGateway`, `PersistentChannelSessionStore`, `CronStore`, `TrajectoryRecorder`) but is missing **three blockers** that must be designed first:

1. **DeliveryRouter** — must exist before any new channel or cron delivery work.
2. **Test runner** — zero unit tests is unacceptable for gateway/session/cron logic.
3. **Cross-surface handoff proof** — the session pointer model must be proven with CLI↔Telegram before Discord/Email are built.

The original overview correctly identified the scope. The revision needed is:
- Elevate DeliveryRouter to Phase 0.
- Mandate Vitest + fake adapters before any live adapter work.
- Require CLI↔Telegram handoff before Discord/Email.
- Architect WhatsApp (config, fake adapter, docs) but mark live adapter experimental.
- Match Hermes email behavior exactly — no stricter rules.

**Repo is ready for v0.9 execution planning once these revisions are accepted.**
