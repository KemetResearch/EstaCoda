# EstaCoda Codebase Assessment: Post-v0.3 / Pre-v0.4

**Date:** 2026-05-02
**Scope:** Full source, contracts, tests, docs, and architecture mapping
**Lines of Code:** ~42,272 TypeScript lines across 127 source files
**Assessment Type:** Grounded audit + roadmap readiness analysis

---

## 1. Executive Verdict

**Overall Health: B+**

EstaCoda v0.3 is a **functionally impressive, architecturally strained** system. It proves the product thesis — agents can execute real work through CLI and Telegram with skills, memory, and tool calling — but the execution spine is beginning to collapse under its own weight. The codebase has outgrown its initial structure.

**The core problem:** The agent loop is a 2,714-line monolith that handles routing, security, skills, provider negotiation, tool execution, memory promotion, trajectory recording, artifact tracking, and prompt assembly in a single class. This is not a theoretical concern. It blocks v0.4 (decomposition), v0.5 (tracing), and v0.7 (evolution) because no component can be tested, replaced, or evolved independently.

**The core strength:** The contract layer is clean, the skill system is mature, security boundaries exist, and the system is live-proven against real providers (Kimi, OpenAI, DeepSeek). This is not a prototype. It is an alpha product with production-grade surface area and research-grade internals.

## 4. What Is Brittle (Critical Debt)

### 4.1 AgentLoop Monolith — CRITICAL
- **File:** `src/runtime/agent-loop.ts`
- **Size:** 2,714 lines, 91728 bytes
- **Methods:** 25+ async methods
- **Responsibilities:** Intent routing, security assessment, skill workflow compilation, provider loop management, tool call planning, tool execution, memory promotion, trajectory recording, artifact collection, prompt assembly, context expansion, progress tracking, fallback handling

**Why this blocks the roadmap:**
- v0.4 requires decomposing the loop — impossible without major surgery
- v0.5 requires structured trajectory recording — currently embedded in loop logic
- v0.7 requires skill evolution hooks — currently ad-hoc inside the loop
- v0.8 requires durable TaskFlow — no state machine boundary exists

**The loop does not have clear phase boundaries.** A run is not: plan → execute → record. It is: route → maybe-plan → maybe-skill → provider-loop-with-tool-calls-interleaved → memory-promote → artifact-collect → respond. Everything is interleaved.

### 4.2 create-runtime.ts God Factory — HIGH
- **File:** `src/runtime/create-runtime.ts`
- **Size:** 830 lines, 63 imports
- **Problem:** Manually constructs 30+ subsystem objects with deep dependency chains
- **Impact:** Any constructor signature change cascades. Testing requires constructing the entire universe. No dependency injection container, no plugin boundary.

### 4.3 smoke.ts Test Monolith — HIGH
- **File:** `src/smoke.ts`
- **Size:** 13,969 lines, 89 imports
- **Problem:** All smoke tests in one file, no formal test framework, no unit tests
- **Impact:** Test execution requires full runtime. No granular failure isolation. Adding a test means editing a 14k-line file.

### 4.4 Trajectory & Artifact Skeletons — MEDIUM
- **TrajectoryRecorder:** 97 lines, in-memory only, no persistence
- **ArtifactStore:** 56 lines, in-memory only, no persistence
- **Impact:** v0.5 (Run Recorder) has a contract but no implementation substrate

### 4.5 Bun Lock-in — MEDIUM
- `bun:sqlite` import in `src/session/sqlite-session-db.ts`
- `$npm_execpath` in package.json scripts assumes Bun
- **Impact:** Cannot run smoke tests under Node. Cannot deploy to standard Node containers without polyfills.

### 4.6 Documentation Drift — LOW
- `AGENTS.md` references `src/gateway/` and `src/intent/` directories that do not exist
- `AGENTS.md` references `tests/` directory that does not exist
- **Impact:** Onboarding friction for new contributors

## 7. Roadmap Readiness by Version

### v0.4 — Agent-Loop Decomposition
**Readiness: BLOCKED until refactoring completes**
- The monolith must be split into: Router, Planner, Executor, Recorder
- `create-runtime.ts` needs DI boundaries or at least subsystem facades
- Trace schema should be finalized before decomposition starts (to give components a contract to emit into)
- **Estimated effort:** 2-3 engineering weeks

### v0.5 — Run Recorder, Trace Schema, Eval Substrate
**Readiness: CONTRACTS EXIST, IMPLEMENTATION MISSING**
- `TrajectoryRecorder` is a skeleton
- `ArtifactStore` is a skeleton
- Eval runner exists as a script but not as a scored system
- **Estimated effort:** 1-2 engineering weeks

### v0.6 — Memory, Dependency Graph, Knowledge Graph
**Readiness: PARTIAL**
- Memory promotion and rendering exist
- No code dependency graph integration
- No knowledge graph implementation
- **Estimated effort:** 2 engineering weeks

### v0.7 — Skill Evolution, Curator, Governed Self-Improvement
**Readiness: BEST-PREPARED VERSION**
- Skill evolution store, learning manager, and proposal flows already exist
- Missing: evidence-backed change manifests, formal curator pipeline, GEPA/DSPy export
- **Estimated effort:** 1-2 engineering weeks

### v0.8 — Durable TaskFlow
**Readiness: NO FOUNDATION**
- No flow state machine
- No step states, wait/resume, or child tasks
- AgentLoop has no persistence hooks
- **Estimated effort:** 3-4 engineering weeks

### v0.9 — Channels, Automations, Operators
**Readiness: PARTIAL**
- Telegram is solid
- Cron runner and store exist
- Missing: channel permissions, operator commands, broader channel matrix
- **Estimated effort:** 2 engineering weeks

### v0.10 — Trusted Extension, Evolution Pipeline, Launch
**Readiness: STUB-ONLY**

- **Estimated effort:** 3-4 engineering weeks

## 11. Final Assessment

EstaCoda v0.3 is a **live-proven alpha with research-grade ambition and maintenance-grade internals.** The product works. The architecture is beginning not to.

The good news: the hardest product problems (provider hardening, skill execution, channel integration, security policy) are largely solved. The remaining work is structural: decompose the loop, record execution, persist state, and build the evolution pipeline.

The v0.4–v0.10 roadmap is sound. The risk is that v0.4 (decomposition) is underestimated. It is not a refactor. It is a re-architecture of the system's heart. It should be treated as the highest-risk, highest-priority engineering run.

**Grade breakdown:**
- Product surface: A-
- Architecture health: C+
- Test discipline: D+
- Security boundaries: B+
- Roadmap alignment: A
- Documentation accuracy: B-

**Overall: B+** — Strong enough to launch an MVP, brittle enough that v0.4 must happen before v0.5–v0.10 can proceed safely.
