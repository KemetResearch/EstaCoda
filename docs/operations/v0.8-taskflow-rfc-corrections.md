# RFC Correction Pass: v0.8 Durable TaskFlow + Operator Control Plane

**Status:** Correction delta to RFC `v0.8.0-draft`  
**Date:** 2026-05-04  
**Constraint:** No implementation. No execution plan. No code changes. No doc index changes.

---

## 1. Summary of Corrections

| # | Area | Correction |
|---|---|---|
| 1 | Atomic state transitions | All transitions must be atomic SQLite transactions. No event-only or snapshot-only transitions. |
| 2 | Locking | Add explicit `flow_locks` table with heartbeat/stale-lock recovery. Flow state alone is too weak. |
| 3 | `/pause` semantics | Introduce `pauseRequested` state. Flow stays `running` during active execution; pause takes effect at the next safe boundary. |
| 4 | `/interrupt` semantics | `/interrupt` is for active execution only. Rejected when no active provider/tool/process is running. |
| 5 | Retry model | Adopt **Option B** (retry creates a new step with `retryOfStepId`). Failed steps stay terminal. |
| 6 | Approval integration | TaskFlow gates are durable wrappers around the existing `WorkspaceApprovalController`. No second authority. |
| 7 | `/compact` safety | `/compact` rejected during active execution. Compresses session context only; durable flow state is non-compactable. |
| 8 | FlowTemplate scope | Remove `FlowTemplate` from v0.8 core. Focus on `FlowInstance`, `FlowStep`, `FlowEvent`, `OperatorEvent`. |
| 9 | Recorder boundaries | `TaskFlowStore` owns flow/orchestration events. `TrajectoryRecorder` stays provider/tool/turn only. No contamination. |
| 10 | Stale process visibility | Persist process metadata. Show stale-process warnings on restart. Do not silently ignore orphans. |
| 11 | Architecture ambition | Option A+ for v0.8, **Option C-compatible by design**. Define callback seam for future mid-turn control. |

## 2. Updated Recommendations

### 2.1 Atomic Transitions (Correction #1)

**Previous:** Event append and snapshot update were described as sequential steps without explicit transaction boundaries.

**Corrected:** Every TaskFlow state transition is a single SQLite transaction with this exact order:

```
BEGIN TRANSACTION
  1. Acquire flow lock (INSERT or UPDATE flow_locks WHERE flow_id = ?)
  2. Validate current state (SELECT status FROM flows WHERE id = ?)
  3. If state mismatch, ROLLBACK and throw StateMismatchError
  4. Append flow event (INSERT INTO flow_events)
  5. Append operator event if applicable (INSERT INTO operator_events)
  6. Update flow snapshot (UPDATE flows SET status = ?, updated_at = ?)
  7. Update step snapshot if applicable (UPDATE flow_steps SET status = ?)
  8. Update related records (approval_gates, flow_run_links, etc.)
  9. Update flow lock heartbeat
COMMIT
```

**Invariants:**
- No event is ever appended without a corresponding snapshot update.
- No snapshot is ever updated without a corresponding event append.
- If the transaction fails, neither event nor snapshot is persisted.
- State validation happens inside the transaction to prevent races.

### 2.2 Locking Model (Correction #2)

**Previous:** "There are no explicit locks in v0.8. The 'lock' is the flow state itself."

**Corrected:** Add a `flow_locks` table and `FlowLockService`.

```sql
CREATE TABLE flow_locks (
  flow_id TEXT PRIMARY KEY,
  owner_id TEXT NOT NULL,           -- e.g., "process:<pid>", "cli:<session_id>"
  locked_at TEXT NOT NULL,
  heartbeat_at TEXT NOT NULL,
  expires_at TEXT NOT NULL,         -- heartbeat_at + lease_duration_ms
  FOREIGN KEY (flow_id) REFERENCES flows(id) ON DELETE CASCADE
);
```

**Lease duration:** 30 seconds (configurable).

**Heartbeat:** The active runner updates `heartbeat_at` and `expires_at` every 15 seconds.

**Acquire lock:**
```
INSERT INTO flow_locks (flow_id, owner_id, locked_at, heartbeat_at, expires_at)
VALUES (?, ?, now, now, now + 30s)
ON CONFLICT(flow_id) DO NOTHING
```
If no rows affected, lock is held by another owner.

**Stale lock recovery:**
```sql
DELETE FROM flow_locks WHERE expires_at < now()
```
Run at startup and before attempting to acquire.

**Lock release:**
- Normal: `DELETE FROM flow_locks WHERE flow_id = ? AND owner_id = ?`
- On crash: lock expires via `expires_at`.

**Owner identification:** `owner_id = "process:<pid>:<boot_time>"` to distinguish restarted processes.

### 2.3 /pause Semantics (Correction #3)

**Previous:** `/pause` immediately set `Flow.status = "paused"` even while the active turn continued.

**Corrected:**

```typescript
type Flow = {
  // ... existing fields ...
  pauseRequestedAt?: string;   // NEW: ISO timestamp when pause was requested
  pauseReason?: string;        // existing, now populated at request time
};
```

**State machine update:**

| Flow status | Active execution? | /pause behavior |
|---|---|---|
| `running` | Yes | Set `pauseRequestedAt`, `pauseReason`. Record `operator-pause-requested`. Flow stays `running`. |
| `running` | No | Set `status = "paused"`. Record `operator-paused`. |
| `paused` | N/A | Reject: "Flow already paused." |
| `waiting` | N/A | Reject: "Cannot pause while waiting. Resolve wait first." |

**Safe boundary detection:**
- TaskFlow checks `pauseRequestedAt` after every completed turn.
- If `pauseRequestedAt` is set and the active step has no more turns to dispatch, transition to `paused`.
- If `pauseRequestedAt` is set mid-turn, the turn completes, then pause takes effect.

**Step semantics:**
- Step status stays `running` during the active turn.
- When the safe boundary is reached, step transitions to `paused` (if it was the active step).

**CLI output:**
```
Pause requested for flow flow_abc123.
Current turn will complete, then the flow will pause.
```

Then, at the safe boundary:
```
Flow flow_abc123 paused at step "run-tests".
Use /resume to continue.
```

### 2.4 /interrupt Semantics (Correction #4)

**Previous:** `/interrupt` allowed while flow is `running` or `waiting`.

**Corrected:** `/interrupt` is for active execution only.

| Flow status | Active execution? | /interrupt allowed? | Correct command |
|---|---|---|---|
| `running` | Yes | **Yes** | `/interrupt` |
| `running` | No | **No** | Nothing to interrupt; use `/pause` or `/cancel` |
| `paused` | No | **No** | Flow already stopped; use `/resume` or `/cancel` |
| `waiting` | No | **No** | Nothing to interrupt; use `/cancel`, `/steer`, or wait resolution |
| `interrupted` | No | **No** | Already interrupted; use `/resume` or `/cancel` |

**Rationale:**
- Interrupt means "stop what is running right now."
- If nothing is running, there is nothing to interrupt.
- This prevents operator confusion between "interrupt" and "cancel."

**Active execution definition:**
- A provider HTTP request is in flight, OR
- A tool is actively executing, OR
- A process registered in the active execution registry is running.

### 2.5 Retry Model (Correction #5)

**Choice:** Option B.

**Rationale:**
- Option A (StepAttempt records) introduces a new abstraction layer and schema complexity.
- Option B keeps the schema flat: every execution unit is a `FlowStep`.
- Retry history is preserved via the `retryOfStepId` chain.
- This is simpler to implement, test, and inspect.

**Model:**

```typescript
type FlowStep = {
  // ... existing fields ...
  retryOfStepId?: StepId;   // NEW: points to the step this is retrying
  attemptNumber: number;    // NEW: 1 for original, 2+ for retries
};
```

**Retry behavior:**
1. Step `step_1` fails with `status = "failed"` (terminal).
2. Operator issues `/retry`.
3. TaskFlow validates:
   - `step_1.idempotent === true` OR operator confirms override.
   - `step_1.retryCount < step_1.maxRetries`.
4. TaskFlow creates `step_1_retry`:
   - `retryOfStepId = "step_1"`
   - `attemptNumber = 2`
   - `status = "pending"`
   - Inherits `name`, `description`, `retryPolicy`, `failurePolicy` from `step_1`.
5. `step_1_retry` is inserted at the same position in the step sequence (replaces the failed step in the execution plan).
6. Flow continues with `step_1_retry`.

**Event recording:**
- `step-failed` on `step_1`.
- `operator-retried` on the flow.
- `step-created` and `step-started` on `step_1_retry`.

**Trace view:**
```
Step 3: run-tests (failed)
  └── Retry 1: run-tests (completed)  ← current
```

### 2.6 Approval Integration (Correction #6)

**Previous:** TaskFlow approval gates were described as potentially independent from the existing approval system.

**Corrected:** TaskFlow gates are durable wrappers around `WorkspaceApprovalController`.

**Architecture:**

```
User requests tool execution
        │
        ▼
ToolExecutor.assessSecurityPolicy()
        │
    ┌────┴────┐
    ▼         ▼
 hard-deny   soft-deny / ask
    ▼         ▼
  fail      WorkspaceApprovalController
                │
            ┌────┴────┐
            ▼         ▼
        granted    denied/pending
            ▼         ▼
        execute   TaskFlow records gate
                  as waiting_for_approval
```

**TaskFlow's role:**
1. When `ToolExecutor` returns `decision: "ask"` and the tool is part of a flow step, TaskFlow creates an `ApprovalGate` record.
2. The gate stores the `approvalControllerGrantId` (reference to the existing controller's grant).
3. TaskFlow sets step status to `waiting_for_approval` and flow status to `waiting`.
4. When operator issues `/approve`, TaskFlow:
   - Calls `WorkspaceApprovalController.grantApproval()` with appropriate scope.
   - Updates `ApprovalGate.status = "approved"`.
   - Resumes the step.
5. When operator issues `/reject`, TaskFlow:
   - Calls `WorkspaceApprovalController.revokeApproval()` or records denial.
   - Updates `ApprovalGate.status = "rejected"`.
   - Applies failure policy.

**Invariant:** `/approve` checks the original `ToolExecutor` decision. If it was `decision: "deny"` with a `deterministicRule`, approval is rejected.

**Schema:**

```sql
CREATE TABLE approval_gates (
  id TEXT PRIMARY KEY,
  step_id TEXT NOT NULL,
  flow_id TEXT NOT NULL,
  status TEXT NOT NULL CHECK(status IN ('pending','approved','rejected')),
  requested_at TEXT NOT NULL,
  resolved_at TEXT,
  resolved_by TEXT,
  reason TEXT NOT NULL,
  risk_class TEXT NOT NULL,
  tool_name TEXT,
  target_key TEXT,
  target_summary TEXT,
  scope TEXT,
  controller_grant_id TEXT,     -- NEW: reference to WorkspaceApprovalController grant
  tool_executor_decision TEXT,  -- NEW: "ask" | "deny" | "allow"
  deterministic_rule TEXT,      -- NEW: if hard-denied, which rule
  FOREIGN KEY (flow_id) REFERENCES flows(id) ON DELETE CASCADE
);
```

### 2.7 /compact Safety (Correction #7)

**Previous:** `/compact` allowed while flow is `running`.

**Corrected:** `/compact` is **rejected** when there is active provider, tool, or process execution.

**Availability matrix (updated):**

| Flow status | Active execution? | /compact allowed? |
|---|---|---|
| `running` | Yes | **No** — active turn in progress |
| `running` | No | **Yes** — between turns |
| `paused` | No | **Yes** |
| `waiting` | No | **Yes** |
| `interrupted` | No | **Yes** |
| No active flow | N/A | **Yes** |

**Rejection message:**
```
Cannot compact: active execution in progress (provider call: run_abc, process: proc_123).
Wait for the current step to complete, or use /pause or /interrupt first.
```

**Non-compactable state (expanded):**
- Active flow ID, current step ID
- All steps with status not in `{completed, failed, skipped, cancelled}`
- Pending approvals
- Pending waits
- Unresolved operator interventions
- Active failure state
- Retry state (`retryCount`, `maxRetries`, `retryOfStepId` chain)
- Cancellation/interruption state
- Artifact links for active/interrupted/failed steps
- Run links for active/interrupted/failed steps
- Checkpoint records
- Active tool/process handles and process metadata
- Security decisions (approval gates, hard-denies)
- All `OperatorEvent` records
- All `ApprovalGate` records
- Flow lock records

**Compactable state:**
- Completed flow steps: tool result text → summary
- Completed turns: message history → `CompactSummary` artifact
- Old trajectory token events: token text discarded, metadata kept
- Redundant planning discussion already captured in flow/run/artifact records
- Stale context beyond N completed turns

**Future flags (not required in v0.8):**
```
/compact --preview    Show what would be compacted without doing it
/compact --dry-run    Same as preview
/compact --aggressive  Also truncate tool results from completed steps to 200 chars
/compress             Alias if useful for Hermes parity
```

**Hermes translation:**
- Hermes treats context compression as first-class session hygiene.
- EstaCoda adopts the ergonomic direction but anchors it to durable state boundaries.
- `/compact` is a session-hygiene command, not a flow-state mutation command.

### 2.9 Recorder Boundaries (Correction #9)

**Previous:** Proposed adding `recordFlowEvent`, `recordOperatorEvent`, `recordStepEvent` to `TrajectoryRecorder`.

**Corrected:** Do not contaminate `TrajectoryRecorder` with TaskFlow orchestration concerns.

**Boundary:**

| Concern | Owner | Stores |
|---|---|---|
| Flow orchestration events | `TaskFlowStore` (new) | `flow_events`, `operator_events` |
| Run-level evidence | `RunRecorder` (existing) | `runs`, tool plans, failures |
| Provider/tool/turn evidence | `TrajectoryRecorder` (existing) | `trajectories`, token events |
| Artifacts | `ArtifactStore` (existing) | `artifacts` |
| Linkage | `TaskFlowStore` | `flow_artifacts`, `flow_run_links` |

**TaskFlowStore interface:**
```typescript
interface TaskFlowStore {
  // Flow state
  createFlow(flow: Flow): Promise<void>;
  updateFlow(flow: Flow): Promise<void>;
  getFlow(id: FlowId): Promise<Flow | null>;
  listFlows(sessionId?: string): Promise<Flow[]>;

  // Step state
  createStep(step: FlowStep): Promise<void>;
  updateStep(step: FlowStep): Promise<void>;
  getStep(id: StepId): Promise<FlowStep | null>;
  listSteps(flowId: FlowId): Promise<FlowStep[]>;

  // Events
  appendFlowEvent(event: FlowEvent): Promise<void>;
  appendOperatorEvent(event: OperatorEvent): Promise<void>;
  listEvents(flowId: FlowId): Promise<FlowEvent[]>;

  // Linkage
  linkArtifact(flowId: FlowId, stepId: StepId, artifactId: string): Promise<void>;
  linkRun(flowId: FlowId, stepId: StepId, runId: RunId, turnIndex: number): Promise<void>;

  // Locks
  acquireLock(flowId: FlowId, ownerId: string, leaseMs: number): Promise<boolean>;
  releaseLock(flowId: FlowId, ownerId: string): Promise<void>;
  heartbeatLock(flowId: FlowId, ownerId: string, leaseMs: number): Promise<void>;
  recoverStaleLocks(): Promise<number>;
}
```

**TrajectoryRecorder stays unchanged.**

**RunRecorder stays unchanged.**

**ArtifactStore stays unchanged.**

TaskFlow links to them by ID via `flow_run_links` and `flow_artifacts` tables.

### 2.10 Stale Process Visibility (Correction #10)

**Previous:** "v0.8 limitation: We do not attempt to detect or kill stale processes from a previous run."

**Corrected:** Persist process metadata and show stale-process warnings.

**Schema addition:**

```sql
CREATE TABLE flow_processes (
  id TEXT PRIMARY KEY,
  flow_id TEXT NOT NULL,
  step_id TEXT NOT NULL,
  process_manager_id TEXT NOT NULL,
  process_type TEXT NOT NULL,     -- "terminal", "process", "browser"
  command_summary TEXT,           -- truncated command for display
  started_at TEXT NOT NULL,
  expected_exit_at TEXT,          -- if known
  status TEXT NOT NULL,           -- "running", "exited", "orphaned", "unknown"
  FOREIGN KEY (flow_id) REFERENCES flows(id) ON DELETE CASCADE
);

CREATE INDEX idx_fp_flow ON flow_processes(flow_id);
CREATE INDEX idx_fp_step ON flow_processes(step_id);
```

**Behavior:**
1. When `ToolExecutor` starts a process, `TaskFlowEngine` records it in `flow_processes`.
2. When the process exits, the record is updated to `status = "exited"`.
3. On restart recovery:
   - Load active flows.
   - For any flow that was `running` at shutdown, check `flow_processes`.
   - If any process has `status = "running"`, mark it as `orphaned`.
   - Mark the flow as `interrupted`.
   - Show stale-process warning.

**CLI warning:**
```
⚠️  Flow flow_abc123 was interrupted during restart.
    1 process may be orphaned: terminal.run (pid unknown, command: "npm test")
    Please verify no stale processes are running.
```

**Cleanup:**
- v0.8 does not attempt to kill stale processes (unsafe without PID verification).
- v0.8 does warn the operator visibly.
- v0.8 provides `estacoda flow processes <flow-id>` to inspect.

### 2.11 Architecture Ambition: Option A+ Now, Option C-Compatible (Correction #11)

**Previous:** Option C was "deferred" without clear future-path guidance.

**Corrected:** Option C is the **long-term architecture ambition**. v0.8 implements Option A+ with explicit seams for Option C.

**Definitions:**

- **Option A:** TaskFlow above AgentLoop. AgentLoop is a black-box per-turn method.
- **Option A+:** TaskFlow above AgentLoop. AgentLoop exposes a minimal `TurnContext` that includes `flowId`, `stepId`, and an optional `AbortSignal`. AgentLoop remains otherwise unchanged.
- **Option C:** TaskFlow above AgentLoop. AgentLoop exposes step-completed and safe-boundary callbacks so TaskFlow can pause mid-turn or between tool calls.

**v0.8 target:** Option A+.

**Option C callback seam (defined but not fully wired):**

```typescript
type AgentLoopHooks = {
  // v0.8: all optional, all no-ops by default
  onTurnStarted?: (context: { flowId?: FlowId; stepId?: StepId }) => void;
  onTurnCompleted?: (context: { flowId?: FlowId; stepId?: StepId; result: TurnResult }) => void;
  onTurnFailed?: (context: { flowId?: FlowId; stepId?: StepId; error: Error }) => void;
  
  // v0.8.x/v0.9: safe boundary callbacks
  onSafeBoundary?: (context: { flowId?: FlowId; stepId?: StepId }) => {
    shouldPause: boolean;
    shouldInterrupt: boolean;
  };
  
  // v0.9+: finer-grained tool call hooks
  onToolCallStarted?: (toolCallId: string, toolName: string, input: unknown) => void;
  onToolCallCompleted?: (toolCallId: string, result: unknown) => void;
};
```

**AgentLoop contract change (minimal):**

```typescript
// Current
type AgentLoopInput = {
  request: string;
  images?: { path: string }[];
  abortSignal?: AbortSignal;
  // ...
};

// v0.8-corrected (backward-compatible)
type AgentLoopInput = {
  request: string;
  images?: { path: string }[];
  abortSignal?: AbortSignal;
  flowContext?: {           // NEW, optional
    flowId: FlowId;
    stepId: StepId;
    turnIndex: number;
    operatorEvents?: OperatorEvent[];  // steering, etc.
  };
  hooks?: AgentLoopHooks;   // NEW, optional
};
```

**Safe boundary model:**

Even in Option A+, TaskFlow must model safe boundaries explicitly:

```typescript
type SafeBoundary = {
  kind: "turn_complete" | "step_complete" | "tool_complete";
  flowId: FlowId;
  stepId: StepId;
};
```

v0.8 only uses `"turn_complete"` boundaries.
v0.8.x/v0.9 can add `"tool_complete"` without architectural rewrite.

**Hermes translation (careful):**
- Hermes validates the UX need for interrupt, redirect, queue modes, and context compaction.
- Hermes does not dictate EstaCoda's internal architecture.
- EstaCoda adopts the interaction behavior (what the operator feels) but owns the runtime semantics (how state changes).
- Do not claim Hermes uses Option C internally unless verified from source code.

## 5. Changed Command Semantics

### 5.1 /pause (updated)

| Aspect | Previous | Corrected |
|---|---|---|
| When allowed | Flow `running` | Flow `running` with active execution, or `running` without |
| Immediate effect | `status = "paused"` | `pauseRequestedAt = now`, record `operator-pause-requested` |
| Flow status during active execution | `paused` | Stays `running` |
| Safe boundary effect | Implicit | Explicit: transition to `paused`, record `operator-paused` |
| Step status | Immediately `paused` | Stays `running` until safe boundary |

### 5.2 /interrupt (updated)

| Aspect | Previous | Corrected |
|---|---|---|
| When allowed | `running` or `waiting` | `running` **with active execution only** |
| Rejection cases | None specified | `running` without active execution, `waiting`, `paused`, `interrupted` |
| Error message | N/A | "Nothing to interrupt. Use /pause, /cancel, or resolve the wait condition." |

### 5.3 /compact (updated)

| Aspect | Previous | Corrected |
|---|---|---|
| When allowed | Always | Rejected during active execution |
| Allowed states | All | `paused`, `waiting`, `interrupted`, `running` (between turns), no active flow |
| Rejection message | N/A | "Cannot compact: active execution in progress." |
| Non-compactable | Listed | Expanded to include process handles, flow locks, retry chains |

### 5.4 /retry (updated)

| Aspect | Previous | Corrected |
|---|---|---|
| Step model | Same step retried | New step created with `retryOfStepId` |
| Failed step state | Could transition to `pending` | Stays terminal `failed` |
| Retry step | N/A | New `FlowStep` with `attemptNumber = 2+` |
| Event recording | `step-retried` on same step | `step-failed` on old step, `step-created` + `step-started` on retry step |

### 5.5 /approve (updated)

## 6. Changed Architecture Placement Language

### 6.1 Previous Text (Section 2.4)

> "Option C: Hybrid — TaskFlow sits above AgentLoop but AgentLoop exposes 'step completed' callbacks so TaskFlow can pause mid-turn. Why deferred: Requires adding callbacks/hooks to AgentLoop... Can be added in v0.8.x if pause-between-turns proves too coarse."

### 6.2 Corrected Text

> **"Option C: TaskFlow Above AgentLoop with Safe-Boundary Callbacks (Long-Term Ambition)"**
>
> Option C is the intended long-term architecture for EstaCoda. It preserves the clean separation of TaskFlow above AgentLoop while allowing finer-grained operator control (pause between tool calls, interrupt during provider streaming with safe recovery, steer mid-turn).
>
> **v0.8 implements Option A+**, which is Option C-compatible:
> - `AgentLoopInput` accepts an optional `flowContext` and `hooks`.
> - All hooks are optional no-ops in v0.8.
> - The safe-boundary model is explicitly defined with `kind: "turn_complete"` as the only v0.8 boundary.
> - Future versions can add `"tool_complete"` and `"step_complete"` boundaries without architectural rewrite.
> - `TaskFlowEngine` checks `pauseRequestedAt` at every safe boundary.
>
> **Design rule:** No v0.8 decision may block Option C from being implemented later. AgentLoop must not accumulate state that assumes turns are atomic black boxes.

### 6.3 AgentLoop Contract Addition

```typescript
// In AgentLoopInput (backward-compatible, optional)
flowContext?: {
  flowId: FlowId;
  stepId: StepId;
  turnIndex: number;
  operatorEvents?: OperatorEvent[];
};

hooks?: AgentLoopHooks;  // all optional, all no-ops in v0.8
```

## 7. Remaining Open Questions Before Execution Planning

### 7.1 Architecture

1. **Should `AgentLoop` receive `flowContext` as a new field, or should TaskFlow wrap `AgentLoop` in an adapter?**
   - Direct field: simpler, but adds a TaskFlow concept to AgentLoop.
   - Adapter: cleaner separation, but requires an adapter layer.

2. **Should the `AbortSignal` for a flow step be created by TaskFlow or AgentLoop?**
   - TaskFlow creates it and passes to AgentLoop.
   - AgentLoop creates it and TaskFlow signals it.

3. **How does `TaskFlowEngine` know when a turn is "complete" (safe boundary reached)?**
   - `AgentLoop.handle()` returns a `TurnResult` → TaskFlow detects boundary.
   - This is the v0.8 model. Confirm it is sufficient.

### 7.2 Persistence

4. **Should `flow_events` and `operator_events` be in the same table or separate?**
   - Same: simpler queries, but event kinds diverge.
   - Separate: cleaner, but joins needed for full trace.
   - *This correction pass keeps them separate.* Confirm before execution.

5. **Should SQLite WAL mode be required for v0.8 atomic transactions?**
   - WAL improves concurrent read/write but requires file-system support.
   - Current mode may be DELETE journal.

6. **How are schema rollbacks handled if migration fails?**
   - SQLite DDL is not fully transactional.
   - Need backup strategy.

### 7.3 Operator Controls

7. **Should `/pause` while `running` without active execution immediately pause, or still go through `pauseRequested`?**
   - Immediate pause is simpler and equivalent.
   - `pauseRequested` only needed when active execution exists.

8. **Should `/steer` queue steering text for the next turn, or append it immediately to the session messages?**
   - Queue: cleaner, but operator may not see it reflected immediately.
   - Append: immediate visibility, but may arrive mid-turn if not careful.

### 7.4 Security / Approval

9. **Does `WorkspaceApprovalController` need modification to support flow-step scope, or can TaskFlow map step-level gates to existing tool+target grants?**
   - Reuse: map step gate to tool+target grant.
   - Extend: add flow-step scope to controller.

10. **Should hard-denies be recorded in `approval_gates` at all, or should they bypass the gate entirely?**
    - Bypass: gate only for "ask" decisions.
    - Record: gate records all security decisions for audit.

### 7.5 Process Ownership

11. **Should `ToolExecutor` be modified to return process IDs, or should `ProcessManager` expose a "last started process" query?**
    - Return: cleaner, but changes ToolExecutor contract.
    - Query: no contract change, but race-prone.

12. **What is the `command_summary` truncation limit for `flow_processes`?**
    - Proposal: 200 chars.

### 7.6 Testing

13. **Should the `FakeTaskFlowRunner` simulate AgentLoop as a black box, or mock provider + tool execution separately?**
    - Black box: simpler, but less granular.
    - Separate mocks: more control, but more setup.

14. **What is the minimum acceptable eval fixture count for v0.8?**
    - Proposal: 15 fixtures (success, wait, pause-request, pause-effect, interrupt, cancel, approve, reject, retry, skip, compact, checkpoint, restart, security-non-bypass, hard-deny-no-approve, stale-lock, failed-cleanup).

### 7.7 Documentation

15. **Should ADR-0004 cover the full state machine, or only the architecture decision (Option A+ / Option C)?**
    - Proposal: ADR-0004 covers architecture decision and callback seam.
    - State machine details go in `docs/architecture/taskflow.md`.

16. **Should the RFC corrections be merged into the main RFC file, or kept as a separate delta?**
    - Proposal: Merge after review approval. Keep delta during review.

### 7.8 v0.9 Handoff

17. **Which v0.8 schema fields must be stable for v0.9 channel integration?**
    - `flows.status` enum must be stable.
    - `flow_events.kind` enum must be stable or append-only.
    - `flow_locks` table must be stable.

18. **Should v0.8 `TaskFlowStore` interface be designed to support multiple concurrent flows (even if v0.8 only runs one)?**
    - Proposal: Yes. Interface accepts `flowId` everywhere. Implementation may serialize in v0.8.
