## 5. Architecture Risks

### Risk 1: Session Leakage Across Surfaces
**Current state:** CLI sessions and channel sessions are naturally separate because they use different session ID generation paths (`cli-${workspace}` vs `channel-${platform}-...`). There is NO mechanism to attach them.
**Risk:** When handoff is implemented, a bug could cause the `ChannelSessionStore` to overwrite a CLI session mapping or vice versa. The v0.9 plan requires explicit attach/detach.
**Mitigation:** Keep session stores separate (SQLiteSessionDB for messages, `cli-sessions.json` for CLI workspace mapping, `channel-sessions.json` for channel keys). Handoff should only create a **pointer** in the channel store pointing to an existing CLI session ID — never merge the stores.

### Risk 2: Channel Auth Weaknesses
**Current state:** Telegram auth is allowlist-based with optional pairing codes. There is no unified auth model.
**Risk:** Each new channel reinvents auth. Discord servers/guilds have different identity models. Email has no persistent user ID — only address.
**Mitigation:** Create a `ChannelIdentity` normalization layer that maps platform-specific identities (Discord snowflake, WhatsApp JID, email address) to a canonical `userId` for allowlist checks.

### Risk 3: Cron Duplicate Execution
**Current state:** Only a `.tick.lock` file prevents concurrent ticks. No per-job execution lock. If a job takes longer than the tick interval, or if the process crashes mid-job and restarts, the same job could run twice.
**Risk:** Silent duplicate execution for scheduled tasks.
**Mitigation:** Add per-job execution status (`idle` | `running` | `completed`) in the store, with a heartbeat timestamp. On restart, jobs with stale `running` status should be marked `failed` or `stalled`, not re-run automatically unless explicitly configured.

### Risk 4: Email Actions Becoming Over-Restricted
**Current state:** No email implementation exists.
**Risk:** The v0.9 plan says "Email should not be stricter than Hermes." If implemented with extra approval layers (e.g., "email-triggered actions require manual approval"), it will violate the parity rule.
**Mitigation:** Email adapter should use the SAME security policy as CLI/Telegram. No email-specific approval gate. Allowed senders and home address are the only email-specific controls.

### Risk 5: Telegram-Only Assumptions Blocking Other Channels
**Current state:** `ChannelGateway` has Telegram-specific formatting hardcoded: `adapter.kind === "telegram" ? "html" : "plain"` and `telegramGatewayCommands()`.
**Risk:** Adding Discord/WhatsApp/Email requires forking behavior inside `ChannelGateway`.
**Mitigation:** Move adapter-specific formatting to the adapter itself, not the gateway. The gateway should call `adapter.delivery.sendText()` with raw text and let the adapter decide formatting.

### Risk 6: CLI-Only Approval Assumptions
**Current state:** Approvals work in Telegram via inline buttons and callback queries. The `ChannelGateway` handles this.
**Risk:** Discord has button interactions, WhatsApp has limited interactivity, Email has no real-time interactivity.
**Mitigation:** Approval delivery should degrade gracefully: Discord uses buttons, WhatsApp uses text commands, Email sends a reply asking the user to reply "approve" or "deny".

### Risk 7: Recorder Gaps for Channel-Triggered Work
**Current state:** `RunRecorder` captures `channel` in `recordRouteUsage` and records all events to `SessionDB`. This already works for any channel.
**Risk:** Cron-triggered runs use fresh sessions but the `RunRecorder` may not capture the `origin` (channel, chatId) in the trajectory.
**Mitigation:** Ensure `createRuntimeCronRunner` passes `origin` metadata to the runtime so it gets recorded.

### Risk 8: Config/Env Fragmentation
**Current state:** Each capability (provider, image, voice, telegram, MCP) has its own config section and env var pattern.
**Risk:** Adding 3 new channels multiplies env vars: `DISCORD_BOT_TOKEN`, `WHATSAPP_SESSION_SECRET`, `EMAIL_IMAP_PASSWORD`, `EMAIL_SMTP_PASSWORD`, etc.
**Mitigation:** Follow the existing pattern: `runtime-config.ts` has `botTokenEnv` for Telegram; do the same for Discord (`botTokenEnv`), WhatsApp (`sessionSecretEnv`), Email (`imapPasswordEnv`, `smtpPasswordEnv`).

### Risk 9: Test Fragility
**Current state:** Zero unit tests. Only smoke tests and eval fixtures.
**Risk:** Adding 3 complex channels with no unit tests means regressions will only be caught late.
**Mitigation:** v0.9 MUST include unit tests for: session key normalization, handoff code lifecycle, cron schedule parsing, delivery target parsing, operator command parsing. Use fakes/mocks for external platform APIs.

## 6. Suggested Adjustments to the v0.9 Plan

### Adjustment 1: WhatsApp Integration Path Must Be Decided Before Planning
The v0.9 overview says "final implementation plan must decide whether v0.9 uses official WhatsApp Business Cloud API or linked-device / bridge-style local integration." This decision is **blocking**. Recommend:
- **Option A (Recommended):** Defer WhatsApp to v0.9.1 or v0.10. The v0.9 plan already has 3 major new channels (Discord, Email, Telegram hardening) plus cron hardening and cross-surface sessions. WhatsApp adds the most integration risk.
- **Option B:** If WhatsApp is mandatory, use a **minimal bridge approach** (e.g., `whatsapp-web.js` or `baileys`) with explicit "linked device" setup, NOT the Business Cloud API. Document the ToS risk clearly.

### Adjustment 2: Reduce "Must-Have" Channels to Discord + Email + Telegram Hardening
The original v0.9 overview lists Telegram, Discord, WhatsApp, and Email. Given the codebase reality (zero channel infrastructure beyond Telegram), implementing 3 new channels simultaneously is high-risk. Suggest:
1. **Track B:** Telegram hardening (pairing reliability, diagnostics, run records)
2. **Track C:** Discord gateway (developer/operator channel — lower risk than WhatsApp)
3. **Track E:** Email gateway (Hermes parity — well-understood semantics)
4. **Defer D (WhatsApp)** to v0.9.1 unless the user explicitly demands it.

### Adjustment 3: Cron Store Should Stay JSON for v0.9, But Execution History Must Be SQLite
The v0.9 plan calls for a persistent scheduled task store. The existing `CronStore` already provides this (JSON). Rather than rewriting it to SQLite (which is a should-have), add a **separate SQLite execution history table** in the existing `SQLiteSessionDB`. This is cheaper and more reliable than JSON for history.

### Adjustment 4: Handoff Codes Should Be Simple, Not Cryptographic
The v0.9 plan proposes a `HandoffCode` type with expiry. The existing codebase already has `TelegramPairingInput` with `code` and `ttlMinutes`. Reuse this pattern: a simple alphanumeric code stored in `~/.estacoda/handoff-codes.json` with expiry. No need for crypto-grade tokens.

### Adjustment 5: Delivery Router Is a Real Gap — Do Not Hardcode
Current cron delivery is hardcoded:
```typescript
if (job.delivery === "origin" && job.origin?.channel === "telegram") { ... }
if (job.delivery.startsWith("telegram:")) { ... }
```
This will become unmanageable with 4 channels. The v0.9 plan correctly identifies `DeliveryRouter` as a shared component. **This is not optional — it is required before adding Discord/Email.**

### Adjustment 6: Gateway Status Must Become Multi-Channel
Current `getTelegramGatewayDiagnostics` is Telegram-only. The v0.9 plan requires a unified gateway status. Rename/refactor to `getGatewayDiagnostics` that iterates all configured adapters and reports health per channel.

### Adjustment 7: `/switch` Should Become `/attach` + `/detach`
Current gateway command is `/switch <session-id>` which changes the current channel's session pointer. The v0.9 plan wants `/attach` (point channel to existing session) and `/detach` (create new independent session). This is a UX rename + behavior change. `/switch` can be aliased to `/attach` for backward compatibility.

## 9. Open Questions / Decisions Needed

1. **WhatsApp integration path:** Defer to v0.9.1, or commit to linked-device bridge (with documented ToS risk)?
2. **Cron store migration:** Keep JSON for jobs, add SQLite for history? Or migrate jobs to SQLite too?
3. **Discord interactivity:** Use slash commands (requires Discord API application command registration) or prefix commands (e.g., `/status`)? Prefix is faster to implement but worse UX.
4. **Email library:** Use `imap` + `nodemailer` npm packages, or raw Node sockets? `nodemailer` is the pragmatic choice.
5. **Delivery target format:** The current `CronDelivery` type is `"local" | "origin" | string`. The v0.9 plan needs structured targets. Propose `cli` | `telegram:<chatId>` | `discord:<channelId>` | `whatsapp:<number>` | `email:<address>` | `local` | `silent`.
6. **Gateway process model:** Current gateway is a foreground process (`bun run dev -- gateway start`). Should v0.9 add a daemon/service mode? Suggest defer to v0.10; document systemd/Docker approach instead.
7. **Session handoff security:** Should handoff codes require the user to already be on the allowlist for the target channel? Yes — otherwise an unpaired user could steal a session with a guessed code.

## 10. Final Recommendation

### v0.9 plan needs revision before execution planning.

**Reasoning:**
- The v0.9 overview is ambitious and well-architected, but it asks for **4 channels + cron hardening + cross-surface sessions + operator controls + tests/docs** in a single version.
- The repository has **zero unit tests**, **no test runner**, and **only one live channel** (Telegram).
- WhatsApp carries the highest integration risk and should be scoped separately.
- The shared gateway core (`ChannelGateway`) is actually well-designed and channel-agnostic, which is a major asset. But it has Telegram-specific formatting leaks that need fixing before scaling.
- Cron has solid foundations but lacks execution history and duplicate prevention — these are genuine v0.9 blockers.

**Recommended scope revision:**
1. **v0.9 core:** Telegram hardening + Discord + Email + Cron reliability + Cross-surface sessions + Operator controls + Tests/docs.
2. **v0.9.1:** WhatsApp gateway (after v0.9 proves the multi-channel abstraction).
3. **v0.10:** Gateway service mode, Slack, advanced delivery routing.

**If the user insists on WhatsApp in v0.9:** Accept only a **minimal DM-only linked-device bridge** with explicit "experimental" labeling and documented platform risk.

**Readiness for execution planning:** The codebase is ready for v0.9 planning **after** the scope revision above. The shared gateway architecture is sound. The main risks are integration (Discord bot registration, email IMAP/SMTP setup) and testing (zero unit test infrastructure).
