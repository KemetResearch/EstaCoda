### Allowable content boundary

These pages may now document the full implemented behavior. They must not document behavior that is planned but not implemented. The implemented behavior is the ceiling.

Specifically:
- `estacoda update --apply` with `ESTACODA_UPDATE_ARTIFACT` is **not** a v0.1.0 launch path. The artifact-only apply path in `update-engine.ts` remains reachable but is not the primary update behavior. Docs should guide users to `estacoda update` (managed-source apply) or package-manager routing.
- Binary artifact update is **deprecated** for v0.1.0 public claims. Do not present it as a recommended path.
- The `estacoda update` default behavior (no flags) performs a dry-run check for non-managed-source installs and applies updates for managed-source installs. This is the primary path to document.

### Direct-source fallback commands

These are valid but are not the primary public commands:

```bash
# Direct-source fallback (pre-release or debug)

# Local script form (only for users who already cloned the repo)

```

Public docs advertise `https://estacoda.kemetresearch.com/install.sh`. The raw GitHub URL is a direct-source fallback for internal use, debugging, or pre-release testing where the website launcher has not been pinned yet.

### Installer entrypoint architecture rule

```text
Repo owns installer logic. Website owns stable public entrypoint.
Docs advertise the website entrypoint.
Raw GitHub URL remains a direct-source fallback.
The website endpoint must be a thin launcher that downloads and executes scripts/install.sh from the EstaCoda repo.
The installer implementation must not depend on estacoda.kemetresearch.com.
That would create circular ownership.
```

### Thin launcher reference

The website endpoint should be a thin launcher, not a copy of the installer:

```bash
#!/usr/bin/env bash
set -euo pipefail

ESTACODA_INSTALL_URL="${ESTACODA_INSTALL_URL:-https://raw.githubusercontent.com/KemetResearch/EstaCoda/main/scripts/install.sh}"

tmp="$(mktemp)"
cleanup() {
  rm -f "$tmp"
}
trap cleanup EXIT

curl -fsSL "$ESTACODA_INSTALL_URL" -o "$tmp"
bash "$tmp" "$@"
```

Before v0.1.0 tag: launcher target = `https://raw.githubusercontent.com/KemetResearch/EstaCoda/main/scripts/install.sh`
After v0.1.0 tag: launcher target = `https://raw.githubusercontent.com/KemetResearch/EstaCoda/v0.1.0/scripts/install.sh`

## 6B. Update Docs Follow-Up Items

These items must be addressed in `getting-started/updating.md`, `operations/update-operations.md`, and `reference/cli-commands.md` (EN + AR).

### Update command surface

| Flag | Behavior | Exit codes |
|---|---|---|
| `estacoda update --check` | Check only. Never modify files. | 0 if update available, 2 if up-to-date, 1 on error |
| `estacoda update` | Default safe apply. Managed-source: fetch, check ff-only, check worktree cleanliness, pull, install deps, build, validate. Non-managed: print routing message. | 0 on success/routing, 1 on error, 3 on dirty worktree |
| `estacoda update --yes` | Apply without interactive confirmation where appropriate. Still refuses unsafe operations (non-ff pull, non-managed dir). | Same as default |
| `estacoda update --backup` | Force backup before applying. | Same as default |
| `estacoda update --no-backup` | Skip backup if user is confident. Not recommended. | Same as default |
| `estacoda update --gateway` | Non-interactive gateway/service update mode. Logs to `~/.estacoda/logs/update.log`. Restarts managed gateway service on success, or prints manual restart instruction. | Same as default |

### Install-method routing to document

| Method | `estacoda update` behavior |
|---|---|
| managed-source | Guarded source update: git fetch, ff-only check, worktree check, pull, pnpm install, build, validate. Rollback to pre-pull SHA on build failure. |
| manual-source | Check and advise only. Prints `git fetch origin && git status`. No self-mutation. |
| homebrew | Prints `brew upgrade kemetresearch/tap/estacoda`. No self-mutation. |
| docker | Prints `docker pull ghcr.io/kemetresearch/estacoda:latest`. No self-mutation. |
| npm-global | Prints `npm install -g estacoda@latest`. No self-mutation. |
| pnpm-global | Prints `pnpm add -g estacoda@latest`. No self-mutation. |
| unknown | Prints contributor-safe reinstall guidance. No self-mutation. |

### Startup update checks to document

- Enabled by default for v0.1.0
- Non-blocking background prefetch after session init
- Only runs in interactive CLI sessions (no args, TTY available)
- 6-hour cache (`~/.estacoda/update-cache.json`)
- Silent failure: network errors do not interrupt the session
- Startup hint rendered when cached status says "update-available"
- Only checks for managed-source and manual-source installs; package-manager installs skip prefetch

### Safety boundaries to document

- No `git reset --hard` outside managed installer directories
- Dirty worktree refusal by default (exit 3)
- `--yes` may auto-stash, proceed, and restore stash on failure (managed-source only)
- Managed clone directories may reconcile to origin/main only if `.install-method.json` stamp proves ownership
- User state under `~/.estacoda` is never destroyed by update
- Build failure triggers automatic rollback to pre-pull SHA
- Gateway mode never restarts user processes blindly; only managed services

### State paths to document

- `~/.estacoda/update-cache.json` â€” update check cache (global, not profile-local)
- `~/.estacoda/logs/update.log` â€” gateway mode update log
- `~/.estacoda/.backups/<label>/` â€” user state backup (created before managed-source mutation when backup mode is default or force)

## 6C. Uninstall Docs Follow-Up Items

These items must be addressed in `getting-started/uninstall.md`, `reference/cli-commands.md`, `reference/troubleshooting.md`, and `operations/backups-and-state.md` (EN + AR).

### Uninstall command surface

| Flag | Behavior |
|---|---|
| `estacoda uninstall` | Default keep-data mode. Removes managed-source code, known wrappers, installer-owned PATH entries, gateway services. Preserves `~/.estacoda`. |
| `estacoda uninstall --purge --yes` | Removes user data too. Requires both flags. `--yes` alone does not purge. |

### Method-specific behavior to document

| Method | Default uninstall behavior |
|---|---|
| managed-source | Gateway teardown first. Remove known wrappers/PATH lines. Remove managed install dir (only if stamp is trusted and installDir is safe). Preserve `~/.estacoda`. |
| manual-source | Gateway teardown first. Remove known wrappers/PATH lines. Leave the clone alone. Preserve `~/.estacoda`. |
| npm-global | Print `npm uninstall -g estacoda`. Package-manager-owned wrappers are not manually removed. |
| pnpm-global | Print `pnpm remove -g estacoda`. Package-manager-owned wrappers are not manually removed. |
| homebrew | Print `brew uninstall estacoda`. `brew untap kemetresearch/tap` is not automatic. |
| docker | Print container/image guidance. Containers, images, and volumes are not removed automatically. |
| unknown | Remove only known source wrappers/PATH lines. Preserve user data. |

### Ownership and trust to document

- Managed-source uninstall requires a valid `.install-method.json` stamp
- Stamp must declare `method: managed-source` with `sourceUrl`, branch metadata, and `installDir`
- Stamp path must resolve inside the declared `installDir`
- `installDir` must be safe: not `/`, not `$HOME`, not `~/.estacoda`, and basename must be `estacoda`, `estacoda.git`, or `estacoda-source`
- Ambiguous installs are treated as manual-source. A plain git checkout is never deleted just because it looks like EstaCoda.

### Purge behavior to document

- `--purge --yes` removes user data after gateway teardown and install-code cleanup
- If other named profiles exist, the active profile is removed but other profiles are preserved
- Bulk named-profile removal requires a future explicit flag
- Full purge removes `~/.estacoda` only when no other named profiles exist

### Gateway teardown to document

- Happens before code or user-state removal
- Uses the existing service-manager abstraction (`uninstallService`)
- No raw `pkill`, `killall`, `systemctl`, or `launchctl` calls in the uninstall path
- Termux: system service removal is skipped; known `$PREFIX/bin/estacoda` wrappers are removed best-effort

### Wrapper cleanup to document

Known wrapper paths inspected:
- `~/.local/bin/estacoda`
- `~/.estacoda/bin/estacoda`
- `/usr/local/bin/estacoda`
- `$PREFIX/bin/estacoda` (Termux)

Wrappers are removed only if they contain known markers ("Generated by scripts/install.sh", "EstaCoda source wrapper", etc.) or symlink to known EstaCoda targets. Non-EstaCoda files are left in place.

### PATH cleanup to document

- Inspects `.bashrc`, `.zshrc`, `.profile`, `.bash_profile`
- Removes lines containing `PATH=` and EstaCoda installer markers
- Only removes installer-added PATH entries, not user-customized ones

### Unsupported behavior to document

- Native Windows uninstall is not supported in v0.1.0
- WSL2 remains best-effort
