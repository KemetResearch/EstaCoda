## 2. Existing Building Blocks for TaskFlow

### 2.1 Runtime Loop

**What exists:** `AgentLoop` class in `src/runtime/agent-loop.ts` (~700 lines). Decomposed into six components:

- `RuntimeRouter` — intent + skill selection
- `ProviderTurnLoop` — provider interaction and tool-call streaming
- `ToolCallPlanner` — tool plan creation from provider output
- `ToolPlanRunner` — sequential tool plan execution with dependency awareness
- `NativeToolExecutor` — deterministic native tool execution (image generation)
- `SkillWorkflowExecutor` — skill workflow step execution
- `RunRecorder` — structured run/trajectory recording

**Reusability for TaskFlow:** High. The decomposition means TaskFlow can orchestrate these same components rather than reimplementing them. However, `AgentLoop.handle()` is a single-turn method — it does not maintain state across turns. TaskFlow would need to sit *above* or *around* `AgentLoop`.

### 2.2 Tool Planning

**What exists:** `ToolCallPlan` type and `ToolCallPlanner` in `src/contracts/tool-plan.ts` and `src/tools/tool-call-planner.ts`.

- Plans have `id`, `tool`, `input`, `source`, `status` (planned|invalid|executed|blocked|unavailable|cancelled)
- `ToolPlanRunner` executes plans sequentially with dependency awareness (reads `dependsOn` field)

**Reusability:** High. TaskFlow steps can reuse `ToolCallPlan` as the step execution unit. The status enum already includes `cancelled`, which is a positive sign.

### 2.3 Tool Execution

**What exists:** `ToolExecutor` in `src/tools/tool-executor.ts`.

- Handles security assessment, validation, execution, result storage
- Records events to `SessionDB` and `TrajectoryRecorder`
- Accepts `AbortSignal` for cancellation
- Returns `ToolExecutionRecord` with decision, risk class, target key, result

**Reusability:** High. TaskFlow step execution should delegate to `ToolExecutor` to avoid duplicating security policy.

### 2.4 Cancellation

**What exists:**

- `AbortController`/`AbortSignal` passed through `AgentLoopInput`, `ProviderTurnLoop`, `ToolExecutor`
- SIGINT handler in `session-loop.ts` aborts the active turn
- `ToolExecutor` checks `signal.aborted` before executing the tool
- `cancelledResponse()` and `buildResumeNote()` in `response-builders.ts` handle cancelled-turn UX
- `RuntimeEvent` includes `agent-cancelled` kind

**Reusability:** Medium. Turn-level cancellation exists, but there is no flow-level cancellation. A TaskFlow would need its own cancellation semantics that cascade to the active turn.

### 2.5 Resume

**What exists:**

- `/resume` CLI command shows the latest interrupted-turn resume note
- `buildResumeNote()` constructs a text summary of where the turn was interrupted
- Resume notes include: stage, original request, skill, provider, completed tools, planned tools, context refs

**Reusability:** Low for true flow resume. Current resume is a *text hint* for the user to manually continue. There is no structured resumption of an interrupted multi-step flow. The `SQLiteSessionDB` stores events, but there is no mechanism to reconstruct execution state from events.

### 2.6 Session Persistence

**What exists:** `SQLiteSessionDB` in `src/session/sqlite-session-db.ts` (~540 lines).

- Stores: sessions, messages, events, trajectories, failures, memory promotions
- Schema supports `SessionEvent` union with 10+ event kinds
- Trajectories stored as JSON blobs with event arrays
- `InMemorySessionDB` for testing

**Reusability:** High for event storage. However, there is no "flow state" table or concept. TaskFlow persistence would require new schema additions.

### 2.7 Run/Trajectory Recorder

**What exists:**

- `RunRecorder` in `src/runtime/run-recorder.ts` (~670 lines) — records run metadata, tool plans, tool results, skill outcomes, artifacts, provider execution, failures
- `TrajectoryRecorder` in `src/trajectory/trajectory-recorder.ts` (~100 lines) — lightweight event sink for runtime events
- 32 trajectory event kinds defined in `src/contracts/trajectory.ts`
- CLI `estacoda trace list/dump/timeline/failures`

**Reusability:** High. TaskFlow should emit to the same recorders rather than duplicating them. The `RunRecorder` already has `recordToolPlan`, `recordToolResult`, `recordFailure`, `recordArtifact`.

### 2.8 Artifact Recorder

**What exists:** `ArtifactStore` in `src/artifacts/artifact-store.ts` (~55 lines).

- In-memory `Map<string, ArtifactRecord>`
- Records path, kind, bytes, summary, mimeType, metadata
- No persistence — artifacts live only for the runtime lifetime

**Reusability:** Low for durable flows. TaskFlow needs artifact persistence across restarts. The current store is too thin and ephemeral.

### 2.9 Approval Controller

**What exists:** `WorkspaceApprovalController` in `src/security/workspace-approval-controller.ts` (~380 lines).

- Supports scopes: `once`, `session`, `always`
- Persistent grants stored in JSON file
- `grantApproval`, `revokeApproval`, `inspectApprovals`
- Integrated with security policy via `assessSecurityPolicy`

**Reusability:** High for step-level approvals. TaskFlow approval gates should reuse this controller. However, there is no "flow-scoped" approval concept.

### 2.10 Security/Risk Policy

**What exists:** `security-policy-factory.ts` (~420 lines).

- Modes: `strict`, `adaptive`, `open`
- Hard safety floor (destructive commands blocked regardless of mode)
- Auxiliary provider assessor for ambiguous cases
- Risk classes: `destructive-local`, `workspace-write`, `external-side-effect`, `credential-access`, `sandbox-escape`, `spend-money`

**Reusability:** High. TaskFlow steps must pass through the same security policy.

### 2.11 Skill Workflow Metadata

**What exists:** `SkillWorkflowExecutor` in `src/runtime/skill-workflow-executor.ts` (~270 lines).

- Executes skill workflow steps sequentially
- Steps have `description`, `condition`, `action`, `toolset`, `maxAttempts`
- Supports step telemetry and outcome recording
- `workflowMode: advisory` (default) — skill teaches, agent decides

**Reusability:** Medium. This is skill-level workflow execution within a single turn. TaskFlow is a *runtime* orchestration layer that may span multiple turns and skills. They operate at different levels but share concepts like step sequencing and retry.

### 2.12 Curator/Evolution Telemetry

**What exists:**

- `SkillEvolutionStore` — JSONL-based proposal storage
- `ChangeManifestStore` — JSONL-based manifest storage
- `SkillLearningManager` — autonomy configuration
- `OptimizationDataset` export

**Reusability:** Medium. Evolution telemetry records skill-level observations. TaskFlow would need flow-level telemetry (step state transitions, operator interventions).

### 2.13 CLI Commands

**What exists:** `session-loop.ts` slash commands:

- `/status`, `/model`, `/tools`, `/skills`, `/memory`, `/security`, `/approvals`, `/resume`, `/sessions`, `/search`, `/switch`, `/reset`, `/new`, `/reload-mcp`, `/cron`, `/trust`, `/untrust`, `/yolo`, `/revoke`, `/doctor`, `/clear`, `/exit`

**Reusability:** Medium. CLI is turn-oriented. TaskFlow needs new flow-oriented commands.

### 2.14 Smoke/Eval Infrastructure

**What exists:**

- Smoke runner (`src/smoke/smoke-runner.ts`) — sequential case execution
- Eval runner (`src/eval/eval-runner.ts`) — assertion-based case execution
- 18 deterministic eval fixtures
- No unit tests

**Reusability:** High. TaskFlow evals should plug into the same eval runner.

## 4. Required Runtime Semantics

### 4.1 Safe Interruption

**Requirement:** Stop active execution immediately but safely, persist interruption, avoid corrupting flow state.

**Current state:** Abort signal cancels the current tool/provider call, but:
- There is no "interrupted" state persisted to the database
- `ToolExecutor` returns a synthetic "cancelled" result when aborted, but does not record an "interrupted" event
- Child processes spawned via `ProcessManager` are NOT tracked by the abort signal — a `terminal.run` process could continue after the turn is aborted

**Gap:** Major. Interruption needs to:
1. Cancel the active provider call
2. Cancel any active tool execution
3. Signal the `ProcessManager` to stop related child processes
4. Persist flow state as "interrupted" with a snapshot
5. Record the interruption in the trajectory

### 4.2 Safe Pause Boundary

**Requirement:** Stop progression at the next safe boundary without cancelling.

**Current state:** No pause concept exists.

**Gap:** Major. Need to define safe boundaries:
- Between tool calls? (within a turn)
- Between provider turns? (between LLM calls)
- Between skill workflow steps?
- Between TaskFlow steps?

The boundary choice affects complexity. Between turns is easiest; between tool calls requires refactoring `AgentLoop`.

### 4.3 Resumable Flow State

**Requirement:** Continue a paused, waiting, or interrupted flow.

**Current state:** Resume is a text note only.

**Gap:** Major. Need:
- Flow state machine with states: `pending`, `running`, `paused`, `waiting`, `interrupted`, `completed`, `cancelled`, `failed`
- Step state machine with similar states
- Durable storage of flow + step state
- Rehydration logic to reconstruct a flow from stored state
- Context reconstruction (what messages to send to the provider on resume)

### 4.4 Cancellation State

**Requirement:** Terminate flow cleanly and persist terminal cancelled state.

**Current state:** Turn cancellation is transient.

**Gap:** Major. Need persistent flow cancellation with reason, timestamp, operator identity.

### 4.5 Operator Event Log

**Requirement:** Record all operator interventions as auditable events.

**Current state:** No operator event log. `TrajectoryRecorder` has 32 event kinds, none are operator commands.

**Gap:** Major. Need new trajectory event kinds: `operator-pause`, `operator-resume`, `operator-interrupt`, `operator-cancel`, `operator-steer`, `operator-approve`, `operator-reject`, `operator-retry`, `operator-skip`, `operator-checkpoint`.

### 4.6 Flow-Safe Compaction

**Requirement:** Compress session/context without losing active flow state.

**Current state:** No compaction mechanism.

**Gap:** Major. Need to define what is compressible:
- Old completed flow steps? (keep summary, discard full tool results)
- Old trajectory events? (keep failure/events, discard token streams)
- Message history beyond a window? (summarize instead of raw messages)

Must not lose: active flow state, pending steps, approvals, waits, artifacts, run links, operator interventions, retry state, unresolved decisions.

### 4.7 Approval/Rejection Persistence

**Requirement:** Persist approval decisions for blocked steps.

**Current state:** `WorkspaceApprovalController` persists `once`, `session`, `always` grants.

**Gap:** Minor. Works for tool-level approvals. Flow-level approval gates need to record the specific step/flow context, not just tool name.

### 4.8 Retry Policy

**Requirement:** Retry failed steps according to policy and idempotency constraints.

**Current state:** `ToolPlanRunner` has basic `maxAttempts` for skill workflow steps.

**Gap:** Medium. Need configurable retry policy (backoff, max attempts, idempotency check, failure class whitelist/blacklist).

### 4.9 Skip Policy

**Requirement:** Skip a step only if policy allows it.

**Current state:** No skip mechanism.

**Gap:** Major. Need skip rules (which step types are skippable, what happens to downstream dependencies).

### 4.10 Checkpoint Persistence

**Requirement:** Persist named recovery points.

**Current state:** No checkpoints.

**Gap:** Major. Need checkpoint schema (name, flow state, step states, context snapshot, artifact refs).

### 4.11 Trace Inspection

**Requirement:** Display flow/run/event trail.

**Current state:** `estacoda trace timeline` shows trajectory events.

**Gap:** Minor. Need flow-aware trace views (show steps, state transitions, operator events).

### 4.12 Active Tool/Process Cleanup

**Requirement:** Clean up running tools/processes on interrupt/cancel.

**Current state:** `ProcessManager` can `stop(id, signal)` processes, but:
- There is no linkage between a turn/flow and the processes it started
- `ToolExecutor` does not track process IDs
- `AgentLoop` does not know about processes started by tools

**Gap:** Major. Need process ownership (which flow/turn started which process) and cascade cleanup.

### 4.13 State Preservation Across Restart

**Requirement:** Flow survives process restart.

**Current state:** `SQLiteSessionDB` persists messages and events, but no flow state.

**Gap:** Major. Need flow state table, step state table, and rehydration logic.

## 6. Architecture Risks

### 6.1 Tight Coupling Risks

**Risk:** TaskFlow could become tightly coupled to `AgentLoop` internals.

**Evidence:** `AgentLoop.handle()` is a monolithic turn method. TaskFlow needs to either:
- (A) Wrap `AgentLoop` at a higher level (cleaner, but limits step granularity)
- (B) Decompose `AgentLoop` further to expose step-level hooks (more invasive)

**Mitigation:** Prefer option A — TaskFlow orchestrates turns, not individual provider calls. This preserves the decomposition already achieved in v0.4.

### 6.2 Duplicated Persistence Concepts

**Risk:** TaskFlow might create a parallel persistence layer instead of extending `SQLiteSessionDB`.

**Evidence:** `SQLiteSessionDB` already has sessions, messages, events, trajectories, failures. Adding flows, steps, and flow events should be schema extensions, not a new database.

**Mitigation:** Extend `SQLiteSessionDB` with flow tables. Do not create a separate `FlowDB`.

### 6.3 TaskFlow Bypassing Security Policy

**Risk:** If TaskFlow executes tools directly instead of delegating to `ToolExecutor`, it could bypass security assessments.

**Evidence:** `ToolPlanRunner` currently calls `toolExecutor.executeTool()` for each plan. TaskFlow must maintain this delegation.

**Mitigation:** Mandate that TaskFlow step execution always routes through `ToolExecutor` or `ToolPlanRunner`. Document this as an invariant.

### 6.4 Operator Commands Bypassing Approval Policy

**Risk:** An operator `/steer` or `/skip` could override a security denial.

**Evidence:** There is no framework for validating operator commands against security policy.

**Mitigation:** Define that operator commands are subject to the same risk classification as agent actions. A `/skip` of a destructive step should require approval.

### 6.5 /compact Losing Active Execution State

**Risk:** Compaction could drop pending approvals, active waits, or unresolved decisions.

**Evidence:** No compaction rules exist. This is a green-field feature.

**Mitigation:** Define an explicit "non-compactable state" inventory. Write a compact-audit log that records what was summarized vs. dropped.

### 6.6 /interrupt Leaving Tools/Processes Running

**Risk:** Interrupting a flow might not clean up child processes.

**Evidence:** `ProcessManager` tracks processes independently. `ToolExecutor` does not report process IDs back to the caller. `AgentLoop` has no process cleanup on abort.

**Mitigation:** Add process ownership (link processes to flow/step IDs). On interrupt, cascade stop signals to owned processes.

### 6.7 /steer Becoming Hidden Prompt Mutation

**Risk:** Operator steering could be implemented as hidden prompt injection instead of auditable events.

**Evidence:** The roadmap explicitly warns against this.

**Mitigation:** `SteerEvent` must be a first-class trajectory event. Steering text must be visible in traces, not hidden in system prompt construction.

### 6.8 TaskFlow Duplicating Trajectory Recorder

**Risk:** TaskFlow might create its own event recording instead of using `TrajectoryRecorder`.

**Evidence:** `TrajectoryRecorder` only has 4 record methods. It may need extension.

**Mitigation:** Extend `TrajectoryRecorder` with flow-aware record methods rather than creating a parallel system.

### 6.9 Skills Incorrectly Turned Into Rigid Workflows

**Risk:** The existence of TaskFlow might encourage making every skill a rigid enforced workflow.

**Evidence:** `ADR-0003` explicitly states skills should stay advisory (`workflowMode: advisory`).

**Mitigation:** Document that TaskFlow is for *runtime* orchestration, not *skill* enforcement. Skills remain Markdown-first and advisory.

### 6.10 Background Autonomy Leaking Into v0.8

**Risk:** v0.8 scope could expand to include background execution, scheduled flows, or channel-triggered flows.

**Evidence:** Cron exists. Telegram channel exists. The roadmap explicitly assigns channels and automations to v0.9.

**Mitigation:** Strictly exclude channel integration, cron-driven flows, and background execution from v0.8. TaskFlow runs in the foreground within an active session only.

## 7. Recommended v0.8 Design Questions

Before writing the execution plan, the following questions must be answered:

### 7.1 Scope and Boundaries

1. Does v0.8 TaskFlow operate *above* `AgentLoop` (orchestrating turns) or *inside* `AgentLoop` (orchestrating steps within a turn)?
2. Is a TaskFlow scoped to a single session, or can it span sessions?
3. Can a TaskFlow invoke multiple skills across multiple turns, or is it bound to one skill invocation?
4. Are child tasks in scope for v0.8, or deferred to v0.9?

### 7.2 State Machine Design

5. What are the exact flow states and legal transitions?
6. What are the exact step states and legal transitions?
7. Does a step map 1:1 to a `ToolCallPlan`, or can a step contain multiple tool calls?
8. How are step dependencies represented? (DAG? Linear sequence?)

### 7.3 Persistence and Recovery

9. What is the SQLite schema for flows, steps, and flow events?
10. How is flow state rehydrated after a process restart?
11. Does the provider message history get truncated/summarized on long flows?
12. How are checkpoints represented? As full state snapshots or incremental event logs?

### 7.4 Operator Controls

13. Which operator controls are must-have vs. should-have vs. defer? (See Section 9.)

## 9. Suggested Scope Classification

### 9.1 Must-Have for v0.8

| Feature | Rationale |
|---------|-----------|
| Flow state machine | Core of TaskFlow — without this, there is no v0.8 |
| Step state machine | Core of TaskFlow |
| Durable flow persistence | Without persistence, flows are not "durable" |
| Flow event log | Required for auditability and traceability |
| Restart recovery | Without this, durability is meaningless |
| Wait/resume/cancel | Core operator-visible semantics |
| Flow-to-run/artifact linkage | Required for inspectability |
| `/status` | Basic operator visibility |
| `/cancel` | Basic operator control |
| `/trace` (flow-aware) | Required for debugging |

### 9.2 Should-Have for v0.8

| Feature | Rationale |
|---------|-----------|
| `/pause` | Important for operator control, but can be simulated with cancel + resume in v0.8.1 |
| `/resume` (structured) | Current text-only resume is insufficient for real flows |
| `/interrupt` | Important for safety, but cancel + restart is a workaround |
| `/approve` and `/reject` (flow-step level) | Important for operational workflows |
| Retry policy | Important for robustness |
| Failure policy | Important for robustness |
| Child tasks | Adds significant complexity; defer if scope is tight |

### 9.3 Defer to v0.9+

| Feature | Rationale |
|---------|-----------|
| `/steer` | Complex semantics; requires careful design to avoid hidden mutation |
| `/compact` | Complex context management; not required for MVP durability |
| `/skip` | Operational convenience; can be deferred |
| `/checkpoint` | Recovery enhancement; basic restart recovery is sufficient for v0.8 |
| `/retry` (operator-initiated) | Can be automated via retry policy instead |
| Child tasks | Spawns architectural questions about parent-child state; better deferred |
| Channel-triggered flows | Explicitly v0.9 per roadmap |
| Background/scheduled flows | Explicitly v0.9 per roadmap |
| Flow visual dashboard | v1.x concern |

### 9.4 Unsafe or Unclear

| Feature | Concern |
|---------|---------|
| `/steer` without audit guarantees | Risk of hidden prompt mutation if not carefully designed |
| `/compact` without preservation rules | Risk of losing active state, pending approvals, or operator events |
| `/interrupt` without process cleanup | Risk of orphaned processes and resource leaks |
| Operator `/approve` overriding security `deny` | Governance risk — needs explicit policy decision |
| Flow-step approvals bypassing tool-level security | Must not create a parallel weaker security path |
| Auto-retry without idempotency checks | Risk of duplicate side effects |

## 10. Audit Summary

**Readiness verdict:** The codebase has strong foundations for v0.8 but significant gaps remain.

**Strengths:**
- Clean agent-loop decomposition from v0.4
- SQLite persistence layer from v0.5
- Security policy and approval controller from v0.4-v0.7
- Trajectory and run recording from v0.5
- Tool planning and execution with dependency awareness

**Critical gaps:**
- No flow or step state machine
- No flow persistence schema
- No structured resume/rehydration
- No process ownership for cascade cleanup
- No operator event log
- No flow-oriented CLI commands
- Artifact store is in-memory only
- No pause, steer, compact, checkpoint, or skip mechanisms

**Recommended next step:** Answer the design questions in Section 7, then write a bounded execution plan that delivers the must-haves first, with explicit decisions on what is excluded.
