## 3. Updated Schema Guidance

### 3.1 Migration Version (Correction #2)

**Current state of `SQLiteSessionDB`:**

The existing `#migrate()` method in `src/session/sqlite-session-db.ts` uses `CREATE TABLE IF NOT EXISTS` with **no schema versioning**. There is no `schema_version` table, no `PRAGMA user_version` check, and no migration framework.

**Corrected approach:**

v0.8 must introduce schema versioning for the first time.

```sql
-- Introduce schema_version table as part of v0.8 migration
CREATE TABLE IF NOT EXISTS schema_version (
  version INTEGER PRIMARY KEY,
  applied_at TEXT NOT NULL
);

-- Detect if this is a fresh database or an existing one
-- If schema_version is empty but legacy tables exist, we are migrating an existing DB
-- In that case, record version 0 as the baseline
INSERT INTO schema_version (version, applied_at)
SELECT 0, datetime('now')
WHERE NOT EXISTS (SELECT 1 FROM schema_version)
  AND EXISTS (SELECT 1 FROM sqlite_master WHERE name = 'sessions');

-- Apply v0.8 migration as version 1
INSERT INTO schema_version (version, applied_at) VALUES (1, datetime('now'));
```

**Migration numbering:**
- Version `0`: Baseline (existing tables before v0.8)
- Version `1`: v0.8 TaskFlow schema additions

**Rules:**
- Do not insert synthetic versions that do not correspond to actual migrations.
- If `schema_version` does not exist, create it and set baseline to `0` if legacy tables exist.
- If `schema_version` exists and shows `0`, apply version `1`.
- If `schema_version` shows `≥1`, skip migration.
- Always backup `.estacoda/session.db` to `.estacoda/session.db.backup.pre-v0.8` before applying.

--   artifacts_json      -> use flow_artifacts table

## 4. Updated Command Semantics

### 4.1 /retry (Correction #1)

**Previous acceptance criteria wording:**
> "/retry rejected for non-idempotent steps without override."

**Corrected wording:**

| Condition | `/retry` Behavior |
|---|---|
| `idempotent === true` | **Allowed**. Create new retry step. |
| `safeToRetry === true` | **Allowed**. Create new retry step. |
| `idempotent === false` | **Rejected**. No operator override in v0.8. |
| `safeToRetry === false` | **Rejected**. No operator override in v0.8. |
| Idempotency unknown | **Rejected**. No operator override in v0.8. |

**Rejection message:**
```
Cannot retry step "deploy-app": it is not marked idempotent or safeToRetry.
Operator override for non-idempotent retry is not available in v0.8.
```

**Retry event recording:**
- `operator-retried` event in `operator_events`.
- `step-retried` event in `flow_events` (logical event).
- New step created in `flow_steps` with `retryOfStepId` and `attemptNumber`.
- Original failed step stays terminal.

### 4.2 Operator-Event Atomicity (Correction #4)

**Rule:** Every operator command that changes flow or step state must execute in a single SQLite transaction:

```
BEGIN TRANSACTION
  1. Acquire flow lock
  2. Validate current state
  3. Append operator_event to operator_events
  4. Append flow_event to flow_events (if state transition occurs)
  5. Update flow snapshot (UPDATE flows)
  6. Update step snapshot if applicable (UPDATE flow_steps)
  7. Update related records (approval_gates, etc.)
  8. Update flow lock heartbeat
COMMIT
```

**Read-only exceptions (no transaction required):**
- `/status`
- `/trace`
- `/compact --preview` (if implemented)
- `estacoda flow list/show/steps/events`

**Enforcement:**
- `TaskFlowStore.atomicTransition()` wraps the above pattern.
- The `OperatorCommandDispatcher` calls `atomicTransition()` for all mutating commands.
- No `operator_event` without corresponding state transition when the command changes state.
- No state transition caused by operator command without `operator_event`.

### 4.3 /compact --preview (Correction #5)

**Status:** Should-have, not must-have.

**If implemented:**
- Shows what would be compacted without modifying state.
- Read-only operation; no transaction required.
- Lists: turns to summarize, token events to truncate, tool results to trim.

**If not implemented in v0.8.0:**
- Acceptable. Manual `/compact` and automatic compaction are the release requirements.
- `--preview` can be added in v0.8.x.

### 4.4 /compact Safety (Reaffirmed)

**Rejected when:**
- Active provider call in flight
- Active tool execution running
- Active process handle registered
- Any `running` step with ongoing execution

**Allowed when:**
- Flow is `paused`
- Flow is `waiting`
- Flow is `interrupted`
- Flow is `running` but between turns (no active execution)
- No active flow exists

### 4.5 Automatic Compaction (Correction #10)

**Config shape:**
```typescript
type CompactionConfig = {
  enabled: boolean;        // default: false in v0.8
  mode: "manual" | "auto"; // default: "manual"
  thresholdTurns?: number; // default: 10 (when auto enabled)
  thresholdTokens?: number; // default: 8000 (when auto enabled)
  preserveOperatorEvents: boolean; // default: true
  preserveApprovalGates: boolean;  // default: true
};
```

**Behavior:**
- `enabled: false` by default. Documented as experimental.
- When enabled, triggers only at safe boundaries (between turns).
- Never runs during active provider/tool/process execution.
- Always records a `compacted` event in `flow_events`.
- Always produces a `CompactSummary` artifact or equivalent durable record.
- Visible in `/trace` and `estacoda flow show`.

## 7. Updated Process Cleanup Guidance

### 7.1 Portability (Correction #9)

**Previous text:**
> "SIGTERM → 5s wait → SIGKILL"

**Corrected text:**

Process cleanup must go through the existing `ProcessManager` abstraction:

```typescript
// In TaskFlowEngine or FlowProcessRegistry
async cleanupProcesses(flowId: FlowId): Promise<ProcessCleanupResult> {
  const processes = await this.store.listActiveProcesses(flowId);
  const results: ProcessCleanupResult = { stopped: [], failed: [] };

  for (const proc of processes) {
    try {
      // Delegate to ProcessManager
      await this.processManager.stop(proc.processManagerId, "SIGTERM");
      
      // Wait for graceful shutdown
      const exited = await this.waitForExit(proc.processManagerId, timeoutMs: 5000);
      
      if (!exited) {
        await this.processManager.stop(proc.processManagerId, "SIGKILL");
        results.stopped.push({ id: proc.id, method: "force" });
      } else {
        results.stopped.push({ id: proc.id, method: "graceful" });
      }
    } catch (error) {
      results.failed.push({ id: proc.id, error: String(error) });
    }
  }

  return results;
}
```

**Rules:**
- TaskFlow core contracts do not hardcode Unix signal names.
- `ProcessManager.stop()` accepts a signal name string; the manager maps to platform-appropriate behavior.
- If `ProcessManager` currently only supports Unix signals, that is an existing limitation, not a TaskFlow limitation.
- Failed cleanup is recorded in `operator_events` with `effect: "cleanup-partial"` and listed in `results.failed`.
- `/status` and `/trace` display cleanup results.

**Stuck process handling:**
```
Flow flow_abc123 interrupted.
Process cleanup: 1 stopped gracefully, 1 force-stopped, 1 failed.
Failed: process xyz (terminal.run) — cleanup error: process not found
```
