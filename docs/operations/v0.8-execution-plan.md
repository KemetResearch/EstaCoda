## 1. Objective and Final v0.8 Scope

### 1.1 Objective

EstaCoda v0.8 must demonstrate:

> **Durable, inspectable, operator-controllable long-running execution where every step has state, every intervention is recorded, risky actions block for approval, failures are recoverable or explainable, and the flow survives process restart.**

By the end of v0.8, a user can:

1. Start a multi-step flow from the CLI.
2. See flow status, current step, pending decisions, and elapsed time via `/status`.
3. Pause a flow with `/pause` and resume later with `/resume` — even after restarting EstaCoda.
4. Interrupt active execution with `/interrupt` safely, without corrupting state or leaving orphaned processes.
5. Cancel a flow with `/cancel` and see a terminal cancelled state persisted.
6. Approve or reject blocked steps with `/approve` or `/reject`.
7. Inspect the full event trail with `/trace` or `estacoda flow show <id>`.
8. Trust that the flow picks up where it left off after restart.
9. Trust that no operator command bypassed security policy, approval policy, or tool execution paths.
10. Trust that every operator intervention is an auditable event, not hidden prompt mutation.
11. Use automatic compaction to manage long-running session context safely.
12. Verify all of the above through deterministic eval fixtures.

| Feature | Deferred To | Rationale |
|---|---|---|
| Channel-triggered flows | v0.9 | Channels multiply risk |
| Cron/background flows | v0.9 | Background autonomy needs policy gates |
| Visual workflow builder | v1.x | Not required for durable execution |
| Full DAG engine | v0.9+ | Linear sequences sufficient for MVP |
| Marketplace/capability installation | v0.10 | Trusted extension is v0.10 scope |
| Complex child-task orchestration | v0.9 | Parent-child state adds complexity |
| Full checkpoint restore | v0.9 if risky | Create-only in v0.8 if restore is unsafe |
| Arbitrary autonomous background operation | post-MVP | Requires sandbox/monitoring |
| Email/calendar/Telegram execution surface | v0.9 | Channel-specific execution |
| Automatic flow creation from skills | v0.8.x | Skills remain advisory |
| FlowTemplate / reusable templates | v0.9/v0.10 | Ad-hoc flow plans only in v0.8 |
| Operator override for non-idempotent retry | post-v0.8 | Conservative safety in v0.8 |
| Mid-turn pause via safe-boundary callbacks | v0.8.x/v0.9 | Option C seam preserved, not wired |
| Aggressive auto-compaction | post-v0.8 | Conservative default in v0.8 |

### 3.2 Key Architectural Decisions (Locked)

| Decision | Choice | Rationale |
|---|---|---|
| TaskFlow position | Above AgentLoop via adapter | Preserves v0.4 decomposition, keeps AgentLoop testable |
| AgentLoop changes | Minimal — optional `flowContext` via adapter | AgentLoop stays TaskFlow-agnostic |
| Tool execution path | Only through `ToolExecutor` | Security policy not duplicated or bypassed |
| Event tables | Separate `flow_events` and `operator_events` | Lifecycle vs intervention separation |
| Approval authority | `WorkspaceApprovalController` remains authoritative | TaskFlow gates are durable wrappers |
| Hard deny handling | No approval gate created; step fails per policy | `/approve` cannot override hard deny |
| Retry model | New step with `retryOfStepId` | Failed steps stay terminal; clear history |
| Retry override | Not allowed in v0.8 | Conservative safety |
| Concurrency | One active runner per flow, multiple persisted flows | Simple, prevents duplicate execution |
| FlowTemplate | Deferred | v0.8 uses ad-hoc FlowPlans |
| Compaction | Manual + automatic, configurable | Hermes-inspired session hygiene |
| Compaction safety | Rejected during active execution | Never loses durable state |
| Option C | Preserve seams, implement A+ | Callback hooks defined but no-ops in v0.8 |
| Atomic transitions | SQLite transaction per transition | No event-only or snapshot-only transitions |
| Locking | `flow_locks` table with lease + heartbeat | Prevents duplicate execution after restart |

## 4. Workstreams / Tracks

v0.8 is organized into **5 workstreams** that can be developed in sequence with some overlap:

### Track 1: Foundation — Persistence + State Model + Atomic Transitions
**Goal:** The database schema, state model, and transaction machinery exist and are tested.

### Track 2: Engine — TaskFlowEngine + Adapter + Locking
**Goal:** TaskFlow can create, run, pause, resume, interrupt, and cancel flows with proper locking.

### Track 3: Operator Control Plane — Commands + Events + Process Ownership
**Goal:** All operator commands work, record events, and clean up processes.

### Track 4: Integration — CLI + Recorder Linkage + Compaction
**Goal:** CLI commands, run/artifact linkage, and compaction are wired and functional.

### Track 5: Verification — Tests + Evals + Docs
**Goal:** Deterministic tests pass, eval fixtures exist, documentation is complete.

## 8. State Model and Command Semantics Summary

### 8.1 Flow States

```
pending → running → completed
        → paused ↔ running
        → waiting ↔ running
        → interrupted ↔ running
        → cancelled (terminal)
        → failed (terminal)
```

### 8.2 Step States

```
pending → running → completed
        → paused ↔ running
        → waiting_for_approval ↔ running
        → waiting_for_input ↔ running
        → interrupted ↔ running
        → failed (terminal)
        → skipped (terminal)
        → cancelled (terminal)
```

### 8.3 Command Semantics (Condensed)

| Command | Allowed When | Effect | Event |
|---|---|---|---|
| `/status` | Always | Read-only display | None |
| `/pause` | Flow `running` | Set `pauseRequestedAt`; pause at next safe boundary | `operator-pause-requested` → `operator-paused` |
| `/resume` | Flow `paused`/`waiting`/`interrupted` | Clear wait/pause/interrupt; dispatch next turn | `operator-resumed` |
| `/interrupt` | Flow `running` with active execution | Abort signal, process cleanup, `interrupted` | `operator-interrupted` |
| `/cancel` | Any non-terminal | Terminal `cancelled`, cascade cleanup | `operator-cancelled` |
| `/steer` | Paused, waiting, or between turns | Queue guidance for next turn | `operator-steered` |
| `/compact` | No active execution | Summarize stale context | `operator-compacted` |
| `/approve` | Step `waiting_for_approval` | Grant approval, resume step | `operator-approved`, `approval-granted` |
| `/reject` | Step `waiting_for_approval` | Deny approval, apply failure policy | `operator-rejected`, `approval-denied` |
| `/retry` | Step `failed`, `idempotent=true` or `safeToRetry=true` | Create new retry step | `operator-retried`, `step-retried` |
| `/skip` | Step `paused`/`interrupted`, `skippable=true` | Skip step, advance | `operator-skipped`, `step-skipped` |
| `/checkpoint` | Always | Create named snapshot | `operator-checkpointed`, `checkpoint-created` |
| `/trace` | Always | Display merged event timeline | None |

### 8.4 Retry Policy (Conservative)

- `/retry` allowed only if `idempotent === true` OR `safeToRetry === true`.
- Manual override for non-idempotent steps is **not** allowed in v0.8.
- Retry creates a new `FlowStep` with `retryOfStepId` and `attemptNumber`.
- Failed original step stays terminal.

### 8.5 Approval Policy

- Hard deny (`decision: "deny"` with `deterministicRule`) → no approval gate created.
- Soft deny/ask (`decision: "ask"`) → approval gate created.
- `/approve` calls `WorkspaceApprovalController.grantApproval()`.
- `/approve` on hard deny is rejected with clear error.

## 14. Explicit v0.9 Handoff Notes

### 14.1 Schema Stability Contract

The following must remain stable for v0.9:

- `flows.status` enum values (append-only)
- `flow_steps.status` enum values (append-only)
- `flow_events.kind` values (append-only)
- `operator_events.kind` values (append-only)
- `flow_locks` table schema
- `approval_gates` table schema
- `flow_processes` table schema

### 14.2 What v0.9 Will Extend

| v0.9 Feature | v0.8 Foundation |
|---|---|
| Channel-triggered flows | `TaskFlowEngine.createFlow()` can be called from gateway |
| Cron/background flows | `FlowLockService` prevents duplicate execution |
| Child tasks | `FlowStep` can be extended with `parentStepId` |
| Full DAG | `FlowStep` dependency field can be added |
| Checkpoint restore | `Checkpoint.snapshot` schema already defined |
| Mid-turn pause | `AgentLoopHooks` seam already defined |
| Visual dashboard | `flow_events` + `operator_events` provide data |

### 14.3 What v0.9 Must Not Break

- Existing `AgentLoop` contract (backward-compatible `flowContext` via adapter)
- Existing `ToolExecutor` security path
- Existing `RunRecorder` and `TrajectoryRecorder` interfaces
- Existing `WorkspaceApprovalController` authority

### 14.4 Known v0.8 Limitations to Address in v0.9

1. Pause only at turn boundaries (mid-turn pause needs Option C callbacks).
2. `/steer` queues for next turn (mid-turn steer needs Option C callbacks).
3. Checkpoint create-only (restore needs validation and testing).
4. No child task orchestration.
5. No channel/cron triggers.
6. Linear step sequences only (no DAG).
7. Process cleanup is best-effort (orphan detection but not automatic kill).
