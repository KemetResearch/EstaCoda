# RFC: v0.8 Durable TaskFlow + Operator Control Plane

**Status:** Draft  
**Target Version:** v0.8.0  
**Date:** 2026-05-03  
**Author:** Hermes Agent (planning assistant)  
**Inputs:** v0.8 Readiness Audit, Roadmap Narrative (v0.4–v0.10), Hermes Agent operator-control patterns  
**Constraint:** No implementation. No execution plan. No code changes. No doc index changes.

---

## 1. Product / Runtime Objective

### 1.1 What v0.8 Must Prove

By v0.8, EstaCoda must demonstrate:

> **Durable, inspectable, operator-controllable long-running execution where every step has state, every intervention is recorded, risky actions block for approval, failures are recoverable or explainable, and the flow survives process restart.**

### 1.2 The Target Experience

A user should be able to:

1. Start a multi-step flow (e.g., "refactor the auth module, run tests, update docs").
2. See the flow’s current step, status, and pending decisions at any time via `/status`.
3. Pause the flow with `/pause` and resume it later with `/resume` — even after restarting EstaCoda.
4. Interrupt a running step with `/interrupt` safely, without corrupting state or leaving orphaned processes.
5. Cancel the entire flow with `/cancel` and see a terminal cancelled state persisted.
6. Approve or reject a blocked high-risk step with `/approve` or `/reject`.
7. Inspect the full event trail with `/trace` or `estacoda flow show <id>`.
8. Trust that the flow will pick up where it left off after a process restart.
9. Trust that no operator command bypassed security policy, approval policy, or tool execution paths.
10. Trust that every operator intervention is an auditable event, not hidden prompt mutation.

### 1.3 What v0.8 Is Not

- Not a visual workflow builder.
- Not a background execution engine (no cron-driven or channel-triggered flows in v0.8).
- Not a marketplace for capabilities.
- Not a system that silently mutates skills, prompts, or runtime code.
- Not a full DAG scheduler (linear/stepped sequences are the v0.8 target; DAG is a should-have if trivial).

### 1.4 Translation from Hermes Operator Patterns

Hermes provides excellent operator ergonomics: slash commands, session browsing, checkpoint rollback, pause/resume, and status views. EstaCoda v0.8 should match that *ergonomic quality* but translate every pattern into runtime semantics:

| Hermes Pattern | EstaCoda v0.8 Translation |
|---|---|
| `hermes chat --checkpoints` | Flow checkpoint snapshots; no automatic restore if unsafe |
| `hermes sessions list/browse` | `estacoda flow list` and flow-aware `/status` |
| `hermes status [--all]` | `/status` shows runtime + active flow + step + approval state |
| `/pause`, `/resume` in Hermes TUI | `/pause` stops at next safe boundary; `/resume` rehydrates from DB |
| Hermes checkpoint rollback | `/checkpoint` creates named snapshot; restore deferred to v0.9 if risky |
| Hermes tool approval flow | TaskFlow step approval gates, linked to existing security policy |
| Hermes memory/session recall | Flow event log + restart recovery, not session search |

**Rule:** Copy the *ergonomics*, not the *implementation*. Every Hermes-inspired behavior must declare what state changes, what persists, what links to evidence, and what policy enforces it.

## 2. Architectural Placement

### 2.1 The Core Question

Where does TaskFlow live relative to `AgentLoop`?

### 2.2 Option A: TaskFlow Above AgentLoop (Recommended)

```
TaskFlowEngine
  └── orchestrates turns
        └── AgentLoop.handle(turnContext)
              └── ProviderTurnLoop
              └── ToolPlanRunner
              └── ToolExecutor  (security policy enforced here)
```

**How it works:**
- `TaskFlowEngine` owns the flow state machine, step sequencing, and operator command dispatch.
- Each step that requires an LLM decision becomes a `turn` dispatched to `AgentLoop.handle()`.
- `AgentLoop` remains responsible for provider interaction, tool planning, and tool execution within a single turn.
- `ToolExecutor` remains the *only* path for tool execution. Security policy is enforced below TaskFlow.
- `RunRecorder` and `TrajectoryRecorder` are shared evidence layers used by both TaskFlow and AgentLoop.

**Pros:**
- Preserves the v0.4 decomposition. `AgentLoop` stays testable and unchanged in its core contract.
- Security policy is not duplicated or bypassed.
- Process ownership can be injected at the TaskFlow layer while still leveraging `ToolExecutor`.
- Restart recovery is simpler: rehydrate TaskFlow state, then resume calling `AgentLoop.handle()`.

**Cons:**
- Step granularity is limited to "one turn per step." If a step needs multiple provider calls without returning to TaskFlow, AgentLoop must handle them internally. This is acceptable for v0.8.
- `/pause` during a turn cannot pause mid-tool-call within AgentLoop. The pause boundary is "between turns."

### 2.3 Option B: TaskFlow Inside AgentLoop (Rejected)

TaskFlow would decompose `AgentLoop.handle()` into step-level hooks and orchestrate provider calls directly.

**Why rejected:**
- Too invasive for v0.8. Would require refactoring the 700-line `AgentLoop` into a state machine.
- Would entangle flow logic with provider token streaming, tool-call parsing, and fallback logic.
- Would make testing harder because flow state and provider state would be coupled.
- The v0.4 decomposition was explicitly designed to keep these layers separable. Violating that now is architectural regression.

### 2.4 Option C: Hybrid — TaskFlow Above, with Step Callbacks (Deferred)

TaskFlow sits above AgentLoop but AgentLoop exposes "step completed" callbacks so TaskFlow can pause mid-turn.

**Why deferred:**
- Requires adding callbacks/hooks to AgentLoop, which is more invasive than Option A but less than Option B.
- Can be added in v0.8.x if pause-between-turns proves too coarse.
- For v0.8, "pause between turns" is sufficient.

### 2.5 Recommended Architecture

**Adopt Option A: TaskFlow above AgentLoop.**

Key invariants:

1. `AgentLoop.handle()` contract is unchanged. It takes a `turnContext` and returns a `turnResult`.
2. `ToolExecutor.executeTool()` is the *only* path for tool execution. TaskFlow never calls tools directly.
3. Security policy is enforced in `ToolExecutor`. TaskFlow does not reimplement or bypass it.
4. `RunRecorder` and `TrajectoryRecorder` are shared. TaskFlow emits flow events; AgentLoop emits turn events.
5. `ProcessManager` processes are linked to `flowId` + `stepId` via an active-execution registry.
6. Cancellation/interruption signals flow downward from TaskFlow → AgentLoop → ToolExecutor → ProcessManager.

### 2.6 Data Flow Diagram (Conceptual)

```

        ▼
┌─────────────────┐
```

### 3.2 Idempotency Model

Every `FlowStep` carries an `idempotent` boolean. The default is `false`.

- If `idempotent === false`, retry policy **must not** permit retry unless the operator explicitly overrides via `/retry`.
- If `idempotent === true`, automatic retry is permitted according to `RetryPolicy`.
- A step can infer idempotency from the tool being called (e.g., `file.read` is idempotent; `file.write` is not unless append-mode).
- The `ToolExecutor` should expose an `isIdempotent(toolName, input)` helper for TaskFlow to consult.

### 3.3 Terminal vs Non-Terminal States

**Terminal states (no further transitions allowed):**
- Flow: `completed`, `cancelled`, `failed`
- Step: `completed`, `failed`, `skipped`, `cancelled`

**Non-terminal states (can transition to other states):**
- Flow: `pending`, `running`, `paused`, `waiting`, `interrupted`
- Step: `pending`, `running`, `paused`, `waiting`, `waiting_for_approval`, `waiting_for_input`, `interrupted`

### 3.4 Current State Snapshot vs Append-Only Event History

TaskFlow uses a **dual model**:

1. **Snapshot tables** (`flows`, `flow_steps`) hold the current state for fast lookup and restart recovery.
2. **Append-only event log** (`flow_events`) holds every state change, operator action, and transition for auditability and trace reconstruction.

**Rule:** The snapshot is the *source of truth* for execution. The event log is the *source of truth* for audit. On restart, the snapshot is loaded. The event log is consulted only if the snapshot is suspect (corruption recovery) or for trace inspection.

## 4. Legal State Transitions

### 4.1 Flow State Transitions

| From → To | start | pause | resume | interrupt | cancel | wait | approval_req | approval | rejection | retry | complete | fail |
|---|---|---|---|---|---|---|---|---|---|---|---|---|
| **pending** | running | — | — | — | cancelled | — | — | — | — | — | — | failed |
| **running** | — | paused | — | interrupted | cancelled | waiting | — | — | — | — | completed | failed |
| **paused** | — | — | running | interrupted | cancelled | — | — | — | — | — | — | — |
| **waiting** | — | — | running | interrupted | cancelled | — | — | running | failed | — | — | — |
| **interrupted** | — | — | running | — | cancelled | — | — | — | — | — | — | failed |
| **completed** | — | — | — | — | — | — | — | — | — | — | — | — |
| **cancelled** | — | — | — | — | — | — | — | — | — | — | — | — |
| **failed** | — | — | — | — | — | — | — | — | — | — | — | — |

**Illegal transitions (explicitly prohibited):**
- `completed → *` (terminal)
- `cancelled → *` (terminal)
- `failed → *` (terminal)
- `pending → paused` (cannot pause before starting)
- `waiting → paused` (wait must resolve first)
- `interrupted → paused` (interrupt is stronger than pause)
- `running → completed` without all steps completing
- `waiting → interrupted` without operator command

### 4.2 Step State Transitions

| From → To | start | pause | resume | interrupt | cancel | wait_approval | wait_input | approval | rejection | retry | skip | complete | fail |
|---|---|---|---|---|---|---|---|---|---|---|---|---|---|
| **pending** | running | — | — | — | cancelled | — | — | — | — | — | skipped | — | failed |
| **running** | — | paused | — | interrupted | cancelled | waiting_for_approval | waiting_for_input | — | — | — | — | completed | failed |
| **paused** | — | — | running | interrupted | cancelled | — | — | — | — | — | — | — | — |
| **waiting_for_approval** | — | — | — | interrupted | cancelled | — | — | running | failed | — | — | — | — |
| **waiting_for_input** | — | — | — | interrupted | cancelled | — | — | running | — | — | — | — | — |
| **interrupted** | — | — | running | — | cancelled | — | — | — | — | running | skipped | — | failed |
| **completed** | — | — | — | — | — | — | — | — | — | — | — | — | — |
| **failed** | — | — | — | — | — | — | — | — | — | — | — | — | — |
| **skipped** | — | — | — | — | — | — | — | — | — | — | — | — | — |
| **cancelled** | — | — | — | — | — | — | — | — | — | — | — | — | — |

**Illegal transitions:**
- Any terminal state (`completed`, `failed`, `skipped`, `cancelled`) cannot transition.
- `waiting_for_approval → waiting_for_input` (wait reason cannot change without resolution)
- `failed → retry` (retry creates a *new* step attempt; the old step stays `failed`)

### 4.3 Transition Enforcement

- All transitions are mediated by `TaskFlowEngine.transitionFlow()` and `TaskFlowEngine.transitionStep()`.
- Illegal transitions throw `IllegalTransitionError` and are recorded as `flow-event` with `kind: "illegal-transition-attempted"`.
- The snapshot is only updated after the event is successfully appended.

### 5.2 /status

**When allowed:** Always.

**What it shows:**

```
EstaCoda v0.8.0 — Session: ses_abc123
Model: kimi/k2.6  Security: adaptive  Workspace: trusted

Active Flow: flow_xyz789
  Status: running
  Step 3/7: run-tests (running)
  Started: 2026-05-03T14:22:10Z
  Elapsed: 4m 12s
  
  Pending: none
  
  Recent Events:
    14:22:10  flow-started
    14:22:15  step-1-completed  analyze-code
    14:24:30  step-2-completed  plan-refactor
    14:26:22  step-3-started    run-tests

No blocked approvals.
Last operator action: none.
```

**State changes:** None.

**Events recorded:** None (read-only).

**What it must not bypass:** Nothing.

**After restart:** Same output, but `Elapsed` shows total time across restarts.

### 5.3 /pause

**When allowed:** Flow is `running`.

**What it does:**
1. Sets `Flow.status = "paused"`.
2. Sets `Flow.pauseReason = <operator text>`.
3. If a step is `running`, sets `Step.status = "paused"` and `Step.pauseReason`.
4. Does **not** abort the active turn immediately. The pause takes effect at the next safe boundary (end of current turn).
5. Records `operator-paused` event.

**State changes:** `running → paused` (flow); `running → paused` (step, if active).

**Events recorded:** `operator-paused`.

**What it must not bypass:** Cannot pause if flow is `waiting` (wait must resolve first).

**After restart:** Flow loads as `paused`. Operator must `/resume`.

**CLI output:**
```
Flow flow_xyz789 paused.
Current step "run-tests" will complete its active turn, then pause.
Use /resume to continue.
```

### 5.4 /resume

**When allowed:** Flow is `paused`, `waiting`, or `interrupted`.

**What it does:**
1. Clears `pauseReason`, `interruptReason`, `waitReason` (if applicable).
2. Sets `Flow.status = "running"`.
3. If step was `paused` or `interrupted`, sets `Step.status = "running"`.
4. If step was `waiting`, resolves the wait condition (for `user_input` waits, the resume text becomes the input).
5. Records `operator-resumed` event.
6. Reconstructs provider context and dispatches the next turn.

**State changes:** `paused/waiting/interrupted → running`.

**Events recorded:** `operator-resumed`.

**After restart:** If a flow was `paused` or `interrupted` at shutdown, restart recovery loads it in that state. The operator must `/resume`.

**CLI output:**
```
Resuming flow flow_xyz789 at step "run-tests" (step 3/7).
```

### 5.5 /interrupt

**When allowed:** Flow is `running` or `waiting`.

**What it does:**
1. Immediately aborts the active turn via `AbortController`.
2. Sets `Flow.status = "interrupted"`.
3. Sets `Flow.interruptReason`.
4. Sets active step `status = "interrupted"`.
5. Triggers active-process cleanup (see Section 10).
6. Records `operator-interrupted` event.
7. Persists interruption snapshot.

**State changes:** `running/waiting → interrupted`.

**Events recorded:** `operator-interrupted`.

**What it must not bypass:** Must not bypass security policy. Must not corrupt artifact state.

**After restart:** Flow loads as `interrupted`. Operator can `/resume` or `/cancel`.

**CLI output:**
```
Flow flow_xyz789 interrupted.
Active step "run-tests" stopped. 1 process cleaned up.
Use /resume to retry from this step, or /cancel to terminate the flow.
```

### 5.6 /cancel

**When allowed:** Any non-terminal flow state.

**What it does:**
1. Sets `Flow.status = "cancelled"`.
2. Sets `Flow.cancelReason`.
3. Cancels all pending steps.
4. Aborts active turn if any.
5. Triggers process cleanup.
6. Records `operator-cancelled` event.
7. Persists terminal cancelled state.

**State changes:** Any non-terminal → `cancelled`.

**Events recorded:** `operator-cancelled`.

**After restart:** Flow loads as `cancelled`. No further action possible.

**CLI output:**
```
Flow flow_xyz789 cancelled.
Reason: operator command
3 steps completed, 4 steps cancelled.
```

### 5.7 /steer (v0.8-Safe Version)

**When allowed:** Flow is `paused`, `waiting`, or between steps within a `running` flow.

**v0.8 restriction:** `/steer` is **not** allowed while a provider turn is actively streaming tokens. The steering text is queued and injected as an operator message on the *next* provider turn.

**What it does:**
1. Records `operator-steered` event with the full steering text.
2. Appends the steering text to the flow context as an `operator` role message.
3. The next turn sent to `AgentLoop` includes this operator message.
4. Does **not** modify system prompt or hidden instructions.

**State changes:** None (flow state unchanged).

**Events recorded:** `operator-steered`.

**What it must not bypass:** Must not bypass security policy. Steering text is visible in traces.

**After restart:** Steering events are in the event log and reconstructed into context on resume.

**CLI output:**
```
Steering recorded for flow flow_xyz789.
"Focus on error handling in the tests."
This will be included in the next provider turn.
```

### 5.8 /compact (v0.8-Safe Version)

**When allowed:** Always.

**v0.8 restriction:** Only compact **completed or stale** conversation context. Never compact active flow state, pending approvals, unresolved events, or operator interventions.

**What it does:**
1. Identifies compactable state:
   - Old completed turns (beyond a context window) → summarized into a `CompactSummary` artifact.
   - Old trajectory token events → discarded (retain event metadata, drop token text).
   - Old tool result text → truncated to summary.
2. Preserves non-compactable state (see Section 6).
3. Records `compacted` event with before/after metrics.
4. Updates `Flow.compactedAt`.

**State changes:** None (flow status unchanged).

**Events recorded:** `compacted`.

**After restart:** Compacted state is preserved; summarized context is reloaded.

**CLI output:**
```
Compacted session context.
Preserved: active flow, 2 pending steps, 1 approval, 3 operator events
Summarized: 4 completed turns (→ 1 summary artifact)
Truncated: 2 old tool results
```

### 5.9 /approve

**When allowed:** Flow or step is `waiting_for_approval`.

**What it does:**
1. Resolves the `ApprovalGate` with `status = "approved"`.
2. Sets `Flow.status = "running"` (if it was `waiting`).
3. Sets `Step.status = "running"` (if it was `waiting_for_approval`).
4. Records `approval-granted` event and `operator-approved` event.
5. Does **not** override a security `deny`. If the tool was hard-denied by policy, approval is rejected with an error.

**State changes:** `waiting_for_approval → running`.

**Events recorded:** `approval-granted`, `operator-approved`.

**What it must not bypass:** Cannot override a hard security deny (see Section 9).

**After restart:** Approval state is persisted. Flow resumes as approved.

**CLI output:**
```
Approved step "deploy-app" (approval req_123).
Flow resumes.
```

### 5.10 /reject

**When allowed:** Flow or step is `waiting_for_approval`.

**What it does:**
1. Resolves the `ApprovalGate` with `status = "rejected"`.
2. Applies the configured `FailurePolicy`:
   - `stop` → flow fails.
   - `skip` → step skipped, flow continues.
   - `escalate` → flow waits for further operator input.
3. Records `approval-denied` event and `operator-rejected` event.

**State changes:** `waiting_for_approval → failed/skipped/waiting` depending on policy.

**Events recorded:** `approval-denied`, `operator-rejected`.

**CLI output:**
```
Rejected step "deploy-app" (approval req_123).
Failure policy: stop
Flow flow_xyz789 failed.
```

### 5.11 /retry (v0.8-Safe Version)

**When allowed:** Step is `failed` or `interrupted`. Flow is `paused` or `interrupted`.

**v0.8 restriction:** Only allowed for steps marked `idempotent === true` OR explicitly marked `safeToRetry === true` in the step metadata. Non-idempotent steps require operator confirmation.

**What it does:**
1. Validates retry eligibility (respects `maxRetries`).
2. If `retryCount >= maxRetries`, rejects with error.
3. If step is not idempotent, prompts for confirmation.
4. Resets step state to `pending` (or `running` if flow resumes immediately).
5. Increments `retryCount`.
6. Records `step-retried` event and `operator-retried` event.
7. Dispatches the step turn again.

**State changes:** `failed/interrupted → pending → running`.

**Events recorded:** `step-retried`, `operator-retried`.

**CLI output:**
```
Retrying step "run-tests" (attempt 2/3).
Step is idempotent. Proceeding.
```

### 5.12 /skip (v0.8-Safe Version)

**When allowed:** Flow is `paused` or `interrupted`, and step is explicitly `skippable === true`.

**v0.8 restriction:** Only allowed for steps marked `skippable === true`. Non-skippable steps cannot be skipped. Steps with downstream dependencies that depend on their output cannot be skipped unless failure policy allows it.

**What it does:**
1. Validates skip eligibility.
2. Sets `Step.status = "skipped"`.
3. Sets `Step.skipReason`.
4. Records `step-skipped` event and `operator-skipped` event.
5. Advances flow to next step.

**State changes:** `interrupted/paused → skipped`.

**Events recorded:** `step-skipped`, `operator-skipped`.

**CLI output:**
```
Skipped step "lint-check".
Reason: operator command
Flow continues to next step.
```

### 5.13 /checkpoint (v0.8-Safe Version)

**When allowed:** Always.

**v0.8 restriction:** Creates named snapshot only. `/restore` command is deferred to v0.9 if restoring from checkpoint is deemed risky. Checkpoints are inspectable via `estacoda flow show <id> --checkpoints`.

**What it does:**
1. Captures current flow snapshot (state, step states, pending approvals, wait reasons, retry counts).
2. Persists as `Checkpoint` record.
3. Records `checkpoint-created` event and `operator-checkpointed` event.
4. Increments `Flow.checkpointCount`.

**State changes:** None.

**Events recorded:** `checkpoint-created`, `operator-checkpointed`.

**CLI output:**
```
Checkpoint "before-deploy" created for flow flow_xyz789.
Step 3/7, state: running
Restore command available in v0.9.
```

### 5.14 /trace

**When allowed:** Always.

**What it does:**
1. Displays flow event timeline.
2. Includes operator events, state transitions, tool executions, approval resolutions.
3. Links to run records and artifacts.

**State changes:** None.

**Events recorded:** None (read-only).

**CLI output:**

## 6. Flow-Safe Compaction

### 6.1 Purpose

Long-running flows accumulate large message histories, tool results, and trajectory events. Compaction reduces context size without losing active execution state or auditability.

### 6.2 Non-Compactable State (Must Be Preserved Exactly)

- Active `flowId` and `currentStepId`
- All steps with status **not** in `{completed, failed, skipped, cancelled}`
- Pending approvals (status = `pending`)
- Pending waits (status = `waiting`)
- Unresolved operator interventions (events that have not been acted upon)
- Active failure state (steps with `failed` status where flow is not yet terminal)
- Retry state (`retryCount`, `maxRetries` for active/interrupted steps)
- Cancellation/interruption state
- Artifact links for active or interrupted steps
- Run links for active or interrupted steps
- Current approval decisions (granted/denied) for active gates
- Unresolved decisions (e.g., operator asked a question, awaiting response)
- Active tool/process handles (see Section 10)
- All `OperatorEvent` records
- All `ApprovalGate` records
- All `Checkpoint` records

### 6.3 Compactable State (May Be Summarized or Truncated)

- Completed steps:
  - Full tool result text → summary (first 500 chars + outcome)
  - Provider token stream events → discard token text, keep metadata
  - Full message history → summarized into `CompactTurnSummary` artifact
- Completed flows:
  - Entire message history → summarized
  - Old trajectory events → archived (moved to `trajectory_archive` table)
- Stale context:
  - Messages older than N turns → summarized
  - Tool results from completed steps → truncated

### 6.4 Summary Artifacts

Compaction produces `CompactSummary` artifacts:

```typescript
type CompactSummary = {
  id: string;
  flowId: FlowId;
  compactedRange: { fromEventId: string; toEventId: string };
  turnSummaries: string[];
  toolOutcomeSummaries: string[];
  operatorActionSummaries: string[];
  createdAt: string;
};
```

### 6.5 Relationship to Memory

Compaction is **independent** of memory. Memory (`MEMORY.md`, `USER.md`) captures long-term learning. Compaction captures short-term context compression. A compacted turn may still produce a memory promotion if the `MemoryProvider` decides it is noteworthy.

### 6.6 Relationship to Trajectory Records

Trajectory events are not deleted during compaction. Token-level `provider-token` events are truncated (text removed, metadata kept). The event itself remains for trace reconstruction.

### 6.7 Relationship to Run Records

`RunRecorder` runs are immutable. Compaction does not modify run records. It may add `CompactSummary` artifacts that link back to run IDs.

### 6.8 Validation Checks

Before compaction:
- Verify no active process handles would be lost.
- Verify all non-compactable state is preserved.
- Verify at least one operator event is retained in full.

After compaction:
- Verify `Flow.compactedAt` is updated.
- Verify `CompactSummary` artifact is recorded and linked.
- Verify `/status` still shows accurate active state.
- Verify `/trace` can reconstruct the full timeline (with summaries for compacted ranges).

### 6.9 Failure Modes

- If compaction fails mid-way, abort and leave state unchanged.
- If compaction produces an invalid summary, detect on next `/trace` and flag.
- If compaction drops non-compactable state, detect via validation and refuse.

- Add `schema_version` table if not present:
  ```sql
  CREATE TABLE schema_version (
    version INTEGER PRIMARY KEY,
    applied_at TEXT NOT NULL
  );
  ```
- Current schema version before v0.8: `5` (assumed from v0.7 evolution schema).
- v0.8 migration: version `6`.

### 7.3 Migration Strategy

- `SQLiteSessionDB` runs migrations on open via `PRAGMA user_version`.
- Migration `5 → 6` creates all tables above.
- Migrations are idempotent (use `CREATE TABLE IF NOT EXISTS`).
- Failed migrations abort startup with a clear error.

### 7.4 Restart Recovery

On `SQLiteSessionDB` initialization:
1. Run pending migrations.
2. Query `SELECT * FROM flows WHERE status IN ('running','paused','waiting','interrupted')`.
3. For each active flow:
   - Load flow snapshot.
   - Load steps.
   - Load pending approvals.
   - Reconstruct wait reasons.
   - Present to operator via `/status` or CLI `estacoda flow list`.
4. Flows in `running` status at shutdown are loaded as `interrupted` (since the process died, we cannot guarantee the turn completed cleanly).
5. Operator must `/resume` to continue.

### 7.5 Stale Lock Recovery

- There are no explicit locks in v0.8. The "lock" is the flow state itself.
- If a flow is `running` but no process owns it, it is treated as `interrupted` on restart.
- If a step is `running` but no turn is active, it is treated as `interrupted`.

### 7.6 Corruption Handling

- If `flows` row is corrupt (invalid JSON), skip it and log an error.
- If `flow_steps` row is corrupt, mark the flow as `failed`.
- If event log is corrupt, reconstruct what is possible from the snapshot.
- Store corruption events in `flow_events` with `kind: "corruption-detected"`.

### 7.7 Snapshot vs Event-Log Relationship

- **Snapshot** (`flows`, `flow_steps`) is read for execution and `/status`.
- **Event log** (`flow_events`) is read for `/trace` and audit.
- On every state transition, the event is appended **before** the snapshot is updated.
- If the snapshot update fails, the event is still in the log. Recovery can reconstruct from log.

## 8. Recorder Integration

### 8.1 Design Principle

**Do not duplicate event systems.** TaskFlow extends the existing recorders rather than creating parallel ones.

### 8.2 What Belongs Where

| Data | Home | Rationale |
|---|---|---|
| Flow state, step state | `flows` / `flow_steps` tables | Fast lookup, restart recovery |
| State transitions, step lifecycle | `flow_events` table | Append-only audit trail |
| Operator commands, interventions | `operator_events` table | Fast operator audit queries |
| Tool calls, provider attempts, token streams | `TrajectoryRecorder` + `trajectory` table | Existing v0.5 substrate |
| Run metadata, tool plans, failures | `RunRecorder` + `runs` table | Existing v0.5 substrate |
| Artifact metadata | `ArtifactStore` + `artifacts` table | Existing v0.5 substrate |
| Artifact-to-flow linkage | `flow_artifacts` table | TaskFlow-specific relationship |
| Run-to-step linkage | `flow_run_links` table | TaskFlow-specific relationship |
| Memory promotions | `memory` system (v0.6) | Independent of TaskFlow |

### 8.3 Linkage IDs

Every turn dispatched by TaskFlow to AgentLoop produces:
- A `runId` from `RunRecorder`
- A `trajectoryId` from `TrajectoryRecorder`

TaskFlow stores these in `flow_run_links`. This allows:
```
flow → step → run → trajectory → tool calls → security decisions
```

### 8.4 How a User Inspects the Chain

```bash
# High-level flow view
estacoda flow show flow_xyz789

# Step detail with run links
estacoda flow show flow_xyz789 --step step_abc

# Trace with full events
estacoda trace flow flow_xyz789

# Security decisions for a step
estacoda trace failures --flow flow_xyz789 --step step_abc
```

### 8.5 TrajectoryRecorder Extension

Add methods to `TrajectoryRecorder`:

```typescript
interface TrajectoryRecorder {
  // existing methods
  record(eventKind: string, data: Record<string, unknown>): void;
  
  // v0.8 additions
  recordFlowEvent(flowId: FlowId, event: FlowEvent): void;
  recordOperatorEvent(flowId: FlowId, event: OperatorEvent): void;
  recordStepEvent(flowId: FlowId, stepId: StepId, event: FlowEvent): void;
}
```

These delegate to the existing `record()` method but prepend `flowId` and `stepId` to the data, ensuring backward compatibility.

## 9. Security and Approval Model

### 9.1 No Bypass of ToolExecutor

**Invariant:** TaskFlow never executes tools directly. Every tool call goes through `ToolExecutor.executeTool()`.

**Enforcement:**
- `TaskFlowEngine` dispatches turns to `AgentLoop`.
- `AgentLoop` uses `ToolPlanRunner`.
- `ToolPlanRunner` calls `ToolExecutor`.
- There is no shortcut.

### 9.2 No Bypass of Security Policy

**Invariant:** Security policy is enforced in `ToolExecutor`. TaskFlow does not reimplement risk classification or hard-block rules.

**Enforcement:**
- TaskFlow may *add* flow-step approval gates, but these gates are **in addition to** tool-level security, not a replacement.
- A tool that is hard-denied by security policy cannot be approved via `/approve`.

### 9.3 No Operator Command Bypass of Approval Policy

**Invariant:** Operator commands are subject to the same risk classification as agent actions.

**Examples:**
- `/skip` of a destructive step is rejected unless the step is explicitly marked `skippable`.
- `/steer` that instructs the agent to bypass security is recorded but the agent still must pass security on subsequent tool calls.
- `/approve` cannot override a hard deny.

### 9.4 No Approval Override of Hard Deny

**Rule:** If `ToolExecutor` returns `decision: "deny"` with `deterministicRule` matching a hard block, `/approve` is rejected.

**Error message:**
```
Cannot approve: this action was hard-denied by security policy (rule: hard-destructive-command).
The operator cannot override hard safety blocks.
```

### 9.5 Idempotency Checks Before Retry

**Rule:** `/retry` is only permitted if:
1. Step `idempotent === true`, OR
2. Operator explicitly confirms with `yes` after a warning.

**Warning message:**
```
Step "deploy-app" is not marked idempotent.
Retrying may cause duplicate side effects.
Confirm retry? (yes/no)
```

### 9.6 Risky Skip Prevention

**Rule:** `/skip` is only permitted if:
1. Step `skippable === true`, AND
2. No downstream step depends on this step's output (or failure policy allows dependency failure).

### 9.7 Safe Cancellation Semantics

**Rule:** `/cancel` terminates the flow but does not delete records. All state is preserved for inspection.

### 9.8 Audit Trail for Approval/Rejection

Every approval/rejection produces:
1. `approval-granted` / `approval-denied` event in `flow_events`.
2. `operator-approved` / `operator-rejected` event in `operator_events`.
3. Updated `ApprovalGate` record in `approval_gates`.
4. Linked to the step, flow, and operator identity.

### 9.9 Distinction Between Blocked States

| State | Meaning | Resolution |
|---|---|---|
| `policy-blocked` | Hard security deny | Cannot be approved. Flow fails or skips. |
| `approval-blocked` | Soft security ask or flow-step gate | Operator `/approve` or `/reject`. |
| `failure-blocked` | Step failed, retries exhausted | Operator `/retry` (if idempotent) or `/skip` (if skippable). |

## 10. Process Ownership and Interruption

### 10.1 The Problem

`ProcessManager` tracks processes independently. `ToolExecutor` does not report process IDs back to the caller. `AgentLoop` has no process cleanup on abort. This means a `/interrupt` might leave orphaned `terminal.run` or `process.start` processes running.

### 10.2 Cancellation Token Model

TaskFlow introduces a **hierarchical cancellation token**:

```typescript
type CancellationToken = {
  flowId: FlowId;
  stepId?: StepId;
  abortController: AbortController;
  childTokens: CancellationToken[];
};
```

- Every flow has a root token.
- Every active step has a child token.
- Every provider turn has a child token.
- Every tool execution receives the step's token signal.

### 10.3 Active Execution Registry

`TaskFlowEngine` maintains an in-memory registry:

```typescript
type ActiveExecution = {
  flowId: FlowId;
  stepId: StepId;
  runId: RunId;
  providerCallId?: string;       // if known
  processIds: string[];          // from ProcessManager
  abortController: AbortController;
};
```

### 10.4 Provider-Call Interruption Limits

- Provider calls are interrupted via `AbortController` signal.
- The provider SDK (OpenAI-compatible) must respect the signal and abort the HTTP request.
- If the provider does not support abortion (e.g., some streaming endpoints), the response is discarded but the connection may linger.
- In v0.8, we accept this limitation. Document: "Provider-call interruption is best-effort."

### 10.5 Terminal/Process Interruption

- When `ToolExecutor` calls `terminal.run` or `process.start`, the `ProcessManager` returns a process ID.
- `ToolExecutor` reports the process ID back to `ToolPlanRunner`.
- `ToolPlanRunner` reports it to `AgentLoop`.
- `AgentLoop` reports it to `TaskFlowEngine`.
- `TaskFlowEngine` registers the process ID in the active execution registry.

On `/interrupt`:
1. Abort the provider call.
2. Signal the `ToolExecutor` abort controller.
3. For each registered process ID, call `ProcessManager.stop(id, "SIGTERM")`.
4. Wait up to 5 seconds for graceful shutdown.
5. If still running, send `SIGKILL`.
6. Record which processes were cleaned up and which were not.

### 10.6 Child Process Cleanup

Same as terminal/process interruption. Child processes are tracked via the active execution registry.

### 10.7 Timeout Behavior

- Graceful shutdown timeout: 5 seconds.
- Force kill timeout: additional 3 seconds.
- If cleanup exceeds 8 seconds total, mark as "cleanup-partial" and continue.

### 10.8 Stale Process Detection

On restart recovery:
1. The active execution registry is empty (in-memory).
2. There is no way to know which processes from a previous run are still alive.
3. **v0.8 limitation:** We do not attempt to detect or kill stale processes from a previous run.
4. **Mitigation:** Document that operators should manually check for orphaned processes after an unclean shutdown.

### 10.9 Failed Cleanup

If process cleanup fails:
1. Record `operator-interrupted` event with `cleanupStatus: "partial"`.
2. Record which process IDs could not be stopped.
3. Continue with flow state transition to `interrupted`.
4. Do **not** leave the flow in an ambiguous state.

### 10.10 State Persisted After Failed Cleanup

- Flow state: `interrupted`
- Step state: `interrupted`
- Event log: `operator-interrupted` with cleanup details
- Snapshot: updated
- Active execution registry: cleared

## 12. Testing Strategy

### 12.1 Deterministic Fake Flow Runner

Create `FakeTaskFlowRunner` for tests:

```typescript
type FakeTaskFlowRunner = {
  createFlow(steps: FakeStep[]): Flow;
  runToCompletion(): Promise<FlowResult>;
  sendCommand(command: OperatorCommand): Promise<void>;
  getState(): Flow;
  getEvents(): FlowEvent[];
};

type FakeStep = {
  name: string;
  behavior: "success" | "fail" | "wait" | "approval" | "throw";
  idempotent?: boolean;
  skippable?: boolean;
};
```

The fake runner simulates AgentLoop and ToolExecutor without real providers or tools.

### 12.2 Test Fixtures

| Test | What It Validates |
|---|---|
| `flow-success` | Flow runs all steps, ends `completed` |
| `flow-wait-resume` | Flow waits, operator resumes, continues |
| `flow-pause-resume` | Flow pauses at boundary, resumes, continues |
| `flow-interrupt-resume` | Flow interrupts, processes cleaned up, resumes |
| `flow-cancel` | Flow cancels, terminal state, no further transitions |

| `flow-approval-reject-skip` | `/reject` with `skip` policy skips step |
| `flow-retry-idempotent` | Failed idempotent step auto-retries |
| `flow-retry-operator` | Operator `/retry` on failed step |
| `flow-skip-skippable` | `/skip` on skippable step succeeds |
| `flow-skip-non-skippable` | `/skip` on non-skippable step rejected |
| `flow-compact-preservation` | Compaction preserves active state |

### 12.3 Testing Without Real Providers

- All TaskFlow tests use `FakeTaskFlowRunner`.
- Provider interaction is mocked at the AgentLoop boundary.
- Tool execution is mocked at the ToolExecutor boundary.
- Security policy is mocked but validates that the mock was called.

## 13. Non-Goals (Explicitly Deferred)

| Feature | Deferred To | Rationale |
|---|---|---|
| Channel-triggered flows | v0.9 | Channels multiply risk; need strong execution foundation first |
| Cron/background flows | v0.9 | Background autonomy requires policy gates beyond v0.8 |

| Marketplace/capability installation | v0.10 | Trusted extension is v0.10 scope |

| Full checkpoint restore | v0.9 | Checkpoint creation is safe; restore needs careful validation |
| Arbitrary autonomous background operation | post-MVP | Requires policy, sandbox, and monitoring not yet built |
| Email/calendar/Telegram execution surface | v0.9 | Channel-specific execution is v0.9 scope |
| Automatic flow creation from skills | v0.8.x | Skills remain advisory; flow creation is operator-initiated in v0.8 |

## 14. Open Questions

### 14.1 Architecture

1. **Does AgentLoop need any contract change to accept a `flowId` and `stepId`?**
   - Option A: Add optional `flowContext` to `AgentLoopInput`.
   - Option B: AgentLoop stays unchanged; TaskFlow wraps the output.
   - *Decision needed before execution plan.*

2. **How does TaskFlow construct the provider context on resume?**
   - Option A: Replay all messages from the session.
   - Option B: Summarize completed steps and send a condensed context.
   - Option C: Hybrid — replay recent N messages, summarize older ones.
   - *Decision needed before execution plan.*

3. **Should TaskFlow be a class instantiated per flow, or a singleton service?**
   - Singleton is simpler but may complicate concurrent flows (if ever supported).
   - *Decision needed before execution plan.*

### 14.2 Persistence

4. **Should flow events be stored in the same table as session events, or separate?**
   - Same table: simpler queries, but event kinds diverge.
   - Separate table: cleaner schema, but joins needed.
   - *Decision needed before schema migration.*

5. **How do we handle schema rollback if v0.8 migration fails?**
   - SQLite does not support transactional DDL reliably.
   - Need a backup-and-restore strategy.
   - *Decision needed before migration code.*

6. **Should checkpoints store full message history, or only state?**
   - Full history: larger but complete.
   - State only: smaller but requires message reconstruction.
   - *Decision needed before checkpoint implementation.*

### 14.3 Operator Controls

7. **Should `/steer` be allowed while a provider is actively streaming?**
   - v0.8-safe: No. Queue for next turn.
   - Advanced: Yes, but requires AbortController + restart of provider call.
   - *Decision needed before `/steer` implementation.*

8. **Should `/pause` attempt to abort the current turn, or wait for the natural boundary?**
   - Natural boundary: simpler, but pause may be delayed.
   - Abort current turn: more responsive, but may lose partial progress.
   - *Decision needed before `/pause` implementation.*

9. **What is the CLI behavior when multiple flows are active in a session?**
   - Hermes has one active session at a time.
   - Should `/status` show all active flows, or only the most recent?
   - *Decision needed before CLI commands.*

### 14.4 Security / Approval

10. **Can an operator `/approve` a tool that was blocked by the auxiliary provider assessor (not a hard deny)?**
    - Proposal: Yes. Only hard-denies are non-overridable.
    - *Decision needed before approval gate design.*

11. **Should flow-step approval gates reuse `WorkspaceApprovalController` or be a separate system?**
    - Reuse: less code, but approval scope is tool+target, not step.
    - Separate: cleaner semantics, but duplicated persistence.
    - *Decision needed before approval implementation.*

12. **What happens if an operator `/approve` while the flow is in a different state (race condition)?**
    - Need atomic state check + transition.
    - *Decision needed before concurrency model.*

### 14.5 CLI / UX

13. **Should `estacoda flow` commands be available outside an active session?**
    - Yes: useful for inspecting past flows.
    - But requires database path resolution without runtime.
    - *Decision needed before CLI wiring.*

14. **Should `/trace` show flow events, trajectory events, or both?**
    - Proposal: Flow events by default. `--deep` flag includes trajectory.
    - *Decision needed before trace command implementation.*

### 14.6 Testing

15. **How many eval fixtures are the minimum acceptable for v0.8?**
    - Proposal: 12 fixtures (covering success, wait, pause, interrupt, cancel, approve, reject, retry, skip, compact, restart, security).
    - *Decision needed before test planning.*

16. **Should TaskFlow tests use the real `SQLiteSessionDB` or an in-memory fake?**
    - Real: tests persistence behavior accurately.
    - Fake: faster, but may miss schema issues.
    - *Decision needed before test infrastructure.*

### 14.7 Documentation

17. **Does v0.8 require a new ADR?**
    - Proposal: Yes — `ADR-0004-taskflow-state-machine.md`.
    - *Decision needed before doc writing.*

18. **Should the v0.8 scope be documented in ROADMAP.md or a separate file?**
    - Proposal: Update ROADMAP.md with v0.8 deliverables.
    - *Decision needed before doc updates.*

### 14.8 v0.9 Handoff

19. **What v0.8 artifacts must be stable for v0.9 channel integration?**
    - Flow state machine must be stable.
    - Operator event schema must be stable.
    - Approval gate schema must be stable.
    - *Decision needed before v0.8 freeze.*

20. **Should v0.8 include any forward-compatible hooks for v0.9 (e.g., channel-triggered flow creation)?**
    - Proposal: No. Keep v0.8 clean. v0.9 will extend the schema.
    - *Decision needed before v0.8 scope lock.*

## Appendix A: Glossary

| Term | Definition |
|---|---|
| **Flow** | A durable, multi-step execution instance with state, events, and recovery. |
| **Step** | A single unit of work within a flow, typically mapping to one AgentLoop turn. |
| **TaskFlow** | The v0.8 runtime subsystem that orchestrates flows. |
| **Operator** | The human user issuing commands like `/pause`, `/resume`, `/steer`. |
| **Turn** | A single provider interaction cycle (prompt → response → tool calls). |
| **Checkpoint** | A named snapshot of flow state for recovery or audit. |
| **Approval Gate** | A blocking point before a step where operator approval is required. |
| **Safe Boundary** | A point in execution where pause/interrupt can occur without corruption (typically between turns). |
| **Restart Recovery** | The process of loading an active flow from persistent storage after a process restart. |
| **Hidden Prompt Mutation** | Changing the system prompt or instructions without recording it as an auditable event. Forbidden. |

## Appendix B: Hermes → EstaCoda Translation Checklist

- [ ] **Run/artifact linkage:** Does the behavior link to `RunRecorder` or `ArtifactStore`?
- [ ] **Security/policy check:** Does the behavior bypass `ToolExecutor` or security policy?
