## 3. Builder Pattern Analysis

### What would a `RuntimeBuilder` look like?

A naive builder would wrap the existing construction order in methods:

```ts
const builder = new RuntimeBuilder(options);
builder.withRegistries();
builder.withProviders();
builder.withTools();
builder.withSkills();
builder.withMemory();
builder.withSecurity();
builder.withRuntimeComponents(); // v0.4 additions
builder.withAgentLoop();
const runtime = builder.build();
```

### Would this reduce the 6 touches?

**No.** Each extracted component still needs:
1. Its constructor arguments resolved from earlier-constructed subsystems
2. Its instance passed to `AgentLoop`

A builder would add an indirection layer without reducing the actual dependency wiring.

### Would this improve testability?

**Marginally, but not enough to justify the work now.**

The real testability blocker is not `createRuntime()` — it is that `AgentLoop` still requires 20+ constructor dependencies. Even with a builder, unit-testing `AgentLoop` would require mocking most of those 20+ dependencies. The builder does not reduce that number.

A more impactful testability improvement would be:
- Extracting `PromptAssembler` from `AgentLoop` (reduces `AgentLoop` deps by ~5)
- Introducing Vitest and writing focused tests for the extracted components (deferred to v0.5)

### Would a DI container help?

**No.** A DI container would add magic (auto-wiring by type name) that conflicts with EstaCoda's explicit construction philosophy. The codebase values readable, traceable object graphs over configuration-driven wiring.

## 5. Decision

**DEFER RuntimeBuilder to v0.5 or later.**

### Rationale

1. **The pain is manageable.** Six mechanical touches over a multi-week decomposition is not a crisis. Each touch was simple, safe, and verifiable with smoke tests.

2. **A builder would not have reduced the touch count.** New components need construction and injection regardless of whether the factory is a function or a builder class.

3. **The real problem is `AgentLoop` constructor bloat, not `createRuntime()` size.** `AgentLoop` still takes 20+ dependencies. A builder does not fix this. Further decomposition of `AgentLoop` (e.g., `PromptAssembler` extraction) would reduce both `AgentLoop` and `createRuntime()` complexity more effectively than a builder.

4. **No unit test framework exists yet.** The primary justification for a builder (testability) cannot be exercised until v0.5 introduces Vitest. Defer the builder decision until after unit tests reveal actual construction pain.

5. **Risk of over-engineering.** A builder introduced now would be a solution in search of a problem. The plan explicitly warns against this: "Builder is a factory with methods, not a container."

### Revisit conditions

Re-assess RuntimeBuilder if any of the following occur:

- `createRuntime()` exceeds 1,200 lines
- A single feature addition requires >3 new constructor calls in `createRuntime()`
- Unit testing reveals that mocking 20+ `AgentLoop` dependencies is impractical
- A plugin/extension system needs dynamic runtime composition
- `AgentLoop` is further decomposed and `createRuntime()` becomes the obvious bottleneck

### Recommended next step (instead of builder)

1. **v0.5:** Introduce Vitest and write focused tests for extracted components. Measure actual construction pain.
2. **v0.5+:** Consider extracting `PromptAssembler` from `AgentLoop` to reduce constructor dependency count.
3. **v0.6+:** If memory/dependency graphs add more construction complexity, re-assess whether a builder or staged factory functions would help.
