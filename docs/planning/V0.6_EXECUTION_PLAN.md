### 1.6 Known Gaps (from code, not docs)

1. **MemoryRenderer renders all entries.** There is no ranking, selection, or relevance filtering. Every non-empty file in `RENDER_ORDER` is included in full.
2. **MemoryPromotion has no trajectory links.** When a preference is promoted, the `sourceSessionIds` are stored, but there is no link to the specific trajectory events that caused the promotion.
3. **No project-level memory scoping enforcement.** `MEMORY.md` is conceptually project-level, `USER.md` is user-level, but both live in `~/.estacoda/` by default. Project overlays exist but scoping is not explicit in the contract.
4. **No code dependency graph.** The docs `dependency-graph.md` and `runtime-knowledge-graph.md` contain Mermaid diagrams only. No code parses imports, exports, or call relationships.
5. **No memory eval fixtures.** While v0.5 added eval fixtures for provider responses and tool security blocks, no evals exist for memory behavior.
6. **Memory content has no timestamps.** Files store text only. No `createdAt`, `updatedAt`, or `lastAccessed` per entry.
7. **Memory search is substring-only.** `LocalMemoryProvider.search()` does a simple `indexOf` over file contents. No semantic search, no scoring beyond binary match.

### 2.3 Mismatches Flagged

**Mismatch 1: "Dependency graph" docs vs reality**
- `docs/architecture/dependency-graph.md` and `runtime-knowledge-graph.md` contain Mermaid diagrams.
- **No runtime code** parses module dependencies, call graphs, or import relationships.

- The roadmap says "project and user memory are scoped separately."
- **Reality:** `MEMORY.md` and `USER.md` are different files, but both default to `~/.estacoda/`. The `MemoryFileKind` type has 4 entries with no scope annotation. Project overlays are supported but scoping is not a first-class concept.

- The roadmap says these should exist in v0.6.
- **Reality:** `inspectPromotions()` and `forgetPromotion()` exist on `LocalMemoryProvider`, but there is no CLI command or agent tool to invoke them. The `memory.curate` tool does append/replace/remove on file contents, not on promotion records.

## 4. v0.6 Objective

**Clear, constrained, implementation-ready objective:**

> Make memory trustworthy by adding provenance, selective rendering, and inspection тАФ and lay the first code dependency graph substrate so the agent can understand project structure beyond file listings.

**More concretely:**

1. Every memory promotion records which trajectory event caused it.
2. Users can inspect and delete memory entries from the CLI.
3. Memory rendering is no longer a blind dump тАФ it respects relevance signals.
4. The system can parse a project's import/export graph and surface it to the agent.
5. Memory behavior is testable through eval fixtures.

---

## 5. Why v0.6 Matters

v0.4 made execution modular. v0.5 made execution observable. **v0.6 makes execution contextual.**

Without v0.6:
- The agent cannot explain *why* it remembers something.
- Memory grows linearly and poisons context with stale entries.
- The agent has no structural understanding of the project (just file listings).
- Users cannot review or correct what the agent has learned.

With v0.6:
- Memory entries carry evidence (trajectory links).
- Rendering is bounded by relevance, not just file size.
- The agent can reason about module relationships.
- Users can inspect and delete learned preferences/facts.

This is the foundation for v0.7's governed skill evolution. A curator cannot propose meaningful skill patches if it cannot trace *why* a memory exists or *how* the project is structured.

## 8. Detailed Task Breakdown

### WS1: Memory Provenance & Inspection

**W1.1: Extend memory contracts with provenance fields**
- Update `MemoryConclusion` in `contracts/memory.ts`:
  - Add `sourceTrajectoryId?: string`
  - Add `sourceEventId?: string`
  - Add `createdAt?: string`
- Update `MemoryPromotionRecord`:
  - Add `sourceTrajectoryId?: string`
  - Add `sourceEventId?: string`
- Update `MemoryPromotionStore` methods to accept and persist provenance.

**W1.2: Wire trajectory events into memory promotion**
- In `RunRecorder` or `AgentLoop`, when a memory conclusion is generated, capture the current trajectory event ID.
- Pass `sourceTrajectoryId` and `sourceEventId` to `memoryProvider.conclude()`.
- Update `LocalMemoryProvider.conclude()` to forward provenance to `MemoryPromotionStore`.

**W1.3: Add memory promotion event to trajectory recorder**
- When a memory is promoted, record a `memory-promotion` trajectory event with:
  - `conclusionId`, `kind`, `targetFile`, `sourceTrajectoryId`

**W1.4: Create `memory-inspector.ts`**
- Functions:
  - `listPromotions(options: { activeOnly?: boolean; kind?: string; limit?: number }): Promise<MemoryPromotionRecord[]>`
  - `findPromotion(id: string): Promise<MemoryPromotionRecord | undefined>`
  - `deletePromotion(id: string): Promise<boolean>` тАФ marks as inactive or removes
- Reads from `MemoryPromotionStore`.

**W1.5: Add `estacoda memory` CLI commands**
- `estacoda memory list [--active-only] [--kind preference|fact] [--limit N]`
- `estacoda memory inspect <id>` тАФ show full record with provenance
- `estacoda memory delete <id>` тАФ soft-delete (mark inactive)
- `estacoda memory trace <id>` тАФ show linked trajectory events

**W1.6: Add `memory.inspect` and `memory.delete` agent tools**
- Agent can list its own memories.
- Agent can delete a memory on user request.
- Both require explicit approval (medium risk).

### WS2: Selective Memory Rendering

**W2.1: Add per-entry metadata to memory files**
- Current format: plain markdown bullet lists.
- New format (backward-compatible): Support optional metadata tags in comments or frontmatter.
- Example: `- Prefer concise replies. <!-- @promoted:2025-05-03 @confidence:0.85 @source:trajectory-abc -->`
- Or store metadata separately in `MemoryPromotionStore` and cross-reference during rendering.

**W2.2: Create `selective-renderer.ts`**
- Input: `MemorySnapshot` + `RenderOptions` (query context, recency bias, max tokens)
- Output: `RenderedMemorySnapshot` with `omittedCount` and `omittedKinds`
- Selection strategy (v0.6):
  1. ** recency**: Entries with newer `updatedAt` rank higher.
  2. **Relevance**: If a query context is provided, prefer entries whose content contains query terms.
  3. **Budget**: Respect `maxChars` per file kind.
  4. **Activity**: Only include active (non-forgotten) entries.
- Fallback: If selection produces empty results, fall back to full dump (with warning).

**W2.3: Integrate selective renderer into `LocalMemoryProvider`**
- `context()` method currently calls `renderMemorySnapshot()` directly.
- Add optional `context(options?: { query?: string })` overload.
- Default behavior remains full render for backward compatibility.
- When query is provided (e.g., from intent router or tool planner), use selective render.

**W2.4: Add freshness display to rendered output**
- Include `<!-- age: 3 days -->` or similar annotation in rendered output.
- Helps the agent understand which memories are recent vs stale.

### WS3: Code Dependency Graph

**W3.1: Create `src/knowledge/code-dependency-graph.ts`**
- Parses TypeScript/JavaScript import/export statements from project files.
- Output: `DependencyGraph` = `{ nodes: ModuleNode[], edges: ImportEdge[] }`
- Scope: Handle `import ... from '...'`, `export ... from '...'`, `require()`.
- Limit: Local project files only. Do not resolve `node_modules` deeply.

**W3.2: Create `src/knowledge/knowledge-graph-provider.ts`**
- Implements a tool interface: `knowledge.query(query: string)`
- Queries:
  - `"what depends on X?"` тЖТ reverse dependency lookup
  - `"what does X import?"` тЖТ forward dependency lookup
  - `"modules in src/tools/"` тЖТ directory-based listing
- Returns structured output, not raw graph.

**W3.3: Add `knowledge` tool to tool registry**
- Tool name: `knowledge.query`
- Risk class: `read` (low risk, only reads project files)
- Available when workspace is trusted.

**W3.4: Cache dependency graph**
- Store parsed graph in `<project>/.estacoda/dependency-graph.json`.
- Invalidate on file modification (check mtime or hash).
- Regenerate lazily on first query after invalidation.

**W3.5: Smoke tests for dependency graph**
- Create a temp project with known imports.
- Query forward and reverse dependencies.
- Assert correct graph structure.

### WS4: Memory Eval & Regression

**W4.1: Create memory eval fixtures**
- `memory-promotion-detects-preference.ts`: Mock user input with preference pattern, assert promotion occurs.
- `memory-rendering-selective.ts`: Create memory snapshot with many entries, assert selective renderer respects budget.
- `memory-provenance-links.ts`: Trigger promotion, assert `sourceTrajectoryId` is set.
- `memory-inspection-lists.ts`: Create promotions, assert inspector lists them correctly.

**W4.2: Extend smoke tests**

## 9. Acceptance Criteria

### 9.1 Functional (Observable Behavior)

- [ ] `estacoda memory list` shows active memory promotions with IDs and content.
- [ ] `estacoda memory inspect <id>` shows provenance (source trajectory ID, event ID, timestamp).
- [ ] `estacoda memory delete <id>` marks the promotion inactive and removes it from rendered output.
- [ ] Memory promotions record `sourceTrajectoryId` and `sourceEventId` when created.
- [ ] Trajectory events include `memory-promotion` kind when promotions occur.
- [ ] Selective renderer omits low-relevance entries when a query context is provided.
- [ ] Selective renderer includes `omittedCount` in output metadata.
- [ ] `knowledge.query` tool returns dependency information for project files.
- [ ] Dependency graph is cached and invalidated on file changes.
- [ ] Memory eval fixtures pass (`estacoda eval memory-*`).

### 9.2 Structural (Architecture Invariants)

- [ ] `MemoryConclusion` and `MemoryPromotionRecord` contracts include provenance fields.
- [ ] `LocalMemoryProvider` still implements `MemoryProvider` without breaking existing callers.
- [ ] `MemoryRenderer` (original) is preserved; selective renderer is a new module.
- [ ] Dependency graph parser does not execute imported code (static analysis only).
- [ ] No new external dependencies for dependency parsing (use regex + TypeScript compiler API if available, or lightweight parser).

### 9.3 Safety (Security/Trust Guarantees)

- [ ] `memory.delete` tool requires explicit approval (medium risk).
- [ ] `memory.inspect` tool is read-only and low risk.
- [ ] Dependency graph parser cannot be tricked into executing code via dynamic imports.
- [ ] Provenance links do not expose sensitive trajectory data (only IDs, not content).
- [ ] Selective renderer cannot be manipulated to hide security-critical memories (e.g., approval settings).

### 9.4 Regression (What Must Not Break)

- [ ] `bun run typecheck` passes.
- [ ] `bun run smoke` passes.
- [ ] Existing memory promotion behavior is unchanged (same detections, same destinations).
- [ ] Existing memory rendering behavior is unchanged when selective rendering is not requested.
- [ ] Existing eval fixtures pass.
- [ ] `estacoda trace` commands still work.
- [ ] `estacoda eval` commands still work.

## 11. Security & Trust Implications

| Concern | Mitigation |
|---------|------------|
| `memory.delete` could remove security policy memories | Require approval; never allow deletion of `SOUL.md` or `AGENTS.md` via inspector |
| Provenance links could leak trajectory content | Store only IDs in provenance, not content |
| Dependency graph parser executes code | Use static regex/AST parsing only; no `require()` or `eval()` |
| Selective renderer hides critical memories | Security-critical entries (in `SOUL.md`, `AGENTS.md`) are always rendered; selective filtering applies only to `MEMORY.md` and `USER.md` |
| Memory inspector exposes all learned preferences | Read-only access is low risk; deletion requires approval |

## 12. Failure Modes

| Failure | Manifestation | Detection |
|---------|--------------|-----------|
| Provenance links are null/undefined | `estacoda memory inspect` shows empty source fields | Smoke test asserts |
| Selective renderer omits too much | Agent loses context, behaves inconsistently | Eval fixture checks omitted count |
| Dependency graph parse fails on complex syntax | `knowledge.query` returns empty or errors | Smoke test with realistic project |
| Cache invalidation misses changes | Stale dependency graph returned | Smoke test modifies file and re-queries |

## 15. Open Questions

These are decisions that **block implementation** and must be resolved:

1. **Q: Should `sourceEventId` reference a trajectory event or a session event?**
   - Trajectory events are v0.5's canonical record. Session events are the legacy record.
   - **Recommendation:** Use trajectory event IDs. They are the future canonical record.
   - **Blocker:** None if we agree.

2. **Q: How should selective renderer handle `SOUL.md` and `AGENTS.md`?**
   - These are agent identity and developer instructions. They should probably always be rendered in full.
   - **Recommendation:** Selective rendering applies only to `MEMORY.md` and `USER.md`. `SOUL.md` and `AGENTS.md` are always rendered in full.
   - **Blocker:** None if we agree.

3. **Q: Should dependency graph use TypeScript compiler API or regex?**
   - TS compiler API is accurate but may be slow and requires `typescript` as a dependency.
   - Regex is fast and dependency-free but incomplete (misses dynamic imports, re-exports).
   - **Recommendation:** Start with regex for v0.6. Consider TS compiler API for v0.7+ if accuracy becomes critical.
   - **Blocker:** None if we agree.

4. **Q: Should memory metadata be inline in markdown or stored separately?**
   - Inline: Human-readable, survives file copy. But pollutes markdown.
   - Separate: Clean markdown. But requires cross-referencing.
   - **Recommendation:** Store metadata in `MemoryPromotionStore` (already exists) and cross-reference during rendering. Keep markdown clean.
   - **Blocker:** None if we agree.
