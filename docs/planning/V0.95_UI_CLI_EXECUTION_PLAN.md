### A.4 Current Command-Output Rendering Patterns

Every command follows one of three patterns:

1. **Lines array:** `const lines = ["header", "  key: value", ""]; return { ok, output: lines.join("\n") }`
   - Used in: `gateway-commands.ts`, `cron-command.ts`, `session-loop.ts` helpers, `cli.ts` commands.
2. **Template literal blocks:** `` output = [`line1`, `line2`].filter(Boolean).join("\n") ``
   - Used in: `cli.ts` setup/settings commands, `one-shot.ts`.
3. **Direct `output.write()`:** Stream-oriented, no buffering. Used in `session-loop.ts` for live events (provider tokens, tool activity).

**No structured output contract exists.** Commands return `string`, not `CommandResultViewModel`.

### A.8 Terminal Capability / Fallback Gaps

**No terminal capability detection exists:**
- No `isTTY` check before ANSI escape codes (`c` in `/clear`, cursor hide/show in `interactive-select.ts`).
- No `NO_COLOR` or `FORCE_COLOR` handling.
- No `TERM=dumb` handling.
- No Unicode capability detection. Emoji (`𓂀`, `💎`, `🧿`) are hardcoded and will print as tofu or mojibake in limited terminals.
- No terminal width detection. `truncate()` in `slash-menu.ts` uses a hardcoded `92` max.
- `interactive-select.ts` uses ANSI cursor controls without checking TTY.
- **No animation gating** exists. No spinner infrastructure exists.

### A.9 Channel-Rendering Implications from v0.9

- v0.9 has four channel backends: Telegram, Discord, Email, WhatsApp.
- Channel rendering is **separate** from CLI rendering. Channels have their own formatters (e.g., Telegram uses HTML/markdown formatting in `src/channels/telegram-formatter.ts` — not fully audited but implied by gateway structure).
- The PRD correctly identifies that channels like Telegram/Discord may use emoji renderers, while CLI default must not assume emoji support.
- **Current gap:** There is no shared `SurfaceAdapter` contract. CLI and channels each build their own strings. The v0.95 architecture must define a `SurfaceAdapter` interface that CLI, Telegram, Discord, etc., can implement.
- `DeliveryRouter` delivers to channels but does not use a unified rendering contract.

### A.10 Stale Assumptions in PRD / Roadmap / Theme Document vs. Current Code

1. **PRD assumes an "enhanced mode" exists to remove.** Current code has **no enhanced mode**. Only plain text output exists. This is a non-issue (nothing to remove).
2. **PRD assumes `src/ui/` contains rendering infrastructure.** Current `src/ui/` has only `settings-labels.ts`. The UI directory must be **created from scratch**.
3. **Theme direction document assumes a CSS variable system exists.** Current `ThemeDefinition` has only branding + activity hex colors. The full token hierarchy (surface, text, border, severity, rail, etc.) must be designed and implemented. CSS conventions are irrelevant for terminal output; tokens map to ANSI/hex.
4. **PRD references `/api/auth/me` field stripping.** This is a Sigil convention, not EstaCoda v2. This section of the PRD is **not applicable** to EstaCoda v2 CLI.
5. **Roadmap mentions "OperatorCommandDispatcher returns plain strings today."** Correct — `taskflow.dispatcher.dispatch()` returns `{ ok, message }` where `message` is a plain string. This must be migrated to return ViewModels or accept a renderer.
6. **PRD says "Current output: string behavior may remain temporarily."** This is accurate — all current output is strings, and backward-compat wrappers will be needed during migration.
7. **Roadmap mentions `runtime.describe()` as a target for ViewModel migration.** Confirmed — `describe()` currently returns a formatted string. The plan must add a **new** structured method (`runtime.getStatus()` or `runtime.describeViewModel()`) while keeping `describe(): string` intact.

## Part B: v0.95 Execution Plan

### B.1 Objective

Move EstaCoda v2 from string-built CLI output to a structured UI architecture:

```
Runtime / Command Data → ViewModel → Renderer → Surface Adapter → Output
```

Introduce terminal capability detection, formalize light/dark theme support with skin overlays, define plain/standard modes, and migrate all major CLI surfaces to the new architecture while preserving exact command semantics and exit codes.

### B.2 Non-Goals

- **Do not** implement a web dashboard or GUI.
- **Do not** add auto/system theme detection in CLI v1.
- **Do not** redesign channel backends (Telegram/Discord/WhatsApp/Email) — but do define the `SurfaceAdapter` contract they can adopt later.
- **Do not** change any tool execution semantics, security policy logic, or runtime behavior.
- **Do not** redesign the interactive onboarding flow structure (only its rendering).
- **Do not** add a separate "enhanced mode." Standard mode is the only rich mode; plain is the fallback.
- **Do not** remove backward-compatible string-returning functions in v0.95.

### B.3 Architecture Principles

1. **ViewModel First:** All new or migrated surfaces must produce a ViewModel before rendering. No direct string building from runtime data in new code.
2. **Renderer Separation:** Renderers handle terminal formatting (ANSI, Unicode, layout). They do not fetch data.
3. **Surface Adapter Separation:** Surface adapters convert rendered output to channel-specific formats (CLI plain text, Telegram HTML, etc.).
4. **Capability-Driven Fallback:** Terminal capability detection determines mode (plain vs. standard), color availability, Unicode support, and animation permission. Never assume capabilities.
5. **Deterministic Plain Output:** `plain` mode, `NO_COLOR`, `TERM=dumb`, non-TTY, CI, and snapshot contexts must produce identical, deterministic output with no ANSI and no animation.
6. **Skin = Token Overlay:** Skins override symbolic tokens (brand treatment, glyphs, spinner frames, progress symbols, tool icon style) but do not replace renderers. The same renderer reads tokens; skins change token values.
7. **Theme = Light | Dark:** Theme selects the base color palette. Skin selects the visual personality. A skin must support both light and dark theme variations.
8. **Emoji Is Skin/Channel Controlled:** Default CLI tool timelines use semantic Unicode/text labels. Emoji icons are only used when the skin or channel explicitly enables them.
9. **Animation Is Capability-Gated:** Spinners and animation are allowed only in interactive TTY standard mode. Animation must be implemented through renderer/status-rail primitives, not ad hoc direct writes.
10. **Preserve Backward Compatibility:** Existing string-returning functions get compat wrappers during migration. Existing tests continue to pass.
11. **Test-Driven Migration:** Every migrated surface gets snapshot tests for plain, standard dark, standard light, no-color, no-Unicode, narrow-width, and Arabic/mixed-script cases.

### B.10 Migration Strategy

1. **Additive-First:** Build all new infrastructure (capabilities, registry, tokens, ViewModels, renderers, adapters, animation) in parallel with existing code. Do not modify existing string builders until the new pipeline is tested.
2. **Strangler Fig:** Migrate one surface at a time. Old string builder becomes a thin wrapper around `ViewModel → Renderer`.
3. **Compat Wrappers:**
   - `runtime.describe()` → **keep exactly as `string`**. Internally may call new structured method + PlainRenderer, but signature and behavior do not change.
   - `renderSlashMenu()` → keep as string. Internally build `ListViewModel` from registry + render.
   - All existing command handlers in `cli.ts` → keep returning `{ output: string }`. Internally use ViewModels where migrated.
4. **Registry as Source of Truth:** After Phase 2, any new command must be added to the registry. `/help`, autocomplete, and startup hints read from the registry. No exceptions.
5. **Test Bridge:** Existing behavior tests (`expect(output).toContain("...")`) continue to pass because compat wrappers preserve string output. New snapshot tests verify the rendering pipeline independently.
6. **Deprecation Path:** Mark string-building helper functions as `@deprecated` after migration. Remove in v0.10 or later, not in v0.95.

### B.11 Backward Compatibility Requirements

- **Command semantics:** All `estacoda <cmd>` behavior must remain unchanged. Exit codes must not change.
- **Slash commands:** All `/commands` in session loop must behave identically.
- **Runtime API:** `Runtime.describe()` must continue to return `string`. New code may use `runtime.getStatus()` or `runtime.describeViewModel()`.
- **Theme API:** The old `kemetBlueTheme` export from `src/theme/kemet-blue.ts` is discarded. It is replaced by the new `lightTheme` / `darkTheme` base themes plus `kemetBlueSkin` overlay. Any code that imported `kemetBlueTheme` directly must be updated to use `resolveTokens(mode, theme, skin)`. The `createRuntime` signature accepting a legacy `ThemeDefinition` may require a compat wrapper during transition.
- **Config API:** No changes to `runtime-config.ts` schema unless adding optional `ui.theme` / `ui.mode` / `ui.skin` fields.
- **Output for scripts:** Plain output must remain parseable. Do not change `ok: true/false` result shapes.
- **String output for tests:** All existing tests must pass without modification during v0.95.

### C.2 Material Changes from This Draft

1. **Runtime.describe() contract preserved.** `describe(): string` is kept intact. New structured status access is provided via `runtime.getStatus()` or `runtime.describeViewModel()`. All new rendering uses the structured object; existing callers are unaffected.
2. **Theme/Skin model separated.** Theme = `light | dark`. Skin = `kemetBlue` (overlay). KemetBlue is the default and only skin in v0.95, but the architecture is extensible for future skins. The existing `src/theme/kemet-blue.ts` is discarded and rebuilt from the approved v0.95 UI direction. KemetBlue supports both light and dark themes. Skins override brand/glyphs/spinner frames/progress symbols/tool icon style.
3. **Animation added to v0.95 scope.** Spinners and progress rail animation are implemented via `AnimationController` primitives. Animation is strictly capability-gated (interactive TTY standard mode only). Explicit tests and manual validation prove provider-token streaming is never corrupted.
4. **Git tags demoted from blocker to release hygiene.** Tag verification stays in final validation checklist but is not a planning-phase concern.
5. **Test baseline clarified.** `bun run test` is the authoritative unit-test gate. `bun run typecheck` is the TypeScript gate. `bun run smoke` is integration-only. Node-only vitest is not authoritative.
6. **Phase 7 expanded.** Now explicitly covers: `/status`, `/model`, `/help`, `/tools`, slash menu/autocomplete, settings, gateway status, gateway diagnose, channels list/status, cron list/show/history/run/pause/resume/remove, sessions list/show/current/attach/detach, handoff output, delivery errors, cron failures, missing config warnings.
7. **Phase 2 added: Command Registry Reconciliation.** A formal command registry is now the single source of truth for command names, aliases, categories, descriptions, visibility, availability, and help metadata. `/help`, autocomplete, startup hints, and docs must not maintain separate hardcoded lists.

### C.3 Remaining Decisions Before Implementation

1. **Test runtime split:** Exclude `bun:sqlite` tests from vitest. Add `bun test` script for full suite. `bun run test` (vitest under Bun) is authoritative.
2. **Default theme for existing users:** If no theme is configured, default to `dark`. Default skin is `kemetBlue` to preserve existing aesthetic.
3. **Flag naming:** CLI accepts `--theme dark|light`, `--mode plain|standard`, `--skin kemetBlue`. Environment variables: `ESTACODA_THEME`, `ESTACODA_MODE`, `ESTACODA_SKIN`.
4. **Layout library:** Add a minimal Unicode width dependency (e.g., `get-east-asian-width` or vendored `wcwidth`). Do not pull in `chalk` or `blessed`.
5. **Animation frame rate:** Default to 80ms per frame. Skin-configurable via `animation.frameIntervalMs`.
