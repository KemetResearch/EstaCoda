# EstaCoda v0.5 Execution Plan: Run Recorder, Trace Schema, and Evaluation Substrate

## Document Type

**Implementation-ready execution plan.** This is not a PRD. It is the directive for the v0.5 engineering run, reconciled against the actual codebase post-v0.4.

## 1. Objective

Make every run **observable, persistable, inspectable, and falsifiable** — and lay the evidence substrate that future self-evolution (v0.7+) will depend on.

Concretely, v0.5 means:

1. Trajectories are no longer lost when the process exits.
2. A user or operator can inspect what happened during a run from the CLI.
3. Failures are classified, not just logged as raw strings.
4. Eval tasks can be run automatically, not just as manual runbooks.
5. The data structures for evidence-backed change proposals exist as typed contracts.
6. Sensitive data in traces is redacted before export or inspection.

The internal standard:

```text
After v0.5, a run should be explainable without reading raw chat logs.
```

## 2. Why v0.5 Matters in the Overall Roadmap

v0.4 decomposed the agent loop. v0.5 makes the decomposed loop **accountable**.

Per the roadmap narrative:

- v0.5 is the **evidence substrate** for AHE-style harness evolution.
- v0.6 (memory) and v0.7 (skill evolution) depend on v0.5's trace and eval infrastructure.
- Without trace persistence, there is no evidence corpus.
- Without failure classification, there is no signal to learn from.
- Without an eval runner, there is no way to prove a candidate is better than baseline.

The roadmap narrative calls v0.5 the place where "the system starts becoming trustworthy." That is the correct ambition, but the implementation must be concrete and bounded.

### 3.4 Mismatches Found Between Roadmap, Docs, and Code

**Mismatch 1: Run recording persistence**
- `ROADMAP.md` (v0.4 section) says: "Run recording is in-memory only."
- **Reality:** `RunRecorder` dual-writes to `SessionDB`, which defaults to `SQLiteSessionDB` in production CLI. Events ARE persisted.
- **Correction needed:** The roadmap understates what v0.4 already achieved. What is actually in-memory-only is the `TrajectoryRecorder`, not the `RunRecorder`.

**Mismatch 2: Skill evolution infrastructure**
- The roadmap narrative describes skill evolution as a v0.7 target.
- **Reality:** `SkillEvolutionStore` already exists with proposals, promotions, observations, eval runs, and JSONL persistence. It is more advanced than the roadmap acknowledges.
- **Correction needed:** v0.7 should build on existing infrastructure, not invent it from scratch.

**Mismatch 3: Eval substrate completeness**
- `docs/evaluation/evaluation-strategy.md` says: "Evaluation substrate: implemented."
- **Reality:** The substrate generates manual runbooks. It does not auto-execute, score, or compare.
- **Correction needed:** Label should be "scaffold implemented; automated runner intended but not implemented."

**Mismatch 4: Trajectory vs Session events**
- `TrajectoryRecorder` and `SessionDB` both record similar events with different schemas.
- **Risk:** Dual recording without a single source of truth creates drift risk. The trajectory should ideally be a derived view of session events, not a parallel stream.
- **Decision needed:** Whether to keep dual recording or derive trajectories from session events.

## 5. Detailed Task Breakdown

### WS1: Trajectory Persistence

**Goal:** Trajectories survive process restart and can be queried.

**W1.1: Add `TrajectoryStore` contract and SQLite implementation**
- Create `src/contracts/trajectory-store.ts`:
  ```ts
  export type TrajectoryStore = {
    save(trajectory: Trajectory): Promise<void>;
    load(id: string): Promise<Trajectory | undefined>;
    listForSession(sessionId: string): Promise<Trajectory[]>;
    listForProfile(profileId: string, options?: { limit?: number; after?: string }): Promise<Trajectory[]>;
  };
  ```
- Create `src/trajectory/sqlite-trajectory-store.ts` implementing `TrajectoryStore`.
  - Store in `~/.estacoda/trajectories.sqlite` or reuse `sessions.sqlite` with a new table.
  - **Decision:** New table in existing `sessions.sqlite` is simpler. Add `trajectories` table with `id`, `session_id`, `profile_id`, `model_id`, `created_at`, `event_json` (JSONL or array), `outcome_json`, `compressed_json`.
- Schema:
  ```sql
  create table if not exists trajectories (
    id text primary key,
    session_id text not null references sessions(id) on delete cascade,
    profile_id text not null,
    model_id text not null,
    created_at text not null,
    completed_at text,
    event_count integer not null default 0,
    events_json text not null,       -- JSON array of TrajectoryEvent
    outcome_json text,               -- JSON object or null
    compressed_json text             -- CompressedTrajectory or null
  );
  create index if not exists idx_trajectories_session on trajectories(session_id, created_at);
  create index if not exists idx_trajectories_profile on trajectories(profile_id, created_at);
  ```

**W1.2: Wire `TrajectoryStore` into `create-runtime.ts`**
- Add `trajectoryStore?: TrajectoryStore` to runtime options.
- Default to SQLite implementation in production, in-memory for tests.
- Inject into `AgentLoop` or `RunRecorder`.

**W1.3: Persist trajectories at session end**
- In `AgentLoop.handle()`, after `trajectoryRecorder.complete(...)` call, save the trajectory via `trajectoryStore.save()`.
- Ensure this happens even on cancellation or error (use `finally` or equivalent).
- The `TrajectoryRecorder` stays in-memory for the active session; persistence happens at completion.

**W1.4: Add trajectory migration to `SQLiteSessionDB`**
- Extend `SQLiteSessionDB.#migrate()` with the trajectories table.
- OR create a separate `SQLiteTrajectoryStore` that manages its own DB.
- **Decision:** Separate DB (`~/.estacoda/trajectories.sqlite`) is cleaner to avoid bloating the session DB. But reusing the session DB connection reduces file handles. **Recommended:** Add table to existing `sessions.sqlite` via migration.

**W1.5: Update smoke tests**
- Add smoke test: create trajectory, complete it, verify it can be loaded back from store.

**W2.3: Implement safe redaction**
- Create `src/trajectory/redaction.ts`:
  ```ts
  export type RedactionPolicy = {
    redactToolInputs?: string[];     // tool names whose inputs should be redacted
    redactToolOutputs?: string[];    // tool names whose outputs should be redacted
    redactSecrets?: boolean;         // strip env var patterns, API keys
    redactFileContents?: boolean;    // strip file.read/write contents over N bytes
    maxContentLength?: number;       // truncate long content fields
  };
  export function redactTrajectory(trajectory: Trajectory, policy: RedactionPolicy): Trajectory;
  ```
- Default policy should redact:
  - `file.read` outputs > 500 chars (truncate)
  - `env` tool outputs entirely
  - Any string matching API key patterns (`sk-...`, `Bearer ...`, etc.)
  - `provider-completion` raw message content (keep metadata)

### WS3: Failure Classification

**Goal:** Every failed run produces a structured classification, not just an error string.

**W3.1: Create `src/contracts/failure.ts`**
```ts
export type FailureClass =
  | "provider-error"          // LLM provider failure (rate limit, timeout, auth)
  | "provider-refusal"        // Provider refused the request
  | "tool-execution-error"    // Tool threw an exception
  | "tool-not-found"          // Tool name invalid or unavailable
  | "tool-blocked"            // Security policy blocked the tool
  | "tool-invalid-args"       // Tool arguments failed schema validation
  | "tool-timeout"            // Tool execution exceeded time limit
  | "plan-dependency-error"   // Tool plan dependency resolution failed
  | "workflow-step-error"     // Skill workflow step failed
  | "budget-exhausted"        // Token or cost budget exceeded
  | "security-escalation"     // Risk escalation aborted the run
  | "user-cancelled"          // User cancelled the run
  | "agent-loop-exhausted"    // Max iterations reached
  | "unknown";                // Unclassified

export type FailureRecord = {
  id: string;
  sessionId: string;
  trajectoryId?: string;
  timestamp: string;
  class: FailureClass;
  sourceEventKind: string;     // e.g., "tool-result", "provider-completion"
  sourceEventId?: string;
  tool?: string;
  message: string;
  recoverable: boolean;
  context?: Record<string, unknown>;
};
```

**W3.2: Create `src/trajectory/failure-classifier.ts`**
```ts
export class FailureClassifier {
  classify(trajectory: Trajectory): FailureRecord[];
  classifyEvent(event: TrajectoryEvent): FailureRecord | undefined;
}
```
- Rules:
  - `provider-completion` with `ok: false` → `provider-error` (check error class for sub-type)
  - `tool-result` with `ok: false` → `tool-execution-error`
  - `tool-gated` → `tool-blocked`
  - `agent-cancelled` → `user-cancelled`
  - `provider-budget-exhausted` → `budget-exhausted`
  - `security-risk-escalated` → `security-escalation`
  - `tool-plan` with `status: "invalid"` → `tool-invalid-args`
  - `tool-plan` with `status: "unavailable"` → `tool-not-found`
  - `provider-iteration` with `exhausted: true` → `agent-loop-exhausted`

**W3.3: Integrate failure classification into run lifecycle**
- After `trajectoryRecorder.complete()`, run `FailureClassifier` on the trajectory.
- Store classified failures alongside the trajectory (new column `failures_json` in trajectories table, or separate `failures` table).
- **Recommended:** Separate table for queryability:
  ```sql
  create table if not exists trajectory_failures (
    id text primary key,
    trajectory_id text not null references trajectories(id) on delete cascade,
    session_id text not null,
    timestamp text not null,
    class text not null,
    message text not null,
    recoverable integer not null default 0,
    context_json text
  );
  create index if not exists idx_failures_class on trajectory_failures(class, timestamp);
  create index if not exists idx_failures_trajectory on trajectory_failures(trajectory_id);
  ```

**W3.4: Add `estacoda trace failures` CLI command**
- List failures with filtering by class, session, date range.

### WS4: Automated Eval Runner

**Goal:** Eval tasks run automatically and produce structured pass/fail results.

**W4.1: Create `src/eval/eval-runner.ts`**
```ts
export type EvalTaskResult = {
  taskId: string;
  status: "pass" | "fail" | "skip" | "error";
  durationMs: number;
  assertions: Array<{ assertion: string; status: "pass" | "fail"; detail?: string }>;
```

**W4.2: Define eval task execution model**
- For tasks with `evidence: "smoke-tested"`: execute the relevant smoke test function programmatically.
- For tasks with `evidence: "live-proven"`: skip by default (require `--live` flag), or mock the provider.
- For tasks with `evidence: "implemented but not live-proven"`: run against the in-memory runtime with mocked provider.
- For tasks with `evidence: "intended but not implemented"`: mark as `skip`.

**W4.3: Refactor smoke tests for eval reuse**
- Current smoke tests are in one 13,969-line file.
- **Decision:** Do NOT refactor all smoke tests in v0.5. Instead, extract 3-5 key smoke test scenarios into standalone `eval/fixtures/` modules that can be imported by both `smoke.ts` and the eval runner.
- Create `src/eval/fixtures/`:
  - `file-roundtrip.ts` — write/read/verify file
  - `provider-completion.ts` — mock provider, verify tool call plan
  - `skill-routing.ts` — verify intent → skill selection
  - `security-block.ts` — verify tool blocking
  - `session-persistence.ts` — verify SQLite session roundtrip
- Each fixture exports a `run(runtime): Promise<FixtureResult>` function.

**W4.4: Replace `scripts/eval-substrate.ts` with `src/eval/cli.ts` or extend it**
- The current script generates manual runbooks.
- **Option A:** Replace with an automated runner.
- **Option B:** Keep the scaffold generator AND add an automated runner.
- **Recommended:** Option B. Rename the script's behavior:
  - `bun run eval:substrate` → still generates scaffold (keep existing).
  - `bun run eval:run` → new automated runner.
  - `bun run eval:run -- --task cli-file-roundtrip` → run single task.

**W4.5: Add golden-flow fixtures**
- Create `evals/golden-flows/` directory.
- A golden flow is a recorded successful run that serves as a baseline.
- Store as JSON: `{ trajectory: Trajectory, expectedOutcome: {...} }`.
- The eval runner can replay or compare against golden flows.
- **Scope for v0.5:** Define the schema and create 1-2 example golden flows manually. Full automation is v0.6+.

## 8. Security and Trust Implications

| Concern | Mitigation in v0.5 |
|---|---|
| Traces may contain secrets | Redaction policy applied by default. API keys, env vars, and long file contents are truncated or stripped. |
| Trace files readable by other users | Store in `~/.estacoda/` with `0700` directory permissions. SQLite DB has umask. |
| Trajectory tampering | Trajectories are append-only during a session. Completed trajectories are immutable. |
| Failure classification could mislead | Classification is heuristic-based, not ground truth. Store confidence field? (v0.5: no, v0.6 maybe). |
| Eval runner executes real tools | Fixtures that use real tools run in isolated temp directories. `--live` flag required for provider-backed tasks. |
| Change manifests could be fabricated | Manifests are locally generated only. No external trust yet. Signature/verification is v0.10+. |

## 9. Risks

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| SQLite schema migration conflicts | Medium | Medium | Add migration incrementally. Test with both fresh DB and existing DB. |
| Trajectory persistence causes I/O bottlenecks | Low | Low | Async writes, WAL mode, batch event inserts if needed. |
| Redaction strips useful debugging info | Medium | Medium | Add `--no-redact` flag for trusted local inspection. |
| Smoke test file grows even larger | High | Medium | Extract fixtures into `src/eval/fixtures/`; import from smoke.ts. |
| Dual recording (SessionDB + Trajectory) drift | Medium | High | Document that trajectories are the canonical evidence format; SessionDB events are operational. Derive trajectories from session events in v0.6 if drift observed. |
| Eval runner requires provider credentials for some tasks | Medium | Low | Skip live-proven tasks by default. Mock provider for automation. |
| Failure classification is incomplete | High | Low | Heuristic-based. "unknown" class is acceptable fallback. Improve over time. |

## 10. Non-Goals

The following are **explicitly out of scope** for v0.5:

1. **Visual dashboard** — No web UI for traces. CLI only.
2. **Real-time streaming** — Traces are persisted at session end, not streamed live.
3. **Self-evolution pipeline** — No candidate generation, no automated promotion. Types and storage only.
4. **Prompt optimization** — AHE warns against prompt-only optimization. v0.5 follows this.
5. **Middleware/runtime evolution** — Change manifest types include `middleware` and `runtime_code` targets, but no implementation.
6. **Team/enterprise audit** — Single-user audit trail only.
7. **Full smoke test refactor** — Only extract 3-5 key fixtures. Full modularization is v0.6+.
8. **Trace replay engine** — Cannot replay a trajectory through the agent loop yet. Schema exists; engine deferred.

## 12. Open Questions / Decisions Needed Before Implementation

### Q1: Single DB or separate DB for trajectories?
- **Option A:** Add `trajectories` and `failures` tables to existing `sessions.sqlite`.
- **Option B:** Create `~/.estacoda/trajectories.sqlite`.
- **Tradeoff:** A is simpler (one connection, one migration path). B is cleaner separation but more file handles.
- **Recommendation:** Option A. Add tables to `sessions.sqlite` via `SQLiteSessionDB.#migrate()`.

### Q2: Should trajectories be derived from session events, or keep dual recording?
- **Current state:** `RunRecorder` writes to both `SessionDB` (events) and `TrajectoryRecorder` (in-memory).
- **Option A:** Keep dual recording. TrajectoryRecorder is the source of truth for the active session.
- **Option B:** Derive trajectories from session events at session end. Eliminates drift.
- **Tradeoff:** A is less refactoring. B is architecturally cleaner but requires a session-event → trajectory-event mapper.
- **Recommendation:** Option A for v0.5. Add a tech-debt note to evaluate Option B in v0.6. The priority is getting persistence working, not refactoring the recording pipeline.

### Q3: How many eval fixtures to extract from smoke tests?
- **Option A:** Extract all smoke test scenarios into fixtures. (High effort, high risk of breakage.)
- **Option B:** Extract 3-5 key fixtures as proof of concept.
- **Recommendation:** Option B. Full extraction is a v0.6 refactoring task.

### Q4: Should redaction be configurable per user?
- **Option A:** Hardcoded default policy.
- **Option B:** Policy stored in `~/.estacoda/config.json`.
- **Recommendation:** Option A for v0.5. Add config option in v0.6 if users request it.

### Q5: Should failure classification run automatically or on-demand?
- **Option A:** Auto-classify at session end (adds latency).
- **Option B:** On-demand when `trace failures` is called.
- **Recommendation:** Option A for simplicity, but make it async (fire-and-forget) so it doesn't block the CLI response. The classification result can be stored lazily.

### Q6: What is the eval runner's relationship to the existing `eval-substrate.ts`?
- **Option A:** Replace the substrate script entirely.
- **Option B:** Keep substrate for manual runs, add `eval:run` for automation.
- **Recommendation:** Option B. The manual runbook is still valuable for live-proven tasks that require human judgment.
