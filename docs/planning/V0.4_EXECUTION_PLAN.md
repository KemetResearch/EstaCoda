# v0.4 Execution Plan: Agent-Loop Decomposition

**Version:** v0.4  
**Status:** ✅ Complete (committed 2026-05-03)  
**Last updated:** 2026-05-03  
**Sources:** `docs/assessments/v0.3-assessment.md`, `docs/architecture/decomposition-targets.md`, `ARCHITECTURE.md`, `ROADMAP.md`, internal v0.4–v0.10 direction  

---

## 1. v0.4 Objective

**Single main goal:** Decompose the 2,714-line `AgentLoop` monolith (`src/runtime/agent-loop.ts`) into inspectable, testable, independently replaceable components with explicit phase boundaries.

**✅ Outcome:** AgentLoop reduced from 2,714 to 809 lines. Six components extracted: `RuntimeRouter`, `ProviderTurnLoop`, `ToolPlanRunner`, `RunRecorder`, `SkillWorkflowExecutor`, `NativeToolExecutor`. All pass `bun run typecheck` and `bun run smoke`.

**Why this matters first:**

- v0.5 (Run Recorder) cannot build a clean trajectory recorder because recording logic is embedded in 25+ private methods inside the loop.
- v0.7 (Skill Evolution) cannot add clean hooks because the loop interleaves skill workflow execution, provider negotiation, and memory promotion with no boundary.
- v0.8 (TaskFlow) cannot add a state machine because the loop owns the entire provider→tool→continue cycle as opaque nested async calls.
- The 830-line `createRuntime()` god factory makes testing impossible: any constructor change cascades through 63 imports.

**Current state grounding:** The product works. Kimi, OpenAI, DeepSeek, and Telegram are live-proven. The skill system is mature. Security boundaries exist. The problem is structural, not functional. The loop handles routing, security, skills, provider negotiation, tool execution, memory promotion, trajectory recording, artifact tracking, and prompt assembly in a single class with shared mutable state.

## 2. Scope

### In scope

- Phase 0: Call graph mapping and cross-component interface design.
- Phase 1: Extract `RunRecorder` with explicit phase hooks.
- Phase 2: Extract `RuntimeRouter` from `IntentRouter` + native intent logic inside `AgentLoop`.
- Phase 3a: Extract provider turn-loop shell (streaming, tool-call extraction, budget).
- Phase 3b: Extract tool-plan runner (planning, execution, concurrency, security approval boundaries).
- Phase 3c: Extract continuation integration (tool-result continuation, failure caps).
- Phase 4: Thin `AgentLoop` to a pure orchestrator.
- Phase 5: Update architecture docs to reflect actual extracted boundaries.
- Phase 6: Assess `RuntimeBuilder` need; implement only if pain is obvious after Phase 4.
- Add Vitest and extract first unit tests from smoke coverage.
- Define `ChangeManifest` types in `src/contracts/` for v0.7.
- Update `AGENTS.md` to remove references to non-existent directories.

### Out of scope

- **No persistence layer for trajectory.** `TrajectoryRecorder` stays in-memory. v0.5 owns SQLite persistence.
- **No full TaskFlow.** Step states, wait/resume, and flow persistence are v0.8.
- **No memory redesign.** Ranking, freshness, and dependency graph are v0.6.
- **No skill evolution pipeline.** Curator, change manifests with evidence, and promotion gates are v0.7.
- **No new channels.** Telegram remains the only product-ready channel.
- **No eval scoring.** The eval substrate exists as a script but v0.5 owns structured scoring.
- **No DI framework.** A builder pattern or phased factory is sufficient. Do not build a container.
- **No behavior changes.** CLI output, provider responses, security decisions, and tool execution must remain identical.

### What should not be refactored yet

- `src/skills/` — mature, well-factored. Only surface changes to support new hooks.
- `src/security/` — policy-driven, stable. Do not change approval semantics.
- `src/providers/` — message normalization and executor are working. Hardening is ongoing but not v0.4 scope.
- `src/channels/` — Telegram adapter is live-proven. No channel refactoring.
- `src/tools/tool-executor.ts` and `src/tools/tool-call-planner.ts` — already extracted. Only change their invocation boundary.
- `src/prompt/prompt-assembly.ts` — 964 lines, but prompt layering is not the v0.4 target.

## 7. Risk Register

| Risk | Why it matters | How to reduce | How to know if it happened |
|------|---------------|---------------|---------------------------|
| **Decomposition breaks live behavior** | AgentLoop is 2,714 lines of interleaved logic. Extracting pieces without changing behavior is hard. | Extract behind interfaces first. Keep existing method bodies as delegates. Migrate incrementally. Verify with smoke after every sub-task. Abort if any extraction task exceeds 800 lines of change. | Smoke fails after a decomposition commit. |
| **Phase 3b breaks security approvals** | Tool execution + security gating is the most sensitive boundary. | Do not change approval semantics. Preserve hard floor exactly. Add dedicated security smoke tests before extraction. | Security smoke tests fail. Adaptive assessor or hard floor behaves differently. |
| **Continuation loop has subtle bugs** | Multi-iteration tool calling has edge cases (failure caps, cancellation mid-stream, partial tool results). | Extract continuation only after tool-plan runner is stable. Add multi-iteration unit tests with mock providers. | Multi-tool smoke tests fail or hang. |
| **Smoke split breaks assertions** | Moving 14k lines of smoke tests risks losing or altering assertions. | Move code blocks, do not rewrite them. Defer smoke split until after Phase 4 (extraction is stable). Use `git diff --check`. | Smoke count changes or assertions differ from v0.3. |
| **Vitest breaks Bun compatibility** | Vitest may have edge cases with Bun's runtime. | Use latest Vitest. Keep test files simple. Have a rollback plan (remove Vitest, keep smoke). | `bun run test` crashes or hangs. |
| **Over-engineering the builder** | Risk of building a full DI container instead of a simple builder. | Phase 6 rule: implement only if pain is obvious. Builder is a factory with methods, not a container. | Builder file exceeds 300 lines or has >5 classes. |
| **Scope creep into v0.5+** | Team adds persistence, eval scoring, or TaskFlow during v0.4. | Explicit out-of-scope list. Require justification for any task not in section 2. Review every PR against scope. | PRs mention SQLite persistence, eval scoring, or flow state machines. |
| **IntentRouter extraction misses edge cases** | Native intent routing has subtle logic for image attachments, voice, and edit prompts. | Add dedicated smoke tests for each native intent before extraction. Run them before and after. | Image generation, voice transcription, or attachment analysis fails in smoke or live test. |
| **Interface design is wrong** | If Phase 0 interfaces don't match reality, every extraction after Phase 1 needs rework. | Phase 0 is a deliverable, not a suggestion. Interfaces are draft but must cover all extraction needs. Revisit after Phase 1 if needed. | Phase 2 or 3 extraction requires changing interfaces defined in Phase 0. |

## 10. Final Builder Instructions

These rules are non-negotiable for any agent or human implementing v0.4:

1. **No code moves before Phase 0 is complete.** The call graph and interface design must exist in `docs/planning/v0.4-call-graph.md` before any extraction begins.

2. **No unrelated refactors.** Do not touch `src/skills/`, `src/security/`, `src/providers/`, `src/channels/`, `src/prompt/prompt-assembly.ts`, or `src/tools/tool-executor.ts` / `tool-call-planner.ts` except to change their invocation boundary.

3. **No dependency additions unless explicitly justified.** Vitest is the only new dev dependency permitted. No production dependencies.

4. **No behavior changes without tests.** If a change alters CLI output, provider response handling, security decisions, or tool execution semantics, it must have a test that catches the difference.

5. **Preserve existing CLI behavior unless explicitly scoped.** All existing commands, slash commands (`/sessions`, `/search`, `/switch`, `/reset`, `/yolo`), and interactive flows must continue to work identically.

6. **Commit in small reviewable layers.** Each task in section 4 should be its own commit or PR. Maximum diff size per commit: 800 lines. If a task exceeds 800 lines, split it into sub-tasks.

7. **Smoke tests are the release gate.** `bun run smoke` must pass after every commit. If smoke fails, the commit is rejected.

8. **Type safety is non-negotiable.** `bun run typecheck` must pass with zero errors after every commit.

9. **Bun-only.** Do not add Node compatibility shims. v0.4 does not solve the Bun lock-in (that is R04, targeted for v0.4 mitigation but not elimination).

10. **Do not build a DI framework.** The builder is a factory with methods. No `register()` / `resolve()` / reflection. If `RuntimeBuilder` exceeds 300 lines, it is over-engineered.

11. **Do not add persistence.** Trajectory and artifacts stay in-memory. v0.5 owns SQLite persistence.

12. **Do not add eval scoring.** The eval substrate script remains as-is. v0.5 owns structured scoring.

13. **Document as you go.** Update `docs/architecture/` files in the same commit that changes the code. Do not defer doc updates to the end.

14. **Abort criterion:** If any extraction task requires changing >800 lines or moving >3 methods simultaneously, pause and revisit Phase 0. The boundary is wrong.
