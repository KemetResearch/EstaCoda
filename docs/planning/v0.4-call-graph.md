## 4. Hot Path Analysis

### 4.1 Typical Provider-Enabled Turn (most common)

```
handle()
  → context expansion (async I/O)
  → intent routing (sync, in-memory)
  → security assessment (sync, policy check)
  → #executeDeterministicNativeTools (usually no-op)
  → #runProviderLoop()
      → #completeWithProvider() ── iteration 0
          → #providerSessionHistory()
          → assembleProviderPrompt()
          → providerExecutor.complete() ── network I/O
      → #executeProviderToolPlans()
          → tool grouping
          → #executeProviderToolPlan() × N (concurrent or sequential)
              → toolExecutor.executeTool() ── may do I/O
      → [if toolCalls returned and not exhausted]
      → #continueProviderAfterTools() ── iteration 1+
          → #providerSessionHistory()
          → assembleProviderContinuationPrompt()
          → providerExecutor.complete() ── network I/O
      → [loop until no toolCalls or budget exhausted]
  → #recordSkillOutcomes()
  → response assembly
  → #promoteRepeatedPreferences()
```

**Latency-critical segments:**
1. `providerExecutor.complete()` — network-bound, streaming
2. `toolExecutor.executeTool()` — variable; may be local CPU or network
3. `sessionDb.listMessages()` + `packSessionHistory()` — disk I/O

**CPU-critical segments:**
1. `groupProviderToolPlans()` — O(n) over tool plans
2. `assembleProviderPrompt()` — string assembly, token counting

### 4.2 Deterministic Skill Workflow Turn (provider unavailable)

```
handle()
  → [same routing as above]
  → #executeSkillWorkflow()
      → compileSkillWorkflowPlan()
      → #executeWorkflowStep() × steps
          → toolExecutor.executeFirstAvailable() or executeTool()
  → [skip provider loop entirely]
  → response assembly
```

This path is entirely local — no network calls except tool execution.

### 4.3 Cancellation Points

`isAborted(input.signal)` is checked at:
1. Start of `handle()` (before context expansion)
2. After context expansion (before routing)
3. Start of `#runProviderLoop()` each iteration
4. Start of `#executeSkillWorkflow()` each step
5. After provider loop completes (before skill outcomes)

**Implication:** Cancellation is coarse-grained. There is no mid-provider-call cancellation propagation — the signal is passed to `providerExecutor.complete()` and `toolExecutor.executeTool()`, but those subsystems must respect it internally.

## 7. Extraction Order with Justification

### Phase 1: RunRecorder (~Days 6–10)

**Why first:**
- All `#record*` methods are **pure side-effects** — they write to `sessionDb` and `trajectoryRecorder` but never return data that alters control flow.
- Lowest behavioral coupling = lowest regression risk.
- No iteration loops, no provider negotiation, no tool execution.
- Easy to verify: if events still land in sessionDb and trajectory, it works.

**What moves:**
- `#recordWorkflowStep`, `#recordWorkflowPlan`, `#recordArtifactsFromExecutions`, `#recordToolPlan`, `#recordProviderBudgetExhausted`, `#recordCancellation`, `#recordProviderIteration`, `#recordRouteUsage`, `#recordSecurityRiskEscalation`, `#recordPromptAssembly`, `#recordSkillOutcomes`, `#markPlannedToolPlansCancelled`, `#appendCancelledAssistantMessage`, `#latestResumeNote`
- 16 free functions from §6.1 + `buildResumeNote`, `summarizeAttachments`, `normalizeAttachments`

**Estimated LOC extracted:** ~700 lines from class + ~350 free functions = **~1,050 lines**.

**Abort check:** This is >800 lines but it's composed of 14 independent methods that can be moved one at a time. The 800-line rule applies to a *single extraction task*, not cumulative. Each method is <100 lines.

### Phase 2: RuntimeRouter (~Days 10–15)

**Why second:**
- Encapsulates all "before provider" decision logic: intent routing, skill selection, security assessment, deterministic native tools, fallback response building.
- Has clear inputs (`AgentLoopInput`) and outputs (`RuntimeRouteResult`).
- Once extracted, `handle()` becomes:
  ```ts
  const route = await this.#runtimeRouter.route(input);
  const loopResult = await this.#providerTurnLoop.run({ ...route, ... });
  ```

**What moves:**
- Intent routing block from `handle()` (lines 206–536, stripped of recording calls)
- `#executeDeterministicNativeTools`
- 8 free functions from §6.2

**Estimated LOC extracted:** ~400 lines.

### Phase 3b: ToolPlanRunner (~Days 20–27)

**Why third (before ProviderTurnLoop):**
- ToolPlanRunner is the most **complex behavioral component** — grouping, concurrency, risk escalation, repeated failure tracking.
- Extracting it first means ProviderTurnLoop can consume it as a dependency, making ProviderTurnLoop thinner.
- Per execution plan: 3b comes before 3a because 3a depends on 3b's output shape.

**Correction:** ToolPlanRunner must be extracted **before or simultaneously with** ProviderTurnLoop. The execution plan says 3a then 3b then 3c, but the dependency graph says 3b before 3a.

**Resolution:** Extract ToolPlanRunner first (as 3a), then ProviderTurnLoop (as 3b). The execution plan's numbering is cosmetic; the dependency graph rules.

## 13. Addendum: Pre-Extraction Clarifications

This section records architectural decisions that must hold before any code is moved.

### 13.1 RuntimeRouter must NOT own security decisions

The initial interface draft (§5.2) incorrectly placed `securityDecision` and `initialRiskClass` inside `RuntimeRouteResult`. This is wrong. Security decisions must remain in `AgentLoop` (or a dedicated `SecurityGate`) because:

- Routing is evidence-gathering (intent labels, suggested skills, provider route).
- Security is policy enforcement (risk class, approval mode, workspace trust).
- Conflating them means `RuntimeRouter` needs access to `SecurityPolicy`, `WorkspaceTrustStore`, and approval state — re-creating the god-object problem.

**Corrected boundary:**

```ts
export interface RuntimeRouter {
  route(input: AgentLoopInput): Promise<RuntimeRouteResult>;
}

export type RuntimeRouteResult = {
  effectiveText: string;
  routedText: string;
  attachments: ChannelAttachment[] | undefined;
  context: ContextExpansionResult | undefined;
  intent: IntentRoute;
  selectedSkill: LoadedSkill | SkillDefinition | undefined;
  selectedSkillInstructions: string | undefined;
  selectedSkillResources: LoadedSkill["resources"] | undefined;
  selectedSkillSetup: SkillSetupContext | undefined;
  deterministicNativeTools: { executions: ToolExecutionRecord[]; plans: ToolCallPlan[] };
  fallbackResponse: AgentLoopResponse;
  // REMOVED: securityDecision, initialRiskClass ← these stay in AgentLoop
};
```

`AgentLoop` will call `assessSecurityPolicy` **after** `RuntimeRouter.route()` returns, using the selected skill and context. This preserves the principle: **router gathers evidence; security enforces policy**.

### 13.2 ToolPlanRunner must be split into explicit sub-steps

The initial interface draft (§5.4) presented `ToolPlanRunner.executePlans()` as a single opaque call. This hides complexity and makes testing difficult. The actual work inside `#executeProviderToolPlans` (lines 1013–1097) breaks into 5 sequential sub-steps that must be visible:

| Sub-step | Responsibility | Current code location |
|----------|--------------|----------------------|
| **1. Conversion** | Convert raw provider `toolCalls` into `ToolCallPlan` objects via `toolCallPlanner.planFromProviderDelta()` | Lines 1038–1052 |
| **2. Grouping** | Group plans into concurrency-safe batches via `groupProviderToolPlans()` | Lines 1053–1054 |
| **3. Security preservation** | Check risk escalation against current baseline; emit `security-risk-escalated` event if crossed | Lines 1054–1064 |
| **4. Execution delegation** | Run each group (concurrent `Promise.all` or sequential `for...of`) via `#executeProviderToolPlan()` → `toolExecutor.executeTool()` | Lines 1066–1091 |
| **5. Result formatting** | Collect executions, update `maxObservedRisk`, return aggregated result | Lines 1093–1097 |

**Revised interface:**

```ts
export interface ToolPlanRunner {
  // Orchestrator method — calls sub-steps in order
  executePlans(input: ToolPlanRunnerInput): Promise<ToolPlanRunnerResult>;

  // Sub-step hooks (exposed for testing / future middleware)
  convertPlans(toolCalls: ProviderToolCall[], toolCallPlanner: ToolCallPlanner): ToolCallPlan[];
  groupPlans(plans: ToolCallPlan[], definitions: Map<string, ToolDefinition>, maxConcurrent: number): ToolPlanGroup[];
  checkRiskEscalation(group: ToolPlanGroup, currentBaseline: ToolRiskClass): ToolRiskClass | undefined;
  executeGroup(group: ToolPlanGroup, input: { trustedWorkspace: boolean; signal?: AbortSignal }): Promise<ToolExecutionRecord[]>;
}

export type ToolPlanGroup = {
  concurrent: boolean;
  plans: ToolCallPlan[];
  definitions: ToolDefinition[];
  riskClass: ToolRiskClass;
};
```

**Why this matters:** Each sub-step is independently testable. In v0.7, the Curator can inspect grouping decisions or risk escalation patterns without parsing an opaque blob.

### 13.3 ProviderTurnLoop vs ProviderTurnExecutor: naming and responsibility resolution

The initial archaeology identified two nearly identical methods (`#completeWithProvider`, `#continueProviderAfterTools`) and proposed unifying them. Before extraction, we must decide:

**Option A: ProviderTurnLoop (outer) + ProviderTurnExecutor (inner)**
- `ProviderTurnLoop`: Owns the iteration budget, decides when to call initial vs continuation, manages `mergeProviderExecutions`, handles abort/signal.
- `ProviderTurnExecutor`: Owns a single provider turn (one `complete()` call). Receives a phase flag (`"initial" | "continuation"`). Assembles prompt, calls provider, maps events, records to sessionDb.

**Option B: Single ProviderTurnLoop with internal phases**
- `ProviderTurnLoop` has a private `#executeTurn(phase)` method.
- No separate `ProviderTurnExecutor` class.
- Simpler but less testable (provider turn logic is not independently mockable).

**Decision: Option A.**

Rationale:
- `ProviderTurnExecutor` is independently testable with a mock `ProviderExecutor`.
- `ProviderTurnLoop` is independently testable with a mock `ProviderTurnExecutor`.
- This matches the decomposition target: "clearer boundaries between router, planner, executor, recorder."
- Post-v0.4, `ProviderTurnExecutor` can be extended for multi-provider negotiation without touching the loop logic.

**Responsibility split:**

| Concern | ProviderTurnLoop | ProviderTurnExecutor |
|---------|------------------|----------------------|
| Iteration budget (max iterations) | ✓ | ✗ |
| Tool-call budget (max calls) | ✓ | ✗ |
| Wall-clock budget | ✓ | ✗ |
| Repeated failure budget | ✓ | ✗ |
| Decide "initial" vs "continuation" | ✓ | ✗ |
| Merge provider results across iterations | ✓ | ✗ |
| Assemble prompt (initial vs continuation) | ✗ | ✓ |
| Call `providerExecutor.complete()` | ✗ | ✓ |
| Map provider runtime events | ✗ | ✓ |
| Record provider completion event | ✗ | ✓ (via RunRecorder) |
| Call `ToolPlanRunner.executePlans()` | ✓ | ✗ |
| Return final merged result | ✓ | ✗ |

### 13.4 ToolPlanRunner-specific abort criteria

The general abort criterion (“any extraction >800 lines”) is not sufficient for ToolPlanRunner because it is the highest-risk extraction. Additional abort criteria:

1. **If grouping behavior changes for any tool combination in smoke.ts:** Immediately revert. Grouping determines concurrency safety. A bug here could run destructive tools in parallel.
2. **If risk escalation event is not emitted in the same session event sequence:** Abort. Other components (security dashboard, audit log) depend on this event ordering.
3. **If `toolPlans` array mutation changes:** Abort. `#executeProviderToolPlans` mutates `input.toolPlans` in-place (pushes new plans). The caller (`#runProviderLoop`) reads `input.toolPlans.length` before and after to detect recoverable feedback. Any change to push timing or plan ordering breaks continuation logic.
4. **If `repeatedFailures` Map key format changes:** Abort. The key is `${toolName}:${resultContent.slice(0, 160)}`. Changing this changes when the repeated-failure budget triggers.
5. **If any tool execution moves from sequential to concurrent or vice versa without explicit `isConcurrentSafeTool` check:** Abort.

**Mitigation:** Before extracting ToolPlanRunner, write unit tests for:
- `groupProviderToolPlans` with mixed safe/unsafe tools
- `maxRiskClass` with all risk class combinations
- `isConcurrentSafeTool` for `terminal.run` and `process.start`
- `recordRepeatedToolFailures` with identical vs different error content

These tests must pass **before** the code moves. They become the safety net.
