# EstaCoda v0.9 — Hermes Comparison & Open Question Resolution

**Status:** Read-only audit phase. No code changes.
**Sources:** Hermes Agent source (github.com/NousResearch/hermes-agent) and documentation (hermes-agent.nousresearch.com/docs).
**Date:** 2026-05-04

## 1. Hermes Comparison Summary

### What Hermes Does That Matters for v0.9

**Gateway Architecture**
- Single gateway process (`gateway/run.py`) starts all configured platform adapters.
- Adapters are thin and share the same runtime path (`AIAgent` created per session).
- Cron delivery prefers live adapters (for E2EE) but falls back to standalone HTTP/send paths.
- Adapters register with a shared `DeliveryRouter` (`gateway/delivery.py`).

**Sessions**
- All sessions (CLI, Telegram, Discord, Email, etc.) live in one SQLite database (`~/.hermes/state.db`).
- Session keys are deterministic: `agent:{profile}:{platform}:{chat_type}:{chat_id}[:{user_id}]`.
- Per-user isolation in groups is the default (`group_sessions_per_user: true`).
- Auto-reset policies: `idle`, `daily`, `both`, `none`.
- **Hermes does NOT have explicit CLI↔Telegram handoff commands.** CLI and gateway share the session DB substrate, but there is no `/attach <session-id>` or handoff code system. Resuming a session from Telegram means messaging the same chat (same session key); resuming from CLI means `hermes sessions resume <id>`.

**Telegram**
- `python-telegram-bot` library.
- Long polling default; webhook mode optional.
- Numeric user IDs for allowlist (`TELEGRAM_ALLOWED_USERS`).
- `/sethome` command sets home channel for cron delivery.
- Privacy mode handling documented.
- Voice memo auto-transcription; TTS delivered as native voice bubbles.
- MEDIA tags extracted for native attachment delivery.
- Approval prompts use inline keyboard buttons.

**Discord**
- `discord.py` library.
- Requires `Message Content Intent` and `Server Members Intent`.
- DM sessions separate from server channel sessions.
- `group_sessions_per_user: true` by default — Alice and Bob in `#research` get separate sessions.
- `DISCORD_ALLOWED_USERS`, `DISCORD_ALLOWED_GUILDS`, `DISCORD_ALLOWED_CHANNELS`.
- `DISCORD_FREE_RESPONSE_CHANNELS` for no-mention-required channels.
- Slash commands and prefix fallback.
- Streaming/progressive responses.
- Voice receiver exists (experimental).
- Allowed mentions locked down (`@everyone`/`@role` disabled by default).

**WhatsApp**
- **Baileys** (`@whiskeysockets/baileys`) — WhatsApp Web bridge, not official API.
- QR code pairing; session saved to `~/.hermes/platforms/whatsapp/session`.
- Single-user DM-first behavior.
- `dm_policy`: `open` | `allowlist` | `disabled`.
- `group_policy`: `open` | `allowlist` | `disabled`.
- `require_mention` configurable.
- Streaming (progressive) responses supported.
- Message chunking at 4,096 characters.
- Progress indicators for tool calls.
- Account risk acknowledged in docs.

**Email**
- Python `imaplib` (receive) + `smtplib` (send).
- `EMAIL_ALLOWED_USERS` comma-separated allowlist.
- `EMAIL_ALLOW_ALL_USERS=true` support.
- `EMAIL_HOME_ADDRESS` for cron default delivery.
- Poll interval default **15 seconds** (`EMAIL_POLL_INTERVAL`).
- On connect: marks ALL existing messages as seen (only processes new mail).
- Self-message filter (prevents reply loops).
- Automated sender filter: `noreply`, `mailer-daemon`, `bounce`, `no-reply`, `Auto-Submitted`, `Precedence: bulk`, `List-Unsubscribe`.
- Reply-in-thread via `In-Reply-To` and `References` headers.
- Thread context stored per sender (`sender_addr` → `{subject, message_id}`).
- Attachments cached locally; images go through `cache_image_from_bytes`, documents through `cache_document_from_bytes`.
- `skip_attachments` config option.
- `MEDIA:/path` outgoing attachments delivered as email attachments.
- **No email-specific approval layer.** Normal runtime policy applies.

**Cron**
- Jobs stored in `~/.hermes/cron/jobs.json`.
- Output saved to `~/.hermes/cron/output/{job_id}/{timestamp}.md`.
- **Fresh session every run:** `_cron_session_id = f"cron_{job_id}_{timestamp}"`.
- **Recursion guard:** `disabled_toolsets=["cronjob", "messaging", "clarify"]`.
- **Duplicate prevention:** File-based lock (`~/.hermes/cron/.tick.lock`) + `advance_next_run()` called **before** execution under the lock.
- **Delivery targets:** `origin`, `local`, `telegram`, `telegram:chatId`, `discord:channelId`, `email:address`, comma-separated combinations.
- `wrap_response` on by default (adds "Cronjob Response: {name}" header).
- `SILENT_MARKER` (`[SILENT]`) suppresses delivery.
- Parallel execution supported; workdir jobs serialized.
- Inactivity timeout default 600s.
- `mark_job_run` updates `last_run_at`, `last_status`, `last_error`, `last_delivery_error` on the job object itself.
- **No separate execution history store.** History is the job's `last_status` + output markdown files.

**Delivery Routing**
- `DeliveryRouter` class exists (`gateway/delivery.py`).
- Parses `origin`, `local`, `telegram`, `telegram:chatId:threadId`.
- Resolves home channels via env vars (`TELEGRAM_HOME_CHANNEL`, `DISCORD_HOME_CHANNEL`, etc.).
- Truncates oversized output to platform limits (`MAX_PLATFORM_OUTPUT = 4000`).
- Saves full output to disk when truncated.
- Prefers live adapter delivery; falls back to standalone.

**Approval / Security**
- `PairingStore` (`gateway/pairing.py`): 8-char codes from 32-char alphabet, 1-hour expiry, max 3 pending per platform, rate limiting (1 req per 10 min), lockout after 5 failed attempts.
- `EMAIL_ALLOWED_USERS` and per-platform `ALLOWED_USERS` for gatekeeping.
- Runtime policy handles tool approvals; no extra channel-specific friction.
- Cron sets `HERMES_CRON_SESSION=1` so approval system can apply `cron_mode`.

**Operator Commands**
- CLI: `hermes sessions list/prune/rename/delete/stats/export`, `hermes gateway status/diagnose`.
- Channels: `/status`, `/help`, `/sessions`, `/new`, `/resume`, `/title`, `/cron list`, `/cron pause`, etc.
- Telegram has callback buttons for approvals.

## 3. Required Parity Decisions

These areas must match Hermes behavior exactly (or as exactly as the TypeScript/Bun stack allows):

1. **Delivery target syntax** — `origin`, `local`, `telegram:chatId:threadId`, `discord:channelId:threadId`, `email:address`, comma-separated multi-target.
2. **Email behavior** — IMAP UNSEEN polling, SMTP reply-in-thread with `In-Reply-To`/`References`, self-message filter, noreply/automated filter, `EMAIL_ALLOWED_USERS`/`EMAIL_ALLOW_ALL_USERS`, `EMAIL_HOME_ADDRESS`, `skip_attachments`, `MEDIA:/path` outgoing attachments, 15s default poll interval.
3. **Discord session model** — DM per user, server channel per-user by default, thread-aware keys, `groupSessionsPerUser: true` default.
4. **WhatsApp library** — Baileys (`@whiskeysockets/baileys`), QR pairing, session persistence, DM-first, allowlist policy.
5. **Cron fresh session default** — Every cron run gets a new session ID. No linked-session default.
6. **Cron recursion guard** — Disable `cronjob`, `messaging`, `clarify` toolsets inside cron executions.
7. **Cron duplicate prevention pattern** — File lock + advance next run before execution.
8. **Gateway process model** — Single process, all adapters started together, shared runtime.
9. **Approval/security model** — `PairingStore`-style code-based pairing (8-char, 1h expiry, rate limits, lockout). No email-specific or channel-specific approval layers beyond global policy.
10. **Home channel model** — `/sethome` on Telegram, `TELEGRAM_HOME_CHANNEL` env var, equivalent for Discord/Email. Cron delivers to home channel when `deliver=origin` and origin is missing.

## 4. Intentional Divergences

These areas deliberately differ from Hermes:

1. **Test runner** — Vitest instead of Python pytest. Necessary because EstaCoda is TypeScript/Bun.
2. **Cron execution history** — SQLite `cron_executions` table instead of job-object mutation only. EstaCoda needs queryable history to satisfy v0.9 acceptance criteria ("inspect what ran, what failed, and what is waiting"). Hermes only tracks `last_status`.
3. **Cross-surface handoff** — Explicit `attach`/`detach` with short-lived handoff codes. Hermes relies on substrate-level sharing (same DB) but has no explicit CLI↔channel handoff. EstaCoda's model prevents accidental context bleed.
4. **Email/Discord implementation language** — Node.js `imap`/`nodemailer` and `discord.js` instead of Python `imaplib`/`smtplib`/`discord.py`. Necessary because EstaCoda is a Node/Bun project.
5. **Cron lock recovery** — PID-based stale lock stealing. Hermes' file lock can deadlock after a crash; EstaCoda adds recovery without changing the core pattern.
6. **Session store surface pointers** — Hermes maps session keys directly to session IDs. EstaCoda adds an explicit `surfacePointer` → `sessionId` mapping layer so attach/detach is a pointer update, not a session migration.

## 5. v0.9 Plan Changes

### Changes to Phases

**No changes to phase ordering** — the Hermes comparison validates the revised audit's phase structure:

- **Phase 0 (DeliveryRouter + Test Runner)** — Hermes has `DeliveryRouter`; we should too. Hermes has extensive tests; we need Vitest.
- **Phase 1 (CLI↔Telegram Handoff + Telegram Hardening)** — Hermes does not have explicit handoff, so EstaCoda's Phase 1 is a genuine improvement that must be proven before generalizing.
- **Phase 2 (Discord + Email + WhatsApp Arch)** — Hermes supports all three; EstaCoda should build them in parallel using the proven router and handoff model.
- **Phase 3 (Cron Reliability)** — Hermes cron is reliable; EstaCoda should match its patterns (fresh sessions, lock+advance, delivery routing) while adding the execution history store.
- **Phase 4 (Operator Surface + Docs)** — Hermes has CLI `gateway status`, `sessions list`, etc. EstaCoda should expose equivalent commands.

### Changes to Acceptance Criteria

1. **Email criteria** — Add explicit criteria:
   - "Email behavior is not stricter than Hermes" → verify by checking that no email-specific approval escalation exists.
   - "Self-messages and automated senders are filtered" → match Hermes' `_is_automated_sender` logic.
   - "Reply-in-thread uses In-Reply-To and References headers" → exact Hermes parity.

2. **Cron criteria** — Modify:
   - "Cron-triggered runs produce trajectory records" → add "with `source: 'cron'` and a fresh session ID per run."
   - "Failed executions are classified" → add failure classes: `script-failed`, `runtime-error`, `delivery-failed`, `timeout`, `provider-error`.
   - "A gateway restart does not duplicate due jobs" → specify "uses file lock + advance next_run before execution, with PID-based stale lock recovery."

3. **Discord criteria** — Add:
   - "Discord sessions are per-user by default in server channels" → match Hermes `group_sessions_per_user: true`.
   - "Discord respects allowed users, guilds, and channels" → match Hermes `DISCORD_ALLOWED_USERS`, `DISCORD_ALLOWED_GUILDS`, `DISCORD_ALLOWED_CHANNELS`.

4. **WhatsApp criteria** — Add:
   - "Live adapter is marked @experimental in source and docs" → because Hermes acknowledges Baileys carries account risk.
   - "WhatsApp uses Baileys linked-device model" → match Hermes exactly.

### Changes to Workstreams

- **Workstream A (DeliveryRouter)** — Explicitly adopt Hermes `DeliveryTarget.parse()` logic as the specification. The router must resolve `origin`, `local`, `telegram:chatId:threadId`, `discord:channelId:threadId`, `email:address`.
- **Workstream B (Test Runner)** — Add explicit requirement: fake adapters must be able to simulate Hermes-equivalent delivery behavior (truncation, media routing, thread_id passing).
- **Workstream C (Handoff)** — Document that this is an **intentional divergence** from Hermes. Add acceptance criterion: "Handoff requires explicit command or short-lived code; no automatic session merging."
- **Workstream F (Email)** — Add requirement: "Email adapter must mirror Hermes `EmailAdapter` behavior exactly, including noreply filtering, thread headers, and attachment caching."
- **Workstream H (Cron)** — Add requirement: "Cron execution history must be queryable (SQLite), not just `last_status` on the job object."

### Changes to Config/Env Shapes

Adopt Hermes env var names where they exist:

- `TELEGRAM_BOT_TOKEN` ✓ (already used)
- `TELEGRAM_ALLOWED_USERS` ✓
- `TELEGRAM_HOME_CHANNEL` → add for cron delivery default
- `DISCORD_BOT_TOKEN` ✓
- `DISCORD_ALLOWED_USERS` ✓
- `DISCORD_ALLOWED_GUILDS` ✓
- `DISCORD_ALLOWED_CHANNELS` ✓
- `DISCORD_HOME_CHANNEL` → add
- `EMAIL_ADDRESS`, `EMAIL_PASSWORD`, `EMAIL_IMAP_HOST`, `EMAIL_SMTP_HOST` ✓
- `EMAIL_ALLOWED_USERS` ✓
- `EMAIL_POLL_INTERVAL` → default 15s (match Hermes)
- `EMAIL_HOME_ADDRESS` → add for cron delivery default
- `WHATSAPP_SESSION_SECRET` → document (Hermes uses session dir, not env secret)

### Changes to Validation Commands

Add Hermes-equivalent smoke tests:

```bash
# Delivery routing
bun test src/channels/delivery-router.test.ts

# Cron lock recovery
bun test src/cron/cron-tick-lock.test.ts

# Email normalization
bun test src/channels/email-adapter.test.ts

# Discord session key generation
bun test src/channels/discord-adapter.test.ts
```

## 6. Remaining Unresolved Questions

After comparing against Hermes docs and source, **zero questions remain unresolved** that block implementation planning.

The following are **not open questions** — they are decisions already made by this comparison:

- Test runner → Vitest
- Cron history → SQLite table
- WhatsApp library → Baileys
- Email library → Node `imap` + `nodemailer`
- Discord library → `discord.js`
- Delivery syntax → Hermes exact parity
- Gateway model → Single process
- Handoff model → Explicit codes (divergence from Hermes)
- Email friction → Mirror Hermes (no extra)
- Cron sessions → Fresh by default
- Discord sessions → Per-user by default

The only items that could become questions later are runtime-specific (e.g., "does `discord.js` work well under Bun?"), but these are implementation risks, not planning blockers. They should be resolved during Phase 0/2 implementation, not during planning.
