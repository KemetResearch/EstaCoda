# الهندسة المعمارية

EstaCoda هو بيئة تشغيل وكيل TypeScript لـ Node.js. ينفذ جلسات مدعومة بمزود عبر CLI وقنوات بعيدة متعددة. كل نظام فرعي قابل للفحص، وكل حدود الحالة صريحة، وكل تصنيف نضج يخبرك بما يمكنك الاعتماد عليه.

تشرح هذه الصفحة كيفية بنية النظام، وأين تعيش الحالة، والأسطح المستقرة أو التجريبية أو غير المدعومة في v0.1.0.

---

## ما هي EstaCoda

في جوهرها، EstaCoda هو حلقة وكيل قائمة على الأدوار:

1. تستقبل الإدخال من سطح (CLI، Telegram، Discord، Email، WhatsApp)
2. تجميع موجه طبقي من تاريخ الجلسة، والذاكرة، والمهارات، والسياق
3. إرسال الموجه إلى مزود نموذج لغوي مُهيأ
4. تنفيذ أي استدعاءات أدوات يطلبها المزود
5. إرجاع النتيجة إلى السطح
6. الاحتفاظ بحالة الجلسة، والذاكرة، والمسار

الحلقة ليست سحرية. إنها بناء صريح، وتفويض صريح، واستمرارية صريحة. كل حدود مرئية.

## نقاط الدخول والأسطح

| نقطة الدخول | الدور | النضج |
|---|---|---|
| `src/index.ts` | تدفق الإقلاع، توزيع الأوامر، استعادة الجلسة | `live-proven` |
| `src/cli/cli.ts` | تحليل وتوزيع أوامر CLI | `live-proven` |
| `src/cli/session-loop.ts` | جلسة طرفية تفاعلية | `live-proven` |
| `src/channels/gateway-runner.ts` | بيئة تشغيل بوابة Telegram | `live-proven` |
| `src/channels/discord-adapter.ts` | محول Discord | `present-not-live-proven` |
| `src/channels/email-adapter.ts` | محول Email | `present-not-live-proven` |
| `src/channels/whatsapp-adapter.ts` | محول WhatsApp | `experimental` |

CLI و Telegram هما السطحان المثبتان حيًا لـ v0.1.0. Discord و Email موجودان في الكود لكن لم يُتحقق منهما شاملاً. WhatsApp محصور بـ `experimental: true`.

## ملكية إدخال CLI والتصيير الطرفي

### محث الخمول

`ReadlinePrompt` يملك إدخال CLI الخامل. هو المسؤول عن buffer الحالي، وعن الإبلاغ بعدد أسطر المحث عندما يلتف النص، وعن تمرير لصق TTY عبر `PasteInterceptor` عندما يكون bracketed paste متاحًا. اعتراض اللصق محصور في مطالبات TTY المدعومة بالـ interceptor. مدخلات البوابة، والأوامر المنفردة، وقنوات غير CLI لا تستخدم هذا المسار.

مطالبات الأسرار حد أمني. يمكنها قبول bytes ملصوقة كإجابة، لكنها لا تنشر callbacks لمعاينة اللصق، ولا تلميحات slash مباشرة، ولا transient chrome يحتوي على نص سري.

### المنطقة المُدارة فوق `readline`

`BottomChromeController` يملك المنطقة المُدارة فوق محث `readline` النشط. أي واجهة مباشرة بجوار المحث، مثل تلميحات slash أو معاينة اللصق، يجب أن تمر عبر:

```typescript
updateManagedRegionAboveReadline({ state, transientLines, promptLineCount })
```

`promptLineCount` جزء من العقد. المحث الملفوف يحتل أكثر من صف، ويجب حسابه عند نمو المنطقة المُدارة أو انكماشها أو اختفائها. إذا اختفت تلميحات slash أو معاينة اللصق، يجب مسح الأسطر القديمة قبل إعادة المحث إلى موضعه.

لا تخلط يدويًا بين `updateTransientLines()` و `updateStateAboveReadline()` لتحديثات live readline. هذا النمط يسهل أن يفقد مزامنة عدد الأسطر عند ظهور transient rows أو اختفائها أو التفافها.

### chrome الدور النشط

بعد الإرسال، لم يعد `readline` يملك المؤشر. chrome الدور النشط يعرض الحالة، المؤقتات، spinner، نشاط الأدوات، مخرجات الإعداد/الموافقة، ورسائل command lane المؤقتة. لا يجب إعادة صندوق المحث الوهمي الذي كان يعرض النص المرسل للقراءة فقط.

`ActiveTurnCommandController` محلي للـ CLI. يربط listener فقط أثناء عمل `runtime.handle()` في TTY تفاعلي، ويخزن الأوامر التي تبدأ بـ `/`، ويصيّرها عبر مسارات transient/status في bottom chrome بدل الكتابة المباشرة إلى الطرفية. `/interrupt` يلغي الدور. `/steer <note>` يلغي الدور ويصفّ إعادة محاولة واحدة في طبقة CLI مع ملاحظة توجيه صريحة؛ ليس primitive لتوجيه runtime أو provider أثناء البث.

## جذر تكوين وقت التشغيل

`createRuntime()` في `src/runtime/create-runtime.ts` هو جذر التكوين. إنها دالة طويلة باستيرادات عديدة تبني 30+ كائن نظام فرعي يدويًا. لا يوجد حاوية حقن تبعية. كل نظام فرعي يُنشأ بحجج بانية صريحة بترتيب ثابت.

ترتيب البناء:

1. مخازن الحالة (memory store، session DB، artifact store، cron store)
2. سجل المزود وحل المسار الإضافي
3. سجل الأدوات
4. سجلات المهارات (official، profile، external)
5. تبعيات الموجه (prompt cache، context expander)
6. مكونات وقت التشغيل المستخرجة (`RunRecorder`، `ToolPlanRunner`، `ProviderTurnLoop`، `SkillWorkflowExecutor`، `NativeToolExecutor`)
7. `AgentLoop`

قواعد التكوين الرئيسية:

- المهارات الرسمية تُحمّل أولاً. مهارات الملف الشخصي والخارجية المُهيأة تُحمّل لاحقًا.
- كتالوج المهارات المرئية يُصفّى لكل جلسة باستخدام شروط وقت التشغيل.
- `vision.analyze` مسجّل كأداة حقيقية ويستخدم المسار الإضافي `vision`.
- خوادم MCP المُهيأة تُحمّل أثناء إنشاء وقت التشغيل وتتوقف أثناء التخلص منه.

## حدود الملف الشخصي والحالة

تنظم EstaCoda كل حالة المشغل حول **الملفات الشخصية**. الملف الشخصي هو حزمة كاملة للإعداد، والاعتمادات، والذاكرة، والمهارات، وهوية البوابة.

### ما يعيش تحت ملف شخصي

`~/.estacoda/profiles/<id>/` يحتوي على:

- `config.json` — مسارات المزود، إعداد القناة، سياسة الأمان
- `.env` — مفاتيح API، رموز البوت، الأسرار
- `auth.json` — رموز OAuth (مثل Codex)
- `skills/` — مهارات مثبتة في الملف الشخصي
- `memory/` — ملفات الذاكرة المحلية للملف الشخصي
- `cron/` — تعريفات الوظائف وحالة التنفيذ
- `gateway/` — مؤشرات جلسة القناة، رموز التسليم، حالة الموافقة
- `logs/` — سجلات وقت التشغيل
- `cache/` — ذاكرة التخزين المؤقت للموجه والتشغيل
- `media/` — الوسائط المحملة من القنوات

### ما هو عالمي

- `~/.estacoda/sessions.sqlite` — قاعدة بيانات الجلسات، الصفوف مُنطقة بـ `profile_id`
- `~/.estacoda/memory/shared/` — ذاكرة مشتركة عبر الملفات الشخصية
- `~/.estacoda/trust.json` — ثقة مساحة العمل لكل دليل
- `~/.estacoda/workspace-approvals.json` — منح موافقة مساحة العمل

### ما هو محلي لمساحة العمل

- `AGENTS.md` — سياق المشروع والاتفاقيات (ليس ذاكرة)
- `trust.json` و `workspace-approvals.json` مُنطقان بالدليل

الثقة منفصلة عن الملفات الشخصية. الملف الشخصي يختار الإعداد والاعتمادات. ثقة مساحة العمل تتحكم في سلوك الأداة المحلية لدليل. لا تتجاوز إحداهما الأخرى.

## الأنظمة الفرعية الأساسية

### طبقة المزود

طبقتان: السجل/التوجيه والتنفيذ.

**السجل:** كتالوغ النماذج أولاً دون اتصال، سجل المزود مع اختيار المسار حسب القدرة، البحث المباشر عن الاعتمادات من `apiKeyEnv` إلى `process.env`.

**التنفيذ:** `ProviderExecutor` يتولى جمع الرموز المتدفقة، تجميع أجزاء استدعاء الأدوات، ومعالجة الاحتياطي. `OpenAICompatibleProvider` هو المحول الاستدلالي الأساسي.

المسارات الإضافية تُحلّ عبر نفس البنية: `vision`، `compression`، `assessor`، `profile_context`، `web_extract`، `session_search`، `skills_library`، `mcp`، `memory_flush`، `delegation`. هذه ميّالات تفضيل، ليست بيئات تشغيل منفصلة.

راجع [مرجع المزودين](../reference/provider-reference.md) للتصنيف الدقيق لكل مزود.

### طبقة الأدوات

`ToolRegistry` يسجل الأدوات المدمجة عند تحميل الوحدة والأدوات المكتشفة عبر MCP عند إنشاء وقت التشغيل. `ToolExecutor` ينفذ إجراءات الأداة الملموسة تحت سياسة الأمان النشطة. `ToolPlanRunner` يحول استدعاءات أدوات المزود إلى خطط قابلة للتنفيذ، ويدير التزامن الآمن للأدوات، ويفرض سقوف الفشل.

فئات مخاطر الأدوات تحكم البوابة: `safe`، `caution`، `external-side-effect`، `irreversible`.

### طبقة المهارات

المهارات تُحمّل من ثلاثة مصادر:

| المصدر | الموقع | القابلية للتغيير |
|---|---|---|
| `official` | مضمن في المستودع | للقراءة فقط |
| `profile` | `~/.estacoda/profiles/<id>/skills/` | قابل للتغيير |
| `external` | جذور خارجية مُهيأة | للقراءة فقط |

الرؤية مستقرة بالجلسة وتُحدّث عبر `/reset` أو جلسة جديدة. تنفيذ المهارة مدعوم بالمزود افتراضيًا؛ يوجد مسار احتياطي حتمي للجلسات بدون مزود.

### طبقة الذاكرة

`MemoryStore` تدير ملفات الذاكرة المحدودة. `LocalMemoryProvider` يقرأ ويكتب `USER.md`، `MEMORY.md`، و `SOUL.md` من الملف الشخصي النشط. `MemoryRecallOrchestrator` يُعد سياق الذاكرة لكل دور لتجميع الموجه.

الاستذكار، والاستذكار الخارجي، وملخصات الضغط هي سياق مرجعي فقط. ليست موثوقة كتعليمات صاحبة سلطة.

### طبقة القناة

`ChannelGateway` هو الجسر المحول العام. يتولى المصادقة، وقوائم السماح، وتعيين الجلسة، وتوصيل التقدم، وتوصيل مطالبات الموافقة، وتوجيه الأوامر. المحولات لا تغير حالة الموافقة أو ذاكرة التخزين المؤقت لوقت التشغيل مباشرة.

`DeliveryRouter` هو مسار التوصيل المُسوّى لجميع القنوات. يتولى التوصيل متعدد الأهداف، واقتطاع النص، واستمرارية الأخطاء، وتوجيه المنتجات.

### طبقة الأمان

حدود الأمان قائمة على القدرات. أوضاع الموافقة: `strict`، `adaptive`، `open`. `adaptive` هو الافتراضي. `/yolo` يبدّل وضع `open` محدودًا بالجلسة لكنه لا يتجاوز الأرضية الصلبة.

الموافقات الدائمة تطابق `targetKey` المُسوّى. ملخصات العرض ليست حدود الموافقة.

الأرضية الصلبة تغطي: الحذف العودي الواسع، عمليات القرص التدميرية، إيقاف التشغيل/إعادة التشغيل، fork-bomb/kill-all، قراءة الأسرار، التثبيت عبر مفسر الأنابيب، git force-pushes.

## بنية الموجه

تجميع الموجه طبقي. مجموعات السياق الرئيسية:

1. الهوية وتوجيه الملف الشخصي
2. الأمان والذاكرة المكتسبة
3. سياق المشروع (بما في ذلك `AGENTS.md`)
4. تاريخ الجلسة
5. استذكار الجلسة والاستذكار الخارجي
6. الرسالة الحية للمستخدم والمرفقات
7. النية وتعليمات المهارة
8. إعداد المهارة ومواردها
9. خطة سير العمل
10. قائمة الأدوات
11. السياق المرجعي الصريح
12. نتائج الأدوات / ملاحظات الاستمرارية

الطبقات القابلة للتخزين المؤقت (الهوية، الأمان، سياق المشروع، موارد المهارة، فهرس المهارات المضغوط) تُعاد بناؤها فقط عند تغيير البيانات الأساسية. الطبقات غير القابلة للتخزين المؤقت (تاريخ الجلسة، الاستذكار، الرسالة الحية) تُعاد بناؤها كل دور.

القواعد الدلالية:

- سياق النظام المستقر بالجلسة مفضل على التغيير منتصف الجلسة.
- المهارات تُكشف تدريجيًا.
- المرفقات هي سياق منظم، ليس نص مستخدم مزيف.
- التنسيق الموجه للقناة يحدث بعد توليد النموذج.
- `AGENTS.md` هو سياق المشروع، ليس ذاكرة.
- الاستذكار وملخصات الضغط هي سياق تاريخي غير موثوق.

## تدفق البيانات

المسار الرئيسي من البداية إلى النهاية:

1. يصل الإدخال من CLI أو قناة
2. يُسوّي وقت التشغيل الرسالة + المرفقات
3. يبني تجميع الموجه طبقي طلبًا للمزود
4. يستجيب المزود بنص و/أو استدعاءات أدوات
5. مخطط الأدوات + المنفذ ينفذان إجراءات ملموسة تحت السياسة
6. موجه الاستمرارية يُغذي نتائج الأدوات إذا لزم الأمر
7. الإخراج النهائي مُنسق لكل سطح
8. حالة الجلسة، والذاكرة، والموافقات، والمسار تُحفظ

`AgentLoop.handle()` ينسق هذا التدفق لكنه يفوض التنفيذ إلى مكونات متخصصة. لا ينفذ تكرارات المزود، أو خطط الأدوات، أو سير عمل المهارات مباشرة.

## نقاط الضعف المعمارية

1. **مصنع `create-runtime.ts` الإلهي** — 900+ سطر، 69 استيراد، لا حدود حقن تبعية. مخاطرة مقبولة. نمط الباني مؤجل.
2. **الربط المتبقي في AgentLoop** — تجميع الموجه، وحقن سياق الذاكرة، والتنسيق بين المكونات لا يزال في `AgentLoop`. حلقة المزود، وتنفيذ الأدوات، وسير عمل المهارات، والنوايا المحلية مُستخرجة بالفعل.
3. **جاهزية البوابة مقابل الحيوية** — `estacoda gateway diagnose` يُبلغ الجاهزية لكل محول، لا حيوية العملية الخلفية.
4. **هياكل المسار/المنتج** — المسار يُحفظ في SQLite. `ArtifactStore` رقيق (في الذاكرة باستمرارية محدودة).
5. **روابط SQLite الأصلية** — `better-sqlite3` يتطلب التجميع عند التثبيت على بعض المنصات.

## كيفية الفحص

```bash
# جاهزية وقت التشغيل ومشاكل الإعداد
estacoda model diagnose

# جاهزية البوابة لكل محول
estacoda gateway diagnose

# حالة البوابة الكاملة
estacoda gateway status

# حالة المزود
estacoda model show

# قائمة الجلسات مع المرفقات السطحية
estacoda sessions list

# حالة القنوات
estacoda channels status
```
