# EstaCoda WhatsApp Bridge

This directory is a standalone npm package for EstaCoda's experimental WhatsApp transport helper.

WhatsApp support uses the unofficial Baileys linked-device API. Meta may suspend accounts that use unofficial WhatsApp libraries. The bridge keeps `@whiskeysockets/baileys` and WhatsApp-specific `@hapi/boom` handling out of the root EstaCoda runtime.

## Dependency Boundary

This package is intentionally not part of the root pnpm workspace. Install bridge dependencies from this directory only:

```bash
cd scripts/whatsapp-bridge
npm ci
```

The root package ships only these bridge helper files:

- `scripts/whatsapp-bridge/package.json`
- `scripts/whatsapp-bridge/package-lock.json`
- `scripts/whatsapp-bridge/bridge.js`
- `scripts/whatsapp-bridge/README.md`

It must not ship `scripts/whatsapp-bridge/node_modules/`, and the root runtime must not import Baileys or `@hapi/boom`.

## Modes

- Normal managed mode: started by the EstaCoda gateway lifecycle manager.
- Pair-only foreground mode: started by `estacoda whatsapp` so the operator can scan the QR code in the terminal.

The CLI wizard supports QR-only WhatsApp device pairing. It does not expose WhatsApp pairing-code setup. QR strings are printed to the terminal by the foreground bridge and are not persisted.

## HTTP Contract

The main runtime talks to the bridge over loopback HTTP with plain JSON:

| Endpoint | Purpose |
|----------|---------|
| `GET /health` | Bridge/socket status, API version, queue depth, dropped inbound count |
| `GET /messages` | Drain inbound message queue |
| `POST /send` | Send text, optionally quoting `replyTo` |
| `POST /edit` | Attempt message edit |
| `POST /send-media` | Send validated local image/video/audio/voice/document path |
| `POST /typing` | Send composing/paused presence |
| `GET /chat/:id` | Check chat/contact information |

Every request must use the per-launch bearer token generated by the lifecycle manager. The bridge validates request bodies, enforces request/response size limits, and returns stable JSON errors with `code` and `message`. Health responses include `apiVersion: "whatsapp-bridge.v1"` so the main runtime can fail closed on incompatible bridge contracts.

All Baileys-facing outbound operations use a 60-second timeout. Long final replies are chunked by the main runtime adapter with a short inter-chunk delay before they reach `/send`.

## Socket Settings

The bridge owns Baileys socket setup and disconnect classification. Socket defaults are kept local to this package:

- `syncFullHistory: false`
- `markOnlineOnConnect: false`
- `browser: ["EstaCoda", "Chrome", "120.0"]`
- `fetchLatestBaileysVersion()`
- `getMessage` callback for retry/E2EE recovery

Logged-out and not-paired states are reported as stable bridge error/status codes. Runtime logged-out handling does not delete auth files.

## Security Boundary

The bridge is a transport helper, not a policy engine.

It does:

- bind only to loopback hosts
- reject non-loopback bind hosts
- validate `Host` against loopback names
- require a per-launch bearer token
- keep an inbound queue capped at 100 messages, dropping oldest on overflow
- avoid logging message content, QR strings, bearer tokens, pairing codes, auth blobs, or secrets

It does not:

- decide workspace trust
- authorize users
- create or consume WhatsApp user authorization codes
- own approvals or runtime security policy
- fetch remote media URLs
- validate workspace paths
- run `ffmpeg`

Media policy stays in the main runtime. The root adapter validates local paths against the trusted workspace and profile-local media/temp roots, caches explicitly allowed remote media URLs locally, enforces upload size limits, and performs optional `ffmpeg` conversion for WhatsApp voice bubbles before handing this bridge a validated local path.

## Logs

Managed gateway runs capture bridge stdout/stderr to:

```text
~/.estacoda/profiles/<profile-id>/logs/whatsapp-bridge.log
```

Explicit dependency repair/install output is captured separately:

```text
~/.estacoda/profiles/<profile-id>/logs/whatsapp-bridge-install.log
```

Use `estacoda gateway diagnose` and `estacoda gateway status` for operator-facing readiness/status. Use this README for the transport contract and dependency boundary.
